# Codex 安装前置条件检查清单

安装 Codex 或任何 AI 编程工具之前，先做一次环境检查，比直接复制命令更稳。新手常见卡点不是“不会安装”，而是系统、终端、Node、Git、权限和网络状态没有记录清楚。下面这份清单可以帮你判断是否适合继续安装，或先把问题整理成待办。

完整解释可读 [安装 Codex 前要准备什么](/blog/codex-install-prerequisites)。如果你已经遇到安装失败或命令不存在，可以看 [安装前置条件常见错误和修复顺序](/blog/codex-install-prerequisites-mistakes)。

## 适合谁

适合准备第一次配置 AI 编程环境的新手。你可能还没接触过命令行，也可能已经安装过 Node 和 Git，但不知道当前版本是否能用。

也适合准备项目前做环境自查的人。客户项目如果一开始就跑不起来，你至少要能说明卡在哪里、需要客户提供什么、是否涉及权限或账号。

## 不适合谁

不适合只想快速复制安装命令、不愿意保存记录的人。没有记录，后续报错很难复盘。

也不适合在客户重要项目里直接试装工具。任何涉及客户代码、私有仓库、账号权限或生产数据的操作，都要先确认授权范围。

## 风险提醒

检查清单不是安装保证。它只能帮助你发现基础条件是否齐全。不同系统、网络、公司设备策略和工具版本都会影响结果。

遇到登录、付费、授权、密钥、客户仓库访问等事项，不要当场硬推进。先记录为待客户确认或明日待办，再决定下一步。

## 具体步骤

### 1. 系统信息

- 你使用的是 Windows、macOS 还是 Linux。
- 系统版本是否记录。
- 是否是个人电脑，而不是公司或客户受管设备。
- 是否有管理员权限。
- 是否能安装开发工具。

如果权限不清楚，先暂停，不要绕开设备策略。

### 2. 终端能力

- Windows 能打开 PowerShell。
- macOS 或 Linux 能打开 Terminal。
- 能运行 `pwd` 查看当前位置。
- 能运行 `cd` 进入项目目录。
- 能复制完整报错，而不是只截半张图。

终端是后面排错的基础，不熟悉也没关系，但要愿意记录。

### 3. Node 和 npm

- `node -v` 能输出版本。
- `npm -v` 能输出版本。
- 项目是否要求特定 Node 版本。
- 电脑上是否存在多个 Node 版本。
- 是否知道当前命令来自哪个安装路径。

如果版本混乱，不要随意重复安装。先记录现状，再决定是否用版本管理工具。

### 4. Git

- `git --version` 能输出版本。
- 能打开项目所在仓库。
- 能查看 `git status`。
- 知道当前分支。
- 知道不要随意删除未提交改动。

Git 是项目安全感的一部分。不会回退，就不要大改客户项目。

### 5. 编辑器和文件

- 能用 VS Code 或其他编辑器打开文件夹。
- 能搜索文件。
- 能查看终端。
- 能区分项目文件和系统文件。
- 能保存截图和日志。

编辑器不是越复杂越好，关键是你能稳定找到文件。

### 6. 网络、账号和权限

- 能访问工具官网或文档。
- 是否需要登录账号。
- 是否需要 API key、token 或平台授权。
- 是否需要客户提供仓库权限。
- 是否涉及付费服务。

任何账号和授权事项，都要写进待确认列表。不要把个人账号混入客户项目。

## 审核结果怎么判断

可以给自己一个结论：可以继续安装、需要先补环境、需要客户确认、暂不处理。可以继续安装表示基础命令和权限都清楚；需要补环境表示缺 Node、Git 或编辑器；需要客户确认表示缺仓库、账号或授权；暂不处理表示风险超出当前能力。

这个结论能帮助你避免盲目尝试。项目场景里，它也能变成客户沟通材料。

## 检查记录怎么保存

建议把检查记录写成固定格式，而不是散在聊天窗口里。至少保存五类信息：电脑系统、终端路径、版本命令输出、失败日志、下一步判断。比如你可以写“Windows 电脑，PowerShell 可用，Node 版本已记录，Git 可用，但客户仓库权限未提供”。这样的记录很短，却能让后续沟通更清楚。

如果你准备把这次环境检查整理成作品集，不要展示客户私有路径、真实账号或内部链接。可以把项目名称匿名化，只展示你如何定位问题、如何判断缺口、如何提出下一步。这样既能证明你有流程，也不会把不该公开的信息带出去。

## 客户项目额外检查

如果目标不是自己的练习项目，而是客户项目，还要多问几项。客户是否允许你在本地克隆代码？是否允许安装依赖？是否提供测试账号？是否有 README 或 onboarding 文档？是否能接受你先做诊断报告，再决定是否进入修复？

这些问题听起来慢，但它们能防止任务变形。环境配置经常会牵出项目代码问题、网络问题、权限问题和部署问题。先把边界写清，后面才不会把所有失败都算到你头上。
