# TypeScript 报错新手怎么排查：常见错误和解决步骤

TypeScript 报错最怕“为了过编译而乱改”。有些改法能让红字消失，却把真实问题藏起来。新手应该把错误当成线索：它指向哪个文件、期待什么类型、现在拿到了什么值、为什么两者不一致。

这篇仍是草稿，发布前需要补充真实案例。

## 适合谁

适合已经遇到 TS 报错、但不知道该改类型还是改数据的新手。你能运行检查命令，也能查看报错行附近代码。

也适合准备接报错修复单的人。理解常见错误，可以避免低估任务范围。

## 不适合谁

不适合客户只给一句“build failed”但没有日志的需求。也不适合没有测试环境的生产系统。

如果报错影响支付、权限、客户数据或安全逻辑，需要更有经验的人复核。

## 常见错误

第一个错误是只看最后一行。TypeScript 报错经常有调用链，真正原因可能在前面。第二个错误是随手加 `any`，短期消除报错，长期失去类型保护。

第三个错误是强行断言类型。比如把可能为空的数据当成一定存在，运行时仍可能出错。第四个错误是同时改配置、依赖和业务代码，最后不知道哪一步起作用。

第五个错误是忽略版本。库升级后类型变化很常见，不能只盯着当前文件。

## 具体步骤

1. 保存完整报错和运行命令。
2. 从第一条错误开始读。
3. 找到报错文件和行号。
4. 写下期望类型和实际类型。
5. 判断是数据问题、类型定义问题、依赖问题还是配置问题。
6. 每次只改一个假设。
7. 运行同一个验证命令。
8. 修复后写清原因和影响范围。

## 可复制回复

```text
我可以先做一轮 TypeScript 报错诊断。请提供完整报错、运行命令、仓库或相关文件、最近改动和期望行为。诊断后我会说明问题属于类型定义、数据结构、依赖版本还是配置范围，再确认修复方案。
```

## 修复后怎么复盘

复盘不要只写“已修复”。要写清楚原问题是什么、为什么会报错、改了哪个文件、用什么命令验证。这样客户能看懂交付，你自己也能把案例整理成作品集。

如果为了快速通过只做了临时处理，要明确写成临时方案，并记录后续需要完善的类型。不要把临时方案包装成最终修复。

## 更稳的修复习惯

每次只处理一类错误。比如先处理空值，再处理属性缺失，最后处理依赖类型。不要看到所有红字就一起改，因为这样很难判断哪一步真的解决问题。

修复时尽量让类型更接近真实数据。如果接口可能返回空数组，就不要把它写成永远有数据；如果字段可能缺失，就要在界面上处理缺失状态。TypeScript 的价值不是让编译器安静，而是让代码更诚实。

如果 AI 给出一大段修改建议，先让它解释为什么这样改，再拆成小步骤验证。不要直接复制大改。对接单来说，能解释修改原因比“改完不报错”更重要。

## 作品集怎么记录

可以把案例写成“问题、定位、修复、验证、边界”五段。不要贴客户代码和敏感截图。用自己的练习项目复现同类问题，会更适合公开展示。

如果案例来自真实客户，必须先去掉客户名称、仓库地址、业务数据和内部截图。更稳的方式是用相同错误类型重做一个演示项目，再公开说明思路。

公开前再人工复核。

## 风险提醒

不要滥用 `any`、不要关闭类型检查、不要忽略空值、不要一次改很多文件。不要在没有授权和备份时改客户共享类型。
