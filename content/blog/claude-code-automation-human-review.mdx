# Claude Code 自动化为什么必须有人审

Claude Code 可以帮你更快读仓库、改文件、整理修复说明，但它不应该绕过人工审核。新手最容易被“自动化”这三个字带偏：让工具一次性改很多内容，然后只看最终结果是否看起来顺眼。真正可交付的流程应该是工具辅助、人来判断、命令验证、客户验收。

如果你正在把 Claude Code 用在 AI 接单流程里，可以先看 [Claude Code 新手检查清单](/blog/claude-code-beginner-mistakes-checklist)、[Bug 修复交付说明模板](/blog/bugfix-delivery-note-template) 和 [/tools/error-explainer](/tools/error-explainer)。这篇重点讲自动化之后为什么还要人工审。

## 适合谁

适合已经会让 Claude Code 修改代码、生成文档、解释报错的新手。你可能希望把重复流程自动化，比如读取错误、定位文件、生成修复建议、整理交付说明，但还不确定哪些步骤必须人工确认。

也适合准备接低风险小单的人。页面小改、报错说明、配置文档、测试环境排查、内容整理，都可以用工具提高速度；但每一步都要有边界和证据。

## 不适合谁

不适合用工具直接操作客户生产环境、付款后台、真实用户数据、密钥或管理员权限。自动化越快，越容易放大错误。没有人工复核的自动化不适合处理高责任任务。

也不适合把模型输出当成客户验收结果的人。模型说“已完成”没有意义，只有 diff、测试、截图、构建结果和客户验收步骤才是证据。

## 具体步骤

第一步，先定义自动化范围。让 Claude Code 做“读取相关文件并总结”“提出修改方案”“改一个明确问题”“整理交付说明”，不要让它一次性完成所有事情。

第二步，修改前看当前状态。确认工作区是否干净、客户需求是否已脱敏、任务是否只涉及低风险文件。已有未提交改动时，不要随便覆盖。

第三步，修改后看 diff。人工确认改动是否符合需求、是否只动目标文件、是否没有泄露密钥、是否没有改变 draft 公开状态、是否没有无关重构。

第四步，运行检查。代码任务看 lint、build、测试或页面手测；内容任务看内容质量、敏感词、SEO；交付任务看说明是否包含修改、验证和未覆盖范围。

第五步，写交付说明。把工具参与过程转化成客户看得懂的结果：原问题、修改内容、测试证据、客户需要确认的事项。

## 风险提醒

不要把客户敏感资料交给自动化流程。能用占位符表达的，就不要放真实信息；能用测试数据验证的，就不要碰生产数据。任何完整密钥、客户账号、订单或后台链接都不应进入聊天记录和提交记录。

不要让工具自行扩大任务范围。它可能建议重构、升级依赖、改命名或清理无关文件。新手接单时，范围越小越容易解释，越容易验收。

## 失败信号

如果你看不懂工具改了什么，说明还不能交付。如果你没有运行任何检查，说明结果还没有证据。如果客户提出的是高权限需求，而你只用一句“Claude Code 可以处理”回应，说明风险判断缺失。

自动化应该让流程更稳，而不是让你少看关键步骤。越是省时间的工具，越要保留清楚的检查点。

## 审核记录怎么写

建议每次自动化后写四行记录：工具做了什么、人工看了什么、跑了什么检查、哪些范围未覆盖。比如“Claude Code 修改了表单提交逻辑；人工确认 diff 只涉及目标组件；`npm run build` 通过；未验证生产订单和支付流程”。

这份记录可以直接转成客户交付说明，也能保护你自己。客户后续追问时，你能说清楚哪些是本次完成项，哪些需要新增范围。新手越早建立记录习惯，越不容易被自动化速度带偏。

## 人工审核不是形式

人工审核不是在最后快速扫一眼，而是要真正做判断。你要判断工具是否理解需求、是否改错文件、是否遗漏客户限制、是否引入新风险。无法判断时，要暂停、缩小范围或请人复核。
