# Claude Code 新手接单前怎么判断使用场景

客户把代码需求发过来时，新手容易被“Claude Code 能帮忙”吸引，直接开始修改。但接单前更重要的是判断使用场景是否合适。一个适合新手的 Claude Code 任务，应该能复现、权限低、范围小、可测试、能写清交付说明。

这篇给一个接单前判断框架。你可以配合 [Claude Code 使用场景检查清单](/blog/claude-code-beginner-use-cases-checklist)、[Claude Code 自动化人工审核清单](/blog/claude-code-automation-human-review-checklist) 和 [/tools/proposal-generator](/tools/proposal-generator) 使用。

## 适合谁

适合准备承接代码阅读、报错诊断、小范围修复、配置文档和交付说明的新手。这些任务能用 Claude Code 提高效率，也能用测试或文字说明验证结果。

也适合已经收到客户需求，但不确定要不要报价的人。你可以先按本文判断是否直接接、先诊断，还是暂不承接。

## 不适合谁

不适合客户要求直接进入生产环境、处理真实数据、保存密钥、修改付款设置或独立处理安全事故的情况。这些需求需要更严格流程，不适合新手用工具直接处理。

也不适合客户不给复现条件的情况。没有错误、链接、截图、命令输出或预期结果，Claude Code 只能猜测，不能形成可靠交付。

## 具体步骤

第一步，确认环境。问题是在本地、测试环境、线上预览，还是生产环境？新手优先选择本地和测试环境问题。

第二步，确认权限。任务是否需要账号、密钥、数据库、付款后台或管理员权限？如果需要，就先建议客户提供脱敏信息和测试环境。

第三步，确认范围。把需求写成一句话：本次只解决哪个问题。如果一句话写不清，说明范围还太大。

第四步，确认验证方式。修完后能不能跑命令、截图、打开页面、用测试数据复现？没有验证方式，就只能交付诊断建议。

第五步，确认沟通方式。把包含项、不包含项、客户需要提供的材料写清楚，再报价。

## 报价建议

信息不完整时，先报诊断小单。诊断包含：阅读需求、整理错误、判断风险、给出下一步材料。确认是低风险问题后，再报价修复。

范围明确时，可以报小范围修复。比如“修复一个页面的按钮问题并提供测试截图”“解释一个构建错误并给出修改建议”“整理一份配置文档”。价格和交付物绑定，不要把长期维护默认包含进去。

## 风险提醒

不要把客户说的“很简单”当成事实。很多复杂问题在客户眼里都很简单，因为他们只看到结果，没有看到背后的权限、环境和依赖。

不要让 Claude Code 直接成为报价依据。报价前要由你判断范围和风险，工具只能辅助你理解材料。

## 客户确认模板

```text
我可以先判断这个需求是否适合用 Claude Code 辅助处理。请先确认：
1. 问题出现在哪个环境？
2. 是否可以提供错误截图或日志？
3. 是否能使用测试数据？
4. 是否涉及密钥、付款、生产数据或管理员权限？
5. 你希望最终看到什么验收结果？
```

客户愿意配合回答，说明任务更容易进入小范围流程。客户拒绝提供信息，却要求你直接承诺结果，就要谨慎。

## 交付范围写法

可以把交付范围写成：“本次包含问题诊断、目标文件小范围修改、基础检查和交付说明；不包含生产环境发布、客户账号代管、真实数据处理、长期维护和新增功能。”这句话比“我可以帮你处理”清楚得多。

如果客户需要更多内容，也可以继续谈，但要作为新增范围处理。新手接单时最重要的是让报价、时间和责任范围一致。

## 验收方式

验收方式要在接单前想清楚。比如客户按指定路径复现一次、你提供构建通过截图、你写出修改文件和测试结果、客户在测试环境确认。没有验收方式的任务，很容易变成无边界沟通。

如果验收只能在生产环境完成，就要提高风险等级。新手可以协助准备检查清单，但不应该在没有明确授权和复核的情况下直接操作。
