# 环境变量 missing 使用前怎么判断是否适合

环境变量缺失看起来像小问题，但项目风险很高。因为它经常涉及 API key、数据库连接、支付密钥、邮件服务、OAuth secret、部署平台权限和客户账号。新手能不能接，关键不在于会不会填变量，而在于能不能保护密钥、拆清环境、写清边界。

更稳的接法是先做“环境变量缺失诊断”，而不是直接承诺配置完成。诊断交付包括缺失变量名、发生环境、变量类型、配置建议、客户待确认事项。具体流程见 [环境变量 missing 怎么排查](/blog/env-variable-missing-fix)。

## 适合谁

适合能读日志、能区分本地和部署环境、能看项目说明、能写交付记录的新手。你可以帮助客户定位变量名和配置位置，但不需要也不应该保存真实密钥。

也适合已有部署练习记录的人。客户项目里，能保护敏感信息比“马上帮你弄好”更重要。

## 不适合谁

不适合客户要求你接收、保存、代管真实密钥的任务。也不适合需要你登录生产部署后台、数据库后台、支付后台或 OAuth 后台的新手独立操作。

如果客户不愿提供日志，只要求你猜缺什么变量，也不适合直接接修复单。可以先接诊断问题整理。

## 风险提醒

环境变量任务的最大风险是泄密和越权。你不能要求客户把密钥发到聊天里，也不能把密钥写进代码、交付记录或截图。客户应在自己的账号和平台里配置敏感值。

另一个风险是责任边界。配置变量后可能影响生产服务、邮件发送、支付、数据库连接和第三方接口。没有授权和回滚方式时，不要直接操作。

## 具体步骤

### 1. 先收材料

客户至少要提供：完整报错、缺失变量名、发生环境、部署平台、项目类型、是否有 `.env.example` 或 README。真实变量值不需要提供。

如果客户只给一句“环境变量 missing”，先问问题，不报价修复。

### 2. 定义第一档服务

第一档叫“环境变量缺失诊断”。交付内容是变量名、用途推断、发生环境、配置位置建议、客户待确认事项。

这档服务可以不接触真实密钥，适合新手。

### 3. 判断是否进入协助配置

只有客户授权明确、配置位置清楚、敏感值由客户自己填写时，才考虑协助配置。你可以指导客户在哪个页面添加变量，但不保存值。

如果需要登录客户平台，建议让客户全程操作，你只提供步骤说明。涉及注册、购买、授权的事项，记录为待办，不替客户完成。

### 4. 报价拆分

报价至少拆成三段：日志诊断、配置说明、验证记录。如果还要部署验证或线上复查，要单独列出。

用 [报价计算器](/tools/pricing-calculator) 可以帮助你把诊断时间和验证时间分开。

## 可以接的任务

可以考虑接：客户提供完整日志，只需要判断缺哪个变量；作品集部署缺变量；开源项目本地 `.env.example` 对不上；预览环境缺少非敏感配置。

这些任务共同点是：不要求你保存密钥，材料清楚，可验证。

## 应该暂停的任务

应该暂停：客户要求你接收数据库连接串；客户把支付密钥发给你；需要你登录生产后台；需要你修改客户 OAuth 应用；需要你代管 API key；客户不允许记录操作。

暂停时写清原因：需要客户自行配置敏感值，或需要更高权限人员确认。

## 报价边界怎么写

报价时建议分成三层。第一层是诊断：读日志、确认变量名、判断发生环境、写客户待确认事项。第二层是配置说明：告诉客户在哪里添加变量、添加后怎么重启或重新部署。第三层才是协助验证：客户配置完成后，你根据日志和页面结果确认问题是否消失。

新手更适合先接第一层和第二层，不建议直接代管第三方后台。涉及真实密钥、账号登录、付费服务、生产配置的事项，写成待确认，不要含糊地放进“我帮你配置好”。

交付边界可以写得很直白：我不接收或保存真实密钥；我只确认变量名、用途和配置位置；敏感值由客户在自己的后台填写；配置后再进行验证。这类说明会让客户一开始觉得慢，但对长期合作更稳。

## CTA：下一步

先把服务范围写成“环境变量缺失诊断”。只收变量名和日志，不收真实密钥。材料齐了，再决定是否进入协助配置。
