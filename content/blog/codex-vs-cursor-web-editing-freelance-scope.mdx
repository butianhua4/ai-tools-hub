# Codex 和 Cursor 网页修改项目能不能接

网页修改任务很适合 AI 工具辅助，但不能因为会用 Codex 或 Cursor 就什么都接。能不能接，取决于任务是否清楚、代码权限是否合理、客户素材是否最终版、验证方式是否可执行、交付范围是否可控。工具只是提高效率，不能替代项目边界。

如果你还没建立工具选择流程，先看 [Codex 和 Cursor 改网页怎么选](/blog/codex-vs-cursor-web-editing)。如果需要逐项判断，用 [网页修改选择清单](/blog/codex-vs-cursor-web-editing-checklist)。

## 适合谁

适合已经能做基础网页修改的新手，例如改文案、调 Tailwind、修移动端、整理组件、跑 build、保存截图。你可以用 Cursor 做局部编辑，用 Codex 做分析、检查和交付记录。

也适合准备接小网站交付检查的人。工具分工明确后，客户更容易理解你交付的是结果和证据，而不是工具演示。

## 不适合谁

不适合客户要求你重做整站、设计品牌、写所有文案、部署上线、长期维护，却只按小修改付费的任务。

也不适合涉及支付、登录、订单、生产数据库、真实用户数据和安全问题的新手独立承接。

## 风险提醒

网页修改项目最容易范围膨胀。客户说“改一下首页”，可能包含设计、文案、图片、移动端、表单、SEO、部署和域名。报价前要拆清楚。

工具风险也要写清。AI 生成的代码必须人工 review，尤其是共享组件、配置文件、依赖和全局样式。客户项目还要注意私有代码和素材授权，不要把不该公开的信息放进外部工具。

## 具体步骤

### 1. 判断任务类型

把任务分成文案替换、样式调整、移动端修复、组件修改、报错排查、部署检查、完整页面制作。新手更适合前四类里的小范围任务。

### 2. 问清材料

至少要问：页面链接、代码仓库或文件、设计稿、最终文案、图片素材、目标设备、验收标准、是否需要部署。材料不足时，先做诊断，不要报价完整修复。

### 3. 设计工具分工

可以写成：Codex 负责拆需求和检查清单，Cursor 负责局部编辑，人工负责运行和截图。这样客户不会以为 AI 自动完成全部工作。

### 4. 设置暂停条件

如果需要客户账号、私有仓库、域名、支付、数据库、真实用户数据或付费服务，应暂停确认。需要注册或授权的事项，记录为客户待办。

### 5. 设计交付物

交付物可以是改动摘要、截图、命令输出、已验证页面、未覆盖范围和客户待确认事项。报价时用 [报价计算器](/tools/pricing-calculator)估算阅读、修改、验证和沟通时间。

## 可以考虑接的任务

客户有一个 3 页以内的小网站，希望你改文案、调按钮、修移动端并提供截图。这类任务可以考虑，尤其能配合 [小网站交付检查](/blog/codex-small-website-delivery-check)。

客户指出一个明确 bug，例如 Tailwind 移动端横向滚动，并提供链接和截图。这也适合新手做小范围诊断或修复。

## 报价怎么拆

可以把网页修改拆成三档。第一档是诊断：阅读页面、列问题、给截图和建议。第二档是局部修复：只修指定页面和指定问题。第三档是交付检查：修复后跑命令、看移动端、检查链接并整理记录。

不要把设计重做、文案创作、图片处理、部署、域名、表单服务和长期维护默认包含在网页修改里。每多一项，风险和时间都会变。

## 交付说明怎么写

交付说明建议包含：本次修改页面、修改内容、使用工具分工、已验证视口、已运行命令、未覆盖范围和客户待确认事项。这样客户可以清楚看到你做的是可验证交付，不是随意生成一版页面。

如果需要起草客户回复，可以用 [Proposal 生成器](/tools/proposal-generator)，但最终要把工具相关表述改成结果相关表述。客户更在意“我会检查哪些页面”，不是“我会用哪个工具”。

## 应该暂停的任务

客户要求你“用 AI 快速重做整个网站”，但没有设计稿、文案、素材、验收标准和部署边界，应暂停。客户要求你处理登录后台、支付、真实订单或生产数据，也不适合作为普通网页修改单。

客户只按截图报价，却要求你保证线上所有页面都修好，也要重新拆范围。
