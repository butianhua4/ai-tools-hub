# 用 Codex 准备作品集页面素材

AI 工具新手常常卡在作品集：不是完全没有做过东西，而是不知道怎么把练习、修复、页面改版、文档整理这些小任务变成可信案例。Codex 可以帮你整理过程、提炼问题、生成页面文案草稿，但作品集素材必须由人工确认。客户或雇主真正想看的不是“我用了 AI”，而是你做了什么、怎么验证、结果有什么边界。

这篇文章给一套保守流程：先筛选适合公开的案例，再整理截图、前后对比、任务背景、交付证据和风险说明。若你只需要一张表，可以看 [作品集素材检查清单](/blog/codex-portfolio-page-assets-checklist)；如果素材准备时已经出现夸大、授权不清、截图混乱等问题，可以看 [常见错误和修复顺序](/blog/codex-portfolio-page-assets-mistakes)。

## 适合谁

适合已经做过一些小项目的人。比如你用 Codex 辅助修过 npm 报错、整理过 README、改过移动端样式、做过一个报价计算器、写过一个 landing page 草稿，或者把客户需求拆成过可执行步骤。这些都可能成为作品集素材，但要整理成“问题、动作、结果、证据”的结构。

也适合准备投 Upwork、Fiverr、独立站询盘或中文私域项目的人。作品集页面不需要一开始就很大，先把 2 到 3 个可核验案例写清楚，比堆一堆没有证据的技能标签更有效。

## 不适合谁

不适合把客户未授权材料直接放上网的人。客户截图、后台页面、私有仓库、聊天记录、真实数据和品牌信息，都需要确认能否公开。不能确认时，可以做匿名化处理，或只写内部学习记录，不放进公开作品集。

也不适合把 AI 生成的演示图、虚构客户和未完成项目包装成真实交付。作品集可以包含练习项目，但要明确说明是练习、复刻、demo 还是真实客户任务。模糊这条边界，后续沟通会很麻烦。

## 风险提醒

作品集最容易出问题的地方是“看起来很漂亮，但无法证明”。如果页面只写“提升效率”“优化体验”“完成自动化”，却没有任务背景、你负责的部分、验证方式和限制说明，读者很难判断你的能力。

另一个风险是泄露信息。截图里可能包含邮箱、订单号、客户名称、内部路径、API key、后台菜单或私人消息。发布前要逐张检查，必要时打码、裁剪或改用模拟数据。需要登录平台、购买域名、开通服务或请求客户授权的事项，应记录为待办，不要自行越过确认流程。

## 具体步骤

1. 先列候选案例。把你做过的小任务按“项目名、问题、你做的动作、结果、证据、是否可公开”记录下来。不要急着写页面。

2. 用 Codex 辅助分类。让它把候选案例分成练习项目、真实任务、内部记录、暂不公开四类，并说明每类缺少什么证据。

3. 人工确认授权。真实客户项目要确认截图、名称、数据和链接是否能展示。不能确认时，先用匿名描述，或只保留在私有作品集中。

4. 收集素材。至少准备项目标题、问题背景、你的职责、关键截图、前后对比、运行或验收记录、使用工具、限制说明和下一步建议。

5. 写成案例结构。建议使用“背景、目标、过程、结果、复盘”五段。用 Codex 生成初稿可以，但要删除夸张承诺，补上真实证据。

6. 做发布前检查。检查是否有敏感信息、是否误导读者、是否把练习项目写成客户项目、是否有明确 CTA。需要工具时，可以用 [Proposal 生成器](/tools/proposal-generator)整理客户沟通记录，用 [报价计算器](/tools/pricing-calculator)估算任务耗时。

## 素材应该长什么样

一个合格素材包不一定复杂，但要完整。比如一个“修复移动端按钮遮挡”的案例，可以包含：原始问题截图、修改后截图、相关文件路径、验证设备尺寸、运行命令、你没有覆盖的浏览器范围，以及客户或自测验收记录。

如果是练习项目，也可以写清楚：这是为了训练 Codex 审核能力而做的 demo，重点是需求拆解、代码差异检查和交付记录。诚实说明来源，不会削弱作品集，反而会让读者知道你有边界感。

## 可以使用的页面结构

```text
项目名称：
项目类型：练习 / 客户任务 / 内部工具 / 开源贡献
问题背景：
我的职责：
使用工具：
关键动作：
验证方式：
结果和限制：
可公开素材：
仍需确认：
```

这个结构适合先放在 [模板库](/templates) 的案例记录里，等素材稳定后再做成作品集页面。

## CTA：下一步

先选 1 个最小案例，不要立刻做完整网站。把截图、任务背景、验证记录和限制说明整理出来，再用 [作品集素材检查清单](/blog/codex-portfolio-page-assets-checklist) 逐项确认。
