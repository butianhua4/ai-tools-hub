# Claude Code 交付说明哪些内容适合写进接单范围

交付说明本身可以成为接单范围的一部分。很多客户不只是需要你改代码，也需要你把问题、修改、验证和下一步讲清楚。对新手来说，把交付说明写进报价范围，能让小单更完整，也能减少后续误解。

这篇适合搭配 [用 Claude Code 接单后交付说明怎么写](/blog/claude-code-delivery-note)、[Claude Code 常用命令检查清单](/blog/claude-code-common-commands-checklist) 和 [/templates](/templates) 使用。

## 适合谁

适合想把代码小单拆成“诊断、修复、交付说明”三段的新手。你不一定能处理复杂系统，但可以把小范围结果交付得更清楚。

也适合给客户做异步协作的人。客户不在线时，交付说明越清楚，对方越容易验收。

## 不适合谁

不适合把交付说明当成万能免责。说明写得再清楚，也不能替代真实验证和客户授权。

也不适合把生产部署、数据库处理、账号权限、长期维护默认包含进一个低价交付包。这些内容需要单独范围和复核。

## 具体步骤

第一，可以写进基础交付范围的是问题回顾。它帮助客户确认你处理的是同一个需求。

第二，可以写进基础范围的是修改摘要。摘要可以列文件、页面、配置、命令结果，但不用暴露不必要的内部细节。

第三，可以写进基础范围的是验证结果。你运行过哪些检查、通过了哪些、没条件运行哪些，都应该说明。

第四，可以写进基础范围的是未覆盖内容。客户最容易误解的就是“你顺便也会做其他事”，所以要主动排除。

第五，可以写进基础范围的是客户验收步骤。交付不是你说完就结束，客户还需要按步骤确认。

## 报价写法

```text
交付包含：
- 问题回顾
- 修改摘要
- 验证结果
- 未覆盖范围
- 客户验收步骤

不包含：
- 生产部署
- 数据库操作
- 账号权限处理
- 长期维护
- 新增需求
```

这段可以放进报价，也可以放进交付说明。它让客户从一开始就知道最终会收到什么。

## 什么时候要单独收费

如果客户要求你整理完整技术报告、录屏讲解、部署文档、长期维护计划或多轮验收支持，这已经超过普通交付说明。可以作为新增服务重新确认。

如果任务本身是诊断单，交付说明就是主要产出。它应该包含错误线索、已排除方向、需要客户补充的材料，而不是只写“暂时不能修”。

## 怎么把交付说明放进套餐

可以把基础套餐写成“修复一个明确问题，并提供简短交付说明”。交付说明包括修改摘要、验证结果、未覆盖范围和客户验收步骤。这样客户知道交付不是一句聊天回复。

如果客户需要更完整的文档，可以作为进阶项：包括截图、命令输出摘要、复现记录、风险说明和后续维护建议。进阶项要单独报价，不要默认包含。

## 范围复核话术

```text
本次交付说明会覆盖本次问题的处理结果和验收方式，不包含生产部署、长期维护或新增需求。如需继续处理新的问题，我们可以重新确认范围。
```

这句话适合放在报价或交付前确认里。它不生硬，但能把边界放到明面上。

## 客户验收支持的边界

基础交付可以包含一次验收说明和一次必要澄清。例如客户看不懂测试步骤，你可以解释在哪里点击、用哪组测试数据、预期看到什么结果。

但多轮远程协助、陪客户部署、排查客户本地环境、继续处理新报错，就应该算新范围。验收支持是帮客户确认本次结果，不是把所有后续问题都接进来。

## 风险提醒

不要承诺交付说明能覆盖所有未来问题。它只证明本次范围内的处理和验证。

不要在说明里写入客户敏感材料。需要引用日志时，用脱敏摘要和关键错误码。
