# Codex 是什么，新手第一天应该怎么用：新手检查清单

如果你正在搜索“Codex 是什么，新手第一天应该怎么用：新手检查清单”，可以先把目标压小：今天只完成一次低风险练习，不接真实客户项目，不碰生产环境，不把 AI 输出直接当成交付。

这份清单适合边做边勾选。你不需要第一天就学完所有开发知识，只要能让 Codex 解释项目、完成一个小修改、跑一次检查、写下记录，就已经比盲目接单稳很多。

## 适合谁

适合刚开始了解 Codex、Claude Code、ChatGPT、Upwork、Fiverr、GitHub 或 Vercel 的中文新手。你可以还不会完整写代码，但愿意按步骤验证工具输出，愿意保存操作记录，也愿意在看不懂客户需求时先停下来确认，而不是直接承诺结果。

也适合正在准备第一个作品集、练习小项目、尝试把 AI 工具用于辅助交付的人。重点不是“自动赚钱”，而是把问题拆小、把风险看清、把可以执行的下一步写出来。如果你愿意把每一次练习都记录成案例，后面才有可能形成可信的作品集。

## 不适合谁

不适合想批量复制文章、批量发送未经审核的 Proposal、规避平台规则、承诺无法完成的功能，或者把 AI 输出当成最终交付的人。也不适合希望马上获得稳定收入的人。自由职业接单需要作品、沟通、交付记录和长期信誉，不是单靠一个工具就能完成。

如果项目涉及客户生产数据库、支付、账号权限、安全漏洞、隐私数据或大规模线上服务，新手不应该独立处理。遇到这些情况，要么只做学习记录，要么找有经验的人一起判断。

## 第一部分：环境检查

先确认你是否能在本地或练习项目里看到真实反馈。不要跳过这一步，因为后面所有修改都依赖环境是否能跑起来。

- 能打开项目文件夹。
- 能找到 `package.json` 或项目说明。
- 能运行 `git status`，知道当前有没有未提交修改。
- 能运行 `npm install` 或知道依赖为什么安装失败。
- 能运行 `npm run build`，或者至少保存构建失败的完整错误。
- 能说清楚项目是练习项目、个人项目，还是客户项目。

如果这里卡住，不要急着让 Codex 改代码。先把错误复制到 [报错解释器](/tools/error-explainer)，或者让 Codex 解释错误含义。

## 为什么新手会遇到

新手容易把“AI 给了一个答案”误认为“项目已经能交付”。但第一天真正要训练的是反馈循环：我问了什么、AI 回了什么、我改了哪里、页面或命令输出是否证明它有效。

如果没有这个循环，你后面看到 Upwork/Fiverr 小项目时，很容易被“simple fix”“quick job”误导。很多简单任务只有在能复现、能验证、能回滚时才真的简单。

## 第二部分：提问检查

第一次向 Codex 提问时，先要求它解释，不要直接改文件。可以检查自己是否做到：

- 说明自己是新手。
- 明确要求“先不要修改文件”。
- 让它解释项目结构。
- 让它指出入口页面、样式文件、构建命令。
- 让它建议一个最低风险的小练习。
- 要求它说明可能影响哪些文件。

这一步的目的，是让你学会控制 AI 的工作范围。Codex 可以很积极，但新手更需要慢一点、可检查一点。

## 具体步骤

1. 选择一个练习项目，不要选择客户生产项目。
2. 运行基础命令并记录结果。
3. 让 Codex 解释项目结构，先不改文件。
4. 选择一个最小修改，例如改一段说明文案、按钮文字或卡片间距。
5. 修改后运行 `npm run build` 或本地预览。
6. 保存修改前后截图、命令输出和改动文件名。
7. 写下“我能解释清楚的部分”和“我还看不懂的部分”。

## 第三部分：最小修改检查

第一天的练习必须足够小。下面这些通常适合：

- 修改一个按钮文案。
- 调整一个标题层级。
- 给卡片增加一行说明。
- 调整一个很明确的间距。
- 修复一个不会影响业务逻辑的小样式问题。

下面这些不适合第一天尝试：

- 重做整个首页。
- 接入支付。
- 修改登录权限。
- 操作生产数据库。
- 自动生成大量文章并发布。
- 批量提交 Proposal。

## 可以复制的命令或模板

你可以复制下面这段作为第一天检查提示词：

```text
我是 Codex 新手。请先不要修改文件。
请用中文帮我做一个第一天上手检查：
1. 这个项目可能是什么类型。
2. 我应该先看哪些文件。
3. 我应该先运行哪些命令。
4. 哪个修改最小、风险最低。
5. 修改后我应该怎么验证。
6. 哪些操作今天不应该做。
```

如果是本地技术练习，可以先执行这些基础检查：

```bash
npm install
npm run build
git status
```

这些命令不会替你解决所有问题，但能帮助你确认项目是否能安装、是否能构建、当前文件是否有未提交修改。第一天能看懂这些输出，比生成一堆代码更重要。

## 常见错误

第一个错误是没有保留修改前状态。无论是截图、命令输出还是 git 状态，都要先记录，否则你很难说明自己到底改了什么。

第二个错误是让 Codex 同时改很多东西。第一天最好每次只改一个点，确认有效后再继续。

第三个错误是把练习当成客户交付。练习可以粗糙，客户交付必须清楚边界、验收和责任。

第四个错误是没有写复盘。当天结束前写 5 行：做了什么、哪里成功、哪里失败、下次问 Codex 要更具体什么、哪些任务不能碰。

## 风险提醒

不要承诺收入结果，不要承诺一定成交，不要用自动化工具批量投标，不要复制别人的文章、案例或截图。AI 辅助可以提高整理效率，但不能替代人工判断。任何报价、工具推荐和流程建议都只是参考，需要结合你的真实能力和客户的真实需求。

如果客户要求站外付款、使用 Telegram 或 WhatsApp 绕开平台、要求免费完成完整测试、要求接触敏感账号或生产数据，要提高警惕。看不懂风险时，最稳妥的动作是暂停。

第一天还要加一条红线：不要为了“看起来进度快”而删除你不理解的文件、关闭 ESLint、跳过 TypeScript 报错、或者把构建失败的项目说成已经完成。慢一点，但记录清楚，才是长期可用的接单训练。

## 总结

这份清单的核心不是让你第一天变成开发者，而是让你建立一个稳的起点：环境能检查，问题能描述，修改能缩小，结果能验证，风险能暂停。

如果今天只能勾完一半，也没关系。不要追求速度，先追求可复现。以后你真正判断小项目能不能做时，靠的就是这些基本动作。
