# AI 新手接项目前的项目适配检查清单

先给结论：AI 新手接项目前，不要只问“我能不能做”，而要逐项确认需求、材料、风险、验证方式、交付边界和沟通记录。清单越具体，越能避免误接复杂项目。

这份清单适合在写 Proposal 前使用。你可以把它复制到 Notion、表格或项目笔记里，每看到一个客户需求就填一次。填完后再决定投、不投，或者继续问问题。

## 适合谁

适合准备投第一个小项目的新手，尤其是想用 AI 辅助做网页修改、文案整理、报错解释、部署检查、简单页面或作品集练习的人。

也适合已经投过几次但总是判断不稳的人。很多问题不是工具不够强，而是你没有固定标准，每次都凭感觉判断。

## 不适合谁

不适合想直接用 AI 替自己决定项目是否能接的人。工具可以帮你拆解，但最终判断必须由人负责。

也不适合高风险项目。涉及支付、生产数据库、客户隐私、安全漏洞、账号高权限和真实业务中断的任务，新手不应该只靠清单就独立承诺。

## 具体步骤

按下面顺序检查，不要跳到报价：

1. 先确认客户到底要什么。
2. 再确认你拿到了哪些材料。
3. 然后判断风险等级。
4. 再写出验证结果的方法。
5. 接着写清交付边界。
6. 最后再决定是否投标和报价。

如果你在第 1 到第 4 步就卡住，不要急着写 Proposal。先向客户提问。

## 需求清晰度检查

```text
需求是否清楚：
1. 我能用一句话复述客户需求。
2. 客户给了当前问题截图或链接。
3. 客户说明了期望结果。
4. 客户说明了这是测试环境还是线上环境。
5. 客户说明了截止时间。
6. 客户说明了验收标准。
7. 我知道本次只改一个问题，还是要做整体检查。
```

如果第 1 项做不到，先不要投。你连客户要什么都说不清，后面报价和交付都会变形。

## 材料完整度检查

```text
已有材料：
1. 当前页面链接：有 / 没有
2. 截图或录屏：有 / 没有
3. 仓库或代码片段：有 / 没有
4. 报错日志：有 / 没有
5. 技术栈说明：有 / 没有
6. 部署平台说明：有 / 没有
7. 测试账号或测试数据：有 / 不需要 / 不适合索要
```

材料不完整时，不代表不能接，但代表不能马上承诺。你可以先发确认问题，而不是直接报价。

## 风险等级检查

```text
低风险信号：
- 静态文案修改
- CSS 样式微调
- 简单页面布局
- 报错解释
- 本地 demo 练习

中风险信号：
- 需要改现有代码仓库
- 需要部署检查
- 需要第三方服务配置
- 需要客户提供账号权限
- 需要在短时间内交付

高风险信号：
- 支付
- 生产数据库
- 安全漏洞
- 用户隐私
- 大规模后端
- 线上业务中断
- 平台外结算请求
```

低风险可以继续评估，中风险要缩小范围，高风险通常不适合新手独立接。

## 验证能力检查

```text
我能如何证明结果：
1. 可以本地运行。
2. 可以截图前后对比。
3. 可以运行构建命令。
4. 可以复现报错并确认消失。
5. 可以打开目标页面检查。
6. 可以让客户在测试环境确认。
7. 可以写清哪些内容没有验证。
```

如果你没有验证方法，就算 AI 给了答案，也不要把项目当成可交付。

## 交付边界检查

```text
本次交付包含：
- ____
- ____
- ____

本次交付不包含：
- ____
- ____
- ____

需要客户确认：
- ____
- ____
- ____
```

边界写不出来，说明你还没有真正理解项目范围。不要用“small fix”“simple task”代替具体范围。

## 可以复制的提问模板

```text
Thanks for sharing the details. Before I confirm the scope, I need to check a few points:
1. Could you share the current page link or screenshots?
2. What should the final result look like?
3. Is this a test environment or a live production site?
4. Do you have the error message, repository, or steps to reproduce?
5. Should this task cover only the listed issue, or do you expect a broader review?

I will confirm the scope and estimate after reviewing these details.
```

这段模板的重点是先拿信息，再确认范围。不要在信息不足时承诺报价和工期。

## 常见错误

第一个错误是跳过材料检查。客户说得越短，你越应该问清楚，而不是让 AI 猜。

第二个错误是把中风险项目当低风险项目。只要需要权限、部署或第三方配置，就要谨慎。

第三个错误是没有写“不包含”。不写不包含，客户可能默认你会解决相关所有问题。

第四个错误是没有留下判断记录。每次投标前填一次清单，以后你会更容易总结哪些项目适合自己。

## 风险提醒

不要承诺收入结果，不要承诺一定成交，不要承诺无法验证的功能，不要把客户高权限账号、真实密钥或隐私数据交给不确定的工具。AI 可以帮你准备材料，但不能替你承担承诺。

如果客户提出平台外结算请求、免费完成完整项目测试、生产数据库访问或高权限账号要求，先暂停。新手更适合从低风险、小范围、可验证的任务开始。

## 总结

AI 新手接项目前，先填清单，再写 Proposal。需求清楚、材料完整、风险可控、结果可验证、边界可说明，才适合继续。只要其中一项缺失，就先问问题或暂缓，不要硬接。
