# 用 Codex 做小网站交付前检查

小网站交付看起来简单：页面能打开、内容差不多、按钮能点，就想发给客户。但真正交付前，还要确认移动端、表单、链接、SEO、隐私信息、部署环境、404、加载状态和客户验收记录。Codex 可以帮助你列检查项、审查文案和解释报错，但最终判断必须由人工完成。

这篇文章适合落地页、作品集页、服务介绍页、模板站、小型 Next.js 或静态网站。若你只需要一张表，可以看 [小网站交付检查清单](/blog/codex-small-website-delivery-check-checklist)；如果已经遇到客户返工、移动端坏掉或上线后才发现错链，可以看 [常见错误和修复顺序](/blog/codex-small-website-delivery-check-mistakes)。

## 适合谁

适合已经能用 Codex、Cursor、Claude Code 或普通编辑器完成小页面，但还没有形成交付检查习惯的新手。你可能会写页面，也能本地预览，但不知道发给客户前要检查哪些细节。

也适合准备做 AI 工具实践的人。客户最终收到的不是你的 prompt，而是一个可打开、可理解、可验收的网站。交付检查能把“我做完了”变成“我验证过这些路径，并记录了剩余风险”。

## 不适合谁

不适合把预览截图当成最终交付的人。截图好看不代表链接、表单、移动端和部署都正常。

也不适合处理高风险系统的新手独立承接，例如登录后台、支付流程、订单系统、生产数据库、真实用户数据和权限系统。小网站检查流程不能替代专业安全审查。

## 风险提醒

小网站常见风险不是大崩溃，而是细节漏掉：手机上按钮遮挡、联系表单无效、外链打不开、图片未授权、客户电话写错、域名还没解析、sitemap 暴露草稿、隐私政策缺失、页面标题仍是模板文案。这些问题会直接影响客户信任。

项目场景还要注意授权和账号。部署平台、域名、邮件服务、表单收件、统计工具都可能需要客户登录或付费。遇到注册、授权、购买、DNS 修改等事项，先记录为客户确认或明日待办，不要替客户擅自操作。

## 具体步骤

1. 确认交付范围。列出本次包含哪些页面、哪些功能、哪些浏览器或设备、哪些内容由客户提供。范围不清，检查也会失焦。

2. 本地跑基础命令。至少运行 lint、build 或项目约定的检查命令。若报错看不懂，先用 [报错解释器](/tools/error-explainer)拆日志。

3. 检查页面路径。首页、关于页、服务页、联系页、模板页、工具页、404 和所有客户要求的链接都要打开一次。

4. 检查移动端。用浏览器响应式视图或真实手机看关键页面。重点看按钮、导航、表单、长标题、图片比例和横向滚动。

5. 检查表单和 CTA。联系按钮、邮件链接、下载按钮、报价入口和外部链接都要验证。若表单需要客户邮箱、API key 或第三方服务授权，列为待确认。

6. 检查公开信息。标题、描述、logo、客户名称、版权年份、联系方式、隐私政策、免责声明和社交分享图都要复核。

7. 写交付记录。记录测试过的页面、命令、截图、未覆盖范围和客户待确认事项。报价和范围拆分可用 [报价计算器](/tools/pricing-calculator)，客户回复初稿可用 [Proposal 生成器](/tools/proposal-generator) 后人工修改。

## Codex 可以帮什么

Codex 很适合帮你生成检查清单、阅读构建报错、找页面里的模板文案、列出可疑链接、整理交付说明。它也可以根据项目结构提醒你哪些页面可能没检查到。

但它不能替你确认真实手机显示、客户授权、域名解析、付款配置、邮件收件和用户实际验收。凡是涉及外部账号、付费服务和客户私有信息的部分，都要人工确认。

## 交付记录模板

```text
项目名称：
交付范围：
已检查页面：
已运行命令：
移动端检查：
表单和链接：
SEO 和公开信息：
未覆盖范围：
客户待确认：
下一步建议：
```

可以把这份记录放进 [模板库](/templates) 的交付记录里，后续每个小网站项目都沿用同一格式。
