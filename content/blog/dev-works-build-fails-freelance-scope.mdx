# npm run dev 能跑但 build 失败使用前怎么判断是否适合

“本地 dev 能跑但 build 失败”可以做成小型诊断服务，但新手不能直接承诺上线成功。build 失败可能来自代码、类型、lint、环境变量、部署平台、路径大小写、依赖版本或客户权限。范围不清时，越急着修越容易越界。

更稳的接法是先做 build 失败诊断，再决定是否进入修复。诊断交付包括失败阶段、完整日志解读、可能原因、低风险下一步、客户待确认事项。具体流程见 [npm run dev 能跑但 build 失败怎么办](/blog/dev-works-build-fails)。

## 适合谁

适合能读取终端日志、能运行 build、能看 `package.json`、能检查环境变量名称、能写交付记录的新手。你不需要一开始就能修所有问题，但要能判断哪些信息还缺。

也适合已有作品集或练习记录的人。客户项目里，过程透明比一句“我会修”更可靠。

## 不适合谁

不适合客户不给日志、不给仓库、不给部署信息，却要求你固定时间修好的任务。也不适合需要你直接登录客户生产部署平台、修改真实环境变量或处理生产数据的新手独立承接。

如果任务涉及支付、认证、企业后台、生产数据库、私有 token、客户部署后台，先暂停并记录为待确认事项。

## 风险提醒

build 失败项目风险在于验收标准容易变大。客户最初可能只说 build failed，后来变成“顺便帮我上线”“顺便修页面”“顺便配环境变量”。新手要把诊断、修复、部署、上线验证分开报价和交付。

另一个风险是敏感配置。环境变量名称可以讨论，具体密钥值不应由你保存。客户应在自己的平台中配置敏感变量，你只提供说明和检查结果。

## 具体步骤

### 1. 先收材料

客户至少要提供：完整 build 日志、项目类型、包管理器、Node 版本、部署平台、是否有环境变量、是否能本地复现。如果缺少这些，只能先问问题。

可以用 [Proposal 生成器](/tools/proposal-generator) 起草确认问题，再人工压缩成简短回复。

### 2. 定义第一档服务

第一档建议叫“build 失败诊断”。交付内容是失败阶段、原因判断、待确认事项和下一步建议。不直接承诺修复。

这档服务适合新手，也可以在不接触客户敏感账号的情况下完成。

### 3. 判断是否进入修复

只有当仓库权限清楚、验证命令明确、环境变量边界清楚、客户允许修改文件时，才进入修复。修复前要确认是否新建分支、是否允许改配置、是否需要部署验证。

如果客户没有测试方式，先补测试方式，再谈修复。

### 4. 报价拆分

报价至少拆成四段：日志阅读、原因定位、修复尝试、验证记录。若还要部署到 Vercel 或其他平台，要把部署配置和线上验证单独列出。

用 [报价计算器](/tools/pricing-calculator) 可以帮助你把诊断和验证时间写清楚。

## 可以接的任务

可以考虑接：作品集 build 失败；开源小项目构建错误；客户提供完整日志，只需要诊断原因；静态站部署前检查；页面代码小改后 build 不通过。

这些任务共同点是风险低、材料清楚、可本地复现、验证命令明确。

## 应该暂停的任务

应该暂停：客户要求你直接登录生产平台；需要处理真实密钥；需要访问私有数据库；没有日志却要求修复；build 失败和上线业务同时绑定；客户不允许保存交付记录。

暂停时可以交付待确认清单，而不是继续猜。

## 交付记录模板

```text
任务：build 失败诊断
原始日志：
失败阶段：
可能原因：
已检查内容：
建议修复：
验证命令：
客户待确认：
下一步：
```

这份记录可以放进 [模板库](/templates) 的项目诊断表，也可以作为客户沟通附件。

## 报价边界怎么写

报价时不要把所有内容写成“修复 build”。更清晰的写法是分成三层。第一层是诊断：阅读日志、定位失败阶段、整理原因和待确认事项。第二层是修复：在授权范围内修改文件，并运行本地验证命令。第三层是部署协助：检查平台配置、环境变量和线上结果。

新手更适合先接第一层。如果客户要求第二层或第三层，要确认仓库权限、分支策略、环境变量由谁配置、是否允许你访问部署后台。涉及账号、token、生产配置、付费服务的内容，先记录为待确认，不要把它们含糊地包进报价。

交付文字也要避免过度承诺。可以写“我会先定位 build 失败阶段并给出可执行建议”，不要写“保证部署成功”。技术服务应当靠证据、验证和边界收尾。

## CTA：下一步

先把任务写成“build 失败诊断”，收集完整日志、部署平台、环境变量边界和验证命令。材料齐了，再判断是否进入修复。
