# Claude Code 安装前置条件检查清单

安装 Claude Code 前，不要只盯着安装命令。先确认系统、终端、Node.js、Git、网络、测试项目和权限边界。很多“安装失败”其实是环境没准备好，或者把客户项目问题和工具安装问题混在了一起。

建议先读 [安装 Claude Code 前新手要先确认哪些条件](/blog/claude-code-install-prerequisites)，再搭配 [Claude Code 常用命令检查清单](/blog/claude-code-common-commands-checklist) 和 [/templates](/templates) 使用。

## 适合谁

适合第一次安装 Claude Code 的新手，也适合准备帮客户做低风险代码诊断的人。你需要先证明自己的基础环境可用，再碰客户项目。

也适合安装失败后复盘。把清单补齐，才能知道问题来自系统、网络、权限、项目还是命令。

## 不适合谁

不适合无法确认电脑权限、无法安装依赖、不能使用终端或没有稳定网络的人。先解决基础条件，再谈工具。

也不适合直接在客户生产项目上试错。安装和练习应该先在自己的测试项目里完成。

## 具体步骤

1. 系统是什么？Windows、macOS、Linux 或 WSL 要写清楚。
2. 终端是什么？PowerShell、Terminal、WSL、Git Bash 的路径和命令表现不同。
3. Node.js 是否已安装？记录版本，不要只说“应该装了”。
4. npm、pnpm 或 yarn 是否可用？包管理器版本也要记录。
5. Git 是否可用？至少能运行 `git status`。
6. 当前项目是否干净？有未提交改动时，不要乱试生成或格式化命令。
7. 网络是否稳定？安装包、登录和更新都可能受网络影响。
8. 账号是否是自己的？不要使用客户账号、验证码或私密 token 来练习安装。
9. 是否有测试项目？先在低风险项目里验证工具。
10. 是否写清本次目标？学习、诊断、修改、提交、部署是不同层级。

## 安装记录模板

```text
系统：
终端：
Node：
包管理器：
Git：
测试项目：
错误原文：
已尝试步骤：
下一步：
```

这份记录可以直接发给自己复盘，也可以在需要时脱敏后请别人帮忙判断。

## 判断结果分类

如果系统、终端、Node、Git、网络和测试项目都清楚，可以进入安装或功能验证。如果其中一项不清楚，先补基础环境。如果涉及客户账号、私有仓库、生产配置，就暂停并确认授权。

分类要写出来。新手最容易一边安装、一边排客户项目、一边处理权限，最后不知道问题来自哪里。

## 安装后第一步

安装成功后，不要马上打开客户项目。先在自己的测试项目里做三件事：读取文件、解释一个错误、生成一个小范围修改建议。确认这些动作可控后，再考虑客户任务。

如果测试项目都不能稳定运行，就不要把问题带进客户环境。先修本机环境，才是最快的路。

## 客户沟通模板

```text
我会先确认本机环境和 Claude Code 基础可用性，再处理你的项目。安装准备阶段不需要你提供账号密码、验证码或私有 token。若后续涉及授权，请你在自己的环境中完成。
```

这段话适合客户不理解为什么要先做环境检查时使用。它能把安全边界讲清楚。

## 交付记录示例

```text
已确认：Windows + PowerShell、Node 版本、Git 可用
已验证：测试项目可读取文件并生成诊断建议
未处理：客户私有仓库、生产部署、账号登录
下一步：客户提供脱敏项目错误后进入诊断
```

安装准备也可以交付清楚结果，而不是只说“环境弄好了”。

## 复查频率

工具升级、Node 版本变化、换电脑、换终端后，都应该重新走一遍清单。不要因为上个月能用，就默认现在环境仍然可用。

如果你准备用它接客户单，建议每次项目前至少确认 Git 状态、Node 版本和测试项目。三分钟检查，能省下很多解释成本。

## 风险提醒

不要保存客户账号、验证码、token 或私有仓库内容。安装阶段只需要确认自己的工具可用。

不要在不清楚影响的情况下运行会删除、重置、迁移或部署的命令。安装检查不是生产运维。
