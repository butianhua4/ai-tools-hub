# Claude Code 自动化接单前怎么判断能不能做

客户说“能不能用 Claude Code 自动帮我处理一下”，新手不能只看工具能不能生成结果。接单前要判断：任务是否低风险、是否可复现、是否能人工审核、是否有测试证据、是否能写清交付边界。只要其中一项不清楚，就不要直接报价为完整修复。

这篇适合和 [Claude Code 自动化人工审核检查清单](/blog/claude-code-automation-human-review-checklist)、[第一次 AI 接单检查清单](/blog/before-first-ai-freelance-job-checklist)、[/tools/proposal-generator](/tools/proposal-generator) 搭配使用。

## 适合谁

适合准备用 Claude Code 辅助处理低风险小单的新手。比如页面小改、报错诊断、文档整理、配置说明、测试环境问题、代码阅读报告。这些任务能拆小，也能人工检查。

也适合收到客户模糊需求后，需要先判断范围的人。自动化不是报价理由，明确范围和验收方式才是报价基础。

## 不适合谁

不适合客户要求工具直接接管生产系统、处理真实用户数据、修改付款后台、保存密钥、绕开授权流程或长期自动执行的任务。这些需求不适合新手独立承接。

也不适合客户拒绝提供复现步骤，却要求你保证结果。没有错误、路径、环境和验收标准，自动化只能猜测，不能可靠交付。

## 具体步骤

第一步，确认任务是否低风险。能在测试环境完成、能用脱敏数据验证、能通过截图或命令证明的任务更适合接。

第二步，确认是否可复现。客户要提供触发路径、错误信息、截图、预期结果和实际结果。不能复现时，先报价诊断，不报价完整修复。

第三步，确认人工审核位置。你要知道哪些步骤由 Claude Code 辅助，哪些步骤必须人工看 diff、跑检查、写说明。

第四步，确认权限边界。不接收客户完整密钥，不代管账号，不处理不必要的生产数据。确实需要授权时，应让客户使用最小权限和可撤销方式。

第五步，确认交付方式。交付不应只是“自动化已执行”，而应包含修改内容、测试证据、未覆盖范围和客户验收步骤。

## 可以接的范围

适合新手的范围包括：生成排查报告、整理错误原因、修一个明确组件、补一份配置文档、验证测试环境流程、根据客户素材整理交付说明。这些任务有清楚边界，也能留下证据。

不建议接的范围包括：自动改完整系统、长期监控生产环境、处理账号权限、批量处理客户真实数据、修改支付或账单设置、独立处理安全事故。工具越强，越要知道哪里不能碰。

## 风险提醒

不要把“自动化”写成卖点里的绝对承诺。更稳的表达是：“我可以用工具辅助诊断和小范围修改，但所有结果会经过人工检查和测试验证。”

不要承诺客户业务结果。Claude Code 可以帮你处理代码和文档，不能保证客户成交、增长、审核通过或系统永远稳定。

## 客户确认模板

```text
我可以先判断这个任务是否适合用 Claude Code 辅助处理。为了控制风险，请先确认：
1. 问题在哪个环境出现？
2. 是否有错误截图或日志？
3. 是否可以使用测试数据？
4. 是否涉及密钥、付款、生产数据或管理员权限？
5. 你希望看到什么验收结果？
```

如果客户愿意配合回答，任务通常更容易拆清楚。如果客户只要求“你直接自动处理”，就要谨慎。

## 报价边界

自动化任务更适合拆成两段报价：先诊断，再修复。诊断阶段交付问题范围、风险判断和建议材料；修复阶段只处理确认过的小范围问题。这样可以避免客户把“自动化”理解成无限修改。

报价里要写清不包含项：不代管账号、不保存密钥、不处理真实生产数据、不长期监控、不承诺客户业务结果。新手越早写清这些内容，越容易保护时间和责任边界。

## 交付验收

交付时不要只写“自动化完成”。要写：工具辅助了什么、人工检查了什么、运行了什么命令、客户需要按什么路径验收。没有这些信息，就算代码暂时能跑，客户也很难判断任务是否完成。
