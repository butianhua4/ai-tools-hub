# 小网站上线检查 SOP 怎么写

小网站上线不是只看首页能打开。对新手来说，SOP 要覆盖内容、链接、移动端、表单、SEO 基础、隐私说明、部署、回滚和验收。否则上线后才发现问题，会很被动。

这篇仍是草稿，发布前需要人工补充真实上线截图和部署平台规则核对。

## 适合谁

适合准备上线个人作品集、服务页、模板下载页、练习项目或客户测试站的新手。你需要一套上线前固定动作。

也适合用 Codex 辅助改站点的人。AI 能帮你修改代码，但上线检查要靠证据。

## 不适合谁

不适合大型商业系统、支付站、会员系统、生产数据库和高合规行业。那些需要更完整的发布流程。

如果涉及客户域名、账号、环境变量或生产发布，必须先确认授权。

## SOP 包含什么

建议分成十项：

1. 内容：标题、文案、CTA、错别字。
2. 链接：导航、按钮、站内外链接。
3. 移动端：手机宽度下是否可读。
4. 表单：提交、错误提示、隐私提醒。
5. SEO：title、description、canonical、sitemap、robots。
6. 安全：没有密钥、客户数据、未授权素材。
7. 构建：lint、build、测试。
8. 部署：预览链接、生产链接、环境变量名称。
9. 回滚：上一版在哪里，怎么恢复。
10. 验收：客户或自己确认什么。

## 具体步骤

1. 先在本地跑检查。
2. 生成预览链接，不直接改生产。
3. 用桌面和手机查看核心页面。
4. 点击所有主要按钮和表单。
5. 检查 sitemap 和 robots。
6. 确认隐私页和免责声明是否存在。
7. 记录部署版本和回滚方式。
8. 最后写上线记录。

## 关于流量统计

上线不等于有流量。要判断流量，需要接入 Search Console、Analytics、Vercel Web Analytics 或其他统计工具，并等待数据产生。没有统计数据时，只能确认页面可访问，不能说已经有流量。

如果需要注册、验证域名、接入 Analytics 或 Search Console，先写进明日待办，由账号拥有者完成授权。

## 上线后的 24 小时

上线后第一天，不要急着改一堆东西。先检查页面是否稳定、表单是否能提交、主要链接是否正常、sitemap 和 robots 是否可访问。如果有统计工具，再看是否开始记录访问；如果没有，就先完成接入待办。

同时保留上线记录：提交版本、部署链接、检查命令、发现的问题和下一步。这样以后遇到页面异常，可以知道从哪一版开始排查。

小站上线是一个循环，不是一次按钮。上线前检查、上线后复查、再根据数据和反馈调整，才是稳定流程。

## 当前没有统计时怎么判断

没有 Analytics、Search Console 或 Vercel Web Analytics 时，能判断的只有技术可用性。比如页面能打开、构建通过、公开页面数量符合预期、草稿没有出现在 sitemap、主要链接没有明显错误。这些都是上线健康检查，不等于访问量。

如果要回答“有没有流量”，必须看统计工具或搜索平台数据。没有接入前，可以把任务拆成两步：第一步先接入或申请权限，第二步等数据产生后再看趋势。不要把浏览器自己打开页面、部署成功或 sitemap 存在当成真实流量证据。

对小站来说，早期更重要的是把可访问性、索引边界和内容质量先守住。流量数据接入后，再决定哪些页面值得继续扩写、哪些入口需要改标题和描述。

这也是为什么上线记录里要区分“页面健康”和“流量表现”。前者今天就能检查，后者必须等真实数据来源。

## 风险提醒

不要公开草稿、客户数据、密钥、后台截图和未授权素材。不要把预览链接当生产发布。不要承诺搜索流量或排名结果。
