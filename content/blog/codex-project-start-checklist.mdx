# Codex 项目启动检查表怎么用

Codex 能帮助你阅读代码、修改文件、运行命令和整理交付记录，但它不能替你理解客户目标，也不能替你确认权限边界。新手用 Codex 接触项目时，最重要的不是马上让它写代码，而是先把项目启动信息整理清楚。

这份检查表适合用在三类场景：你自己的练习项目、作品集项目、小范围客户任务。无论是哪一种，都先确认仓库、目标、运行命令、测试方式、数据边界和交付记录。需要模板可以看 [模板库](/templates)，需要估算服务时间可以用 [报价计算器](/tools/pricing-calculator)。

## 适合谁

适合已经能打开项目目录、能读 README、能运行基本命令的新手。你不需要完全掌握项目里的每一行代码，但要能说清当前任务是什么、你准备让 Codex 做什么、你如何验证结果。

也适合准备把 AI 辅助编程写进项目流程的人。客户通常不关心你用了哪个工具，而关心结果是否符合需求、是否能运行、是否有记录、是否没有越过权限边界。

## 不适合谁

不适合拿到客户仓库后立刻让 Codex 大范围改代码的人。没有目标、测试命令和回滚方式，AI 改得越快，后续越难解释。

也不适合处理生产数据、支付逻辑、身份认证、客户私有密钥、企业内部系统的新手独立推进。遇到登录、连接仓库、授权、安装付费工具或访问私有资源的事项，先记录为待确认或明日待办。

## 风险提醒

Codex 项目启动的风险集中在三处：权限过大、上下文不完整、验证不足。权限过大指你允许工具接触它不需要的文件、账号或数据；上下文不完整指你没有给清楚目标和验收标准；验证不足指你只看 AI 说完成，却没有跑测试、看 diff、读页面或检查日志。

如果项目里出现 `.env`、token、客户凭据、私有仓库、生产数据库、真实用户数据，不要复制到聊天里，也不要让 AI 自行决定怎么处理。先做记录，等客户或负责人确认。

## 具体步骤

1. 写一句任务目标：例如“修复移动端按钮溢出”或“整理 README 安装步骤”，不要写成“帮我优化一下”。
2. 记录项目位置：本地路径、仓库名、当前分支、是否有未提交改动。
3. 阅读入口文件：README、package.json、配置文件、路由入口、测试说明。
4. 记录运行命令：安装依赖、启动、lint、test、build。没有命令就写“未确认”，不要假设。
5. 定义不可触碰范围：密钥、生产数据、客户账号、支付、权限配置、部署后台。
6. 让 Codex 先解释计划，再小步修改。每次只处理一个明确问题。
7. 修改后检查 diff、运行验证命令、保存结果和剩余风险。

## 检查表字段怎么填

第一栏写“任务目标”，用一句可验证的话。目标不能只有形容词，比如“更好看”“更智能”“更专业”。更稳的写法是“把报价页移动端卡片从两列改为一列，并确保按钮不换行”。

第二栏写“当前证据”。证据可以是报错日志、截图、客户消息、失败命令、页面路径或测试结果。没有证据时，Codex 很容易围绕猜测工作。

第三栏写“验证方式”。前端任务可以用截图、浏览器检查和 build；后端任务可以用测试、接口请求和日志；文档任务可以用链接、目录和命令复跑。

第四栏写“暂停条件”。只要出现客户账号、真实密钥、生产数据库、第三方授权、付费工具或权限不清，就先暂停，把事项写进待确认。

## 和 Codex 沟通时怎么写提示

一个好的启动提示可以包含：目标、当前文件、限制、验证命令和交付格式。示例：

```text
请先阅读 README 和 package.json，确认这个项目如何启动。
目标：修复移动端报价页按钮溢出。
限制：不要修改认证、支付、环境变量和部署配置。
验证：修改后运行 lint 和 build，并总结改了哪些文件。
```

这类提示不会让结果自动正确，但能减少无关改动。你仍然要看 diff 和验证结果。

## 项目前怎么用

做项目时，把检查表变成客户确认问题：项目仓库在哪里；是否有测试环境；目标页面或报错是什么；验收标准是什么；哪些文件不能动；是否涉及账号或密钥；客户是否接受先诊断再报价。

如果客户只能给模糊描述，可以先接“项目启动诊断”小单，不直接承诺修完。交付内容是项目结构、风险、可执行下一步和待确认事项。
