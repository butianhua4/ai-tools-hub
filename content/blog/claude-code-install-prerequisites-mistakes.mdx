# Claude Code 安装准备常见错误

安装 Claude Code 前，新手最容易犯的错是“先试了再说”。结果命令跑了一堆，错误原文没记录，系统和终端没确认，Node 版本不知道，还把客户项目问题误判成工具安装失败。安装准备看起来慢，其实是在给后面的排查提速。

这篇适合搭配 [Claude Code 安装前置条件检查清单](/blog/claude-code-install-prerequisites-checklist) 和 [Claude Code 常用命令检查清单](/blog/claude-code-common-commands-checklist) 使用。

## 适合谁

适合第一次安装 Claude Code，或者已经安装失败但不知道哪里出错的新手。你需要把环境信息补齐，而不是不断换命令。

也适合准备接客户小单的人。你要先确认自己的工具链稳定，再处理客户问题。

## 不适合谁

不适合没有电脑安装权限、没有稳定网络、无法使用终端或不愿记录错误的人。基础条件不清楚时，继续试只会更乱。

也不适合把客户项目当成安装测试项目。客户项目应该在你本机工具可用后再打开。

## 具体步骤

错误一：不确认系统。Windows、macOS、Linux、WSL 的路径和权限不同，排查方式也不同。

错误二：不确认终端。PowerShell、Terminal、Git Bash、WSL 的命令行为可能不一样，不能混着说。

错误三：不看 Node 版本。安装和项目运行都可能依赖版本，先记录版本，再判断问题。

错误四：不记录错误原文。只说“失败了”没有排查价值。要记录完整但脱敏的错误输出。

错误五：直接在客户项目里试错。安装阶段应该先用自己的测试项目，不要在客户仓库里尝试未知命令。

错误六：混淆安装问题和项目问题。工具安装成功，但客户项目构建失败，这是两类问题。

## 修正模板

```text
我先确认基础环境：
- 系统和终端
- Node 和包管理器版本
- Git 是否可用
- 测试项目是否能正常读取
- 错误原文
确认这些后，再判断是安装问题还是项目问题。
```

这段可以作为自己的排查流程，也可以在需要时发给客户说明为什么先做环境确认。

## 怎么避免越试越乱

安装失败后，先停下来整理记录。不要连续复制多个来源的命令。每多试一条命令，都可能增加新的变量。

更稳的方式是一次只改一个条件：比如先换终端，再换 Node 版本，再检查网络。每一步都记录结果，Claude Code 或其他助手才有足够上下文帮你判断。

## 客户任务前怎么自检

接客户项目前，先在自己的测试项目里确认三件事：能读取文件、能解释错误、能生成小范围修改建议。如果这三件事做不到，就先别碰客户项目。

还要确认当前工作区是否干净。有未提交改动时，安装或格式化命令可能让问题变复杂。

## 改写示例

不推荐写：“我装不上，可能是工具问题。”这句话没有环境信息。

更稳的写法是：“当前在 Windows PowerShell 中安装失败，Node 版本已记录，Git 可用；错误出现在依赖下载阶段，下一步先确认网络和包管理器配置，暂不进入客户项目。”这段话能让别人真正帮你判断。

## 复盘怎么做

每次安装问题解决后，记录最后有效的一步。是换了终端、修了 Node 版本、调整了网络，还是发现项目缺配置？下次遇到相似问题，可以先查这份记录。

如果问题来自客户项目，也要分开写。安装成功是一条记录，客户项目失败是另一条记录，别混成一团。

## 什么时候暂停求助

如果错误里出现账号、token、私有仓库地址或客户内部路径，先暂停，把敏感内容遮掉再求助。不要为了快点解决安装问题，把客户信息暴露出去。

如果你解释不清自己运行过哪些命令，也先暂停整理记录。没有记录时，别人只能陪你从头猜，效率很低。

## 风险提醒

不要让客户把账号、验证码、密钥或私有 token 发给你帮助安装。涉及账号的动作，应由账号所有者自己完成。

不要在安装准备阶段处理生产部署、数据库、环境变量或客户真实数据。那已经不是安装问题。
