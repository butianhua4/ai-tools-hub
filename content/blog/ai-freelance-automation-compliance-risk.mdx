# AI 自动化项目有哪些合规风险

先给结论：AI 自动化可以用来提高整理效率，但不能用来不遵守平台规则、替代人工审核、批量联系客户或发布未经审核内容。对新手来说，安全的自动化是“帮我准备材料”，危险的自动化是“替我做决定并对外行动”。

如果你正在做 AI 工具实践网站、Upwork Proposal 工具、Codex 报错解释器或 500 篇 SEO 草稿系统，最重要的不是速度，而是审核机制。所有文章默认 draft，所有投标需要人改，所有客户沟通需要人确认，这些规则看似慢，其实是在保护网站和账号。

## 适合谁

适合想用 Codex、Claude Code、ChatGPT、Zapier、Make、GitHub Actions 或 Vercel 自动化提高效率的新手。你可以让工具生成选题、整理草稿、检查 SEO、提醒你复核，但你需要保留最终决定权。

也适合担心“是不是以后要注册很多平台、是不是能被动收款”的人。自动化不是让你马上接很多平台，而是先把流程做清楚：哪些可以自动，哪些必须人工确认。

## 不适合谁

不适合想让系统自动投标、直接发送未经审核的客户消息、自动发布 500 篇文章、自动伪装案例或自动处理客户敏感数据的人。这些行为很容易伤害平台账号、网站信誉和用户信任。

也不适合想用 AI 输出直接替代专业判断的人。涉及法律、财务、税务、安全、支付、生产数据库和隐私数据时，新手不要靠自动化硬做。

## 这个问题是什么

AI 自动化项目的合规风险，主要来自三个地方：平台规则、内容质量、客户信任。

平台规则方面，Upwork、Fiverr、Google、GitHub、Vercel 等平台都有各自限制。比如客户沟通、支付、账号权限、内容索引、自动化访问，都可能有边界。

内容质量方面，自动生成不等于原创合格。大量草稿可以存在，但不能未经审核就公开。每篇文章都应该有具体问题、可执行步骤、风险提醒、适合谁和不适合谁。

客户信任方面，AI 可以帮你写初稿，但不能替你承担承诺。客户如果问“你能不能做”，你必须基于真实能力回答，而不是让工具替你夸大。

## 为什么新手会遇到

第一，新手容易把“自动化”理解成“完全不用管”。真正可靠的自动化通常是把重复任务标准化，比如检查 frontmatter、生成草稿、跑 build、检查 sitemap，而不是替你公开发布或联系客户。

第二，新手容易追求数量。500 个选题计划是内容库，不是 500 篇马上公开的文章。批量生成可以节省准备时间，但批量发布未经审核内容会让网站质量变差。

第三，新手容易忽略账号后果。自由职业平台、广告平台、搜索平台都看重规则和信任。一次错误自动化，可能比手动慢一点的成本更高。

## 先判断你属于哪种情况

如果自动化只发生在本地或草稿阶段，例如生成 MDX 草稿、记录 content-log、运行质量检查，这是低风险方向。

如果自动化会对外产生行为，例如发布文章、提交 Proposal、给客户发消息、部署生产站、修改付款信息，就必须有人工确认。

如果自动化会处理敏感信息，例如 API Key、客户账号、支付数据、用户隐私、生产数据库，新手应该默认停下来，先咨询有经验的人或只做文档化建议。

## 具体步骤

第一步，给所有自动化任务分级。低风险是整理、检查、提醒；中风险是生成草稿、生成 Proposal 初稿、生成报价建议；高风险是对外发布、对客户发送、修改生产环境。

第二步，给高风险动作加人工确认。例如发布文章必须 `status=review` 且 `qualityScore>=80`，真正发布还必须带 `--confirm`。投标也应该先人工阅读客户需求，再改 Proposal。

第三步，把默认状态设成安全。文章默认 draft/noindex；模板付费默认即将上线；AI API 默认模拟；广告位默认占位；邮箱表单默认前端占位。

第四步，保留日志。生成草稿写入 content-log，发布写入 publish-log，自动化维护写进 docs。出问题时能追踪，不靠记忆猜。

第五步，定期抽查公开页面。检查 `/blog` 是否只展示已发布文章，`/sitemap.xml` 是否没有草稿，`robots.txt` 是否指向正确 sitemap。

## 可以复制的合规分级表

```text
允许自动化：
- 搜集选题
- 生成原创草稿
- 检查 SEO
- 检查质量分
- 更新工具数据草稿
- 检查死链
- 创建待审核 Pull Request

必须人工确认：
- 发布文章
- 修改 noindex
- 提交 Proposal
- 回复客户
- 部署影响生产的配置
- 接入收款或广告

禁止自动化：
- 复制别人文章
- 洗稿
- 直接发送未经审核的客户消息
- 自动投标
- 伪造案例
- 抓取平台禁止抓取的数据
```

## 常见错误

第一个错误是把 `draft` 当成公开内容。草稿可以很多，但只有人工审核后才能公开。否则 sitemap 会变得臃肿，搜索引擎也会看到不成熟内容。

第二个错误是自动提交 Proposal。Proposal 是对客户的承诺，必须基于真实能力修改。自动生成只能是初稿。

第三个错误是把规则写在脑子里，而不是写进代码和脚本。更好的做法是让脚本阻止低质量文章发布，让 sitemap 自动排除 draft。

## 风险提醒

不要为了提高效率而跳过人工审核，不要用自动化制造流量、点击或客户消息，不要把平台外结算、虚假案例、未经授权数据处理包装成“自动化能力”。这些都不是可持续路径。

自动化越强，默认安全状态越重要。对新手站点来说，宁可慢一点，也不要让系统自动做不可逆的公开动作。

## 总结

AI 自动化不是越多越好，而是越清楚越好。允许自动化准备材料，禁止自动化替人承诺；允许自动化检查质量，禁止自动化未审发布。把这条线守住，网站和项目流程才有机会长期稳定。
