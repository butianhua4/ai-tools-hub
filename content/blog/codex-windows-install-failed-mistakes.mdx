# Windows 安装 Codex 常见错误和修复顺序

Windows 上安装 Codex 失败时，很多新手会快速进入“不断试命令”的状态。今天换 PowerShell，明天换 CMD，后天重装 Node，再改 PATH。最后原始错误找不到了，电脑环境也更乱。更稳的做法是先保留现场，再按顺序排查。

这篇文章整理常见错误和修复顺序。完整流程见 [Windows 上 Codex 安装失败怎么排查](/blog/codex-windows-install-failed)，短表版见 [Windows 安装失败检查清单](/blog/codex-windows-install-failed-checklist)。

## 适合谁

适合已经遇到命令不存在、权限不足、路径找不到、Node/npm 版本异常、Git 不可用、下载失败的新手。

也适合准备做环境诊断小单的人。你需要先学会把混乱状态整理成可解释报告。

## 不适合谁

不适合在公司设备或客户设备上随意改系统设置的人。受管设备可能有安全策略，不能按个人电脑方式处理。

也不适合把客户凭据、token、私有路径贴到公开工具里分析的人。日志要先脱敏。

## 风险提醒

常见风险是修改过多变量。你同时重装 Node、改 PATH、切换终端、清缓存、开管理员权限，就很难知道哪一步有效。第二个风险是权限误判。看到权限错误，不等于应该立刻强行提升权限。

做项目时还要注意客户边界。需要客户账号、公司代理、VPN、私有仓库或付费服务时，应先确认。

## 具体步骤

### 第一步：回到原始错误

找到第一次失败的命令和日志。记录当前目录、终端类型、Node/npm/Git 版本。没有这些信息，后面都是猜。

### 第二步：确认终端和路径

PowerShell、CMD、Git Bash、VS Code 终端可能读取不同环境。先固定一个终端，确认当前路径和项目文件。不要几个终端来回切换。

### 第三步：检查基础工具

运行 `node -v`、`npm -v`、`git --version`。如果命令不存在，先补基础工具；如果版本不一致，先记录路径差异；如果版本太旧，再根据项目要求决定升级。

### 第四步：判断权限类型

access denied 可能是目录权限、管理员权限、安全软件、公司策略或文件占用。先判断是哪一类，不要直接管理员运行所有命令。

### 第五步：小步修复

一次只改一件事。比如只重启终端、只切换目录、只确认 Node 路径、只重新运行版本命令。每一步都写结果。

### 第六步：形成结论

结论可以是环境缺工具、路径错误、权限待确认、网络问题、项目依赖问题。不要只写“安装失败”。

## 常见错误清单

第一类是跑错目录。第二类是混用 PowerShell、CMD 和 VS Code 终端。第三类是重复安装 Node。第四类是乱改 PATH。第五类是忽略 Git 状态。第六类是把权限问题当成工具问题。

第七类是没有保存日志。第八类是客户项目里直接试高风险命令。第九类是把账号和密钥写进聊天记录。

## 修复顺序示例

如果 `node` 命令不存在，先确认是否安装 Node，再重启终端，再检查 PATH。不要先安装 Codex。基础工具不可用时，后续命令都不稳定。

如果 PowerShell 提示脚本执行策略问题，先确认命令是否真的需要执行脚本、是否是个人电脑、是否可以用项目推荐方式。公司设备要找管理员确认。

## 交付记录怎么写

交付记录包含：原始错误、当前终端、目录、版本输出、已尝试动作、判断结论、客户待确认事项。可以从 [模板库](/templates) 拿环境检查模板，用 [报价计算器](/tools/pricing-calculator)估算诊断时间。

客户沟通可以用 [Proposal 生成器](/tools/proposal-generator)起草，但发送前要删掉过满承诺。

## 复盘时看什么

修复顺序是否合理，比某一次命令是否成功更重要。复盘时先看三个问题：有没有保留第一条原始错误，有没有一次只改一个变量，有没有把客户或设备边界写清楚。如果这三个问题都没有答案，后续排查很容易变成反复猜测。

还要区分“安装工具失败”和“项目运行失败”。前者关注 Node、npm、Git、终端、路径和权限；后者可能涉及项目依赖、脚本、环境变量、README、构建命令和代码本身。新手常见误区是把项目运行问题继续当成安装问题处理，结果一直重装基础工具，却没有读项目日志。

如果复盘发现已经尝试了太多动作，建议先停止新增命令，把现有记录整理成时间线。时间线里写清楚每一步之前的状态、执行的命令、结果变化和下一步判断。这样做虽然慢一点，但更容易找到真正的转折点。
