# Vercel 部署检查清单

Vercel 部署前后都需要检查。部署前看本地 build、依赖、环境变量和路由；部署后看公开页面、移动端、sitemap、robots、表单和客户确认事项。Codex 可以帮你整理清单和解释日志，但每一项都要人工验证。

完整流程可读 [用 Codex 检查 Vercel 部署问题](/blog/codex-vercel-deployment)。如果部署失败后已经改乱，可以看 [常见错误和修复顺序](/blog/codex-vercel-deployment-mistakes)。

## 适合谁

适合准备部署小网站、作品集、Next.js 项目、落地页的新手。你可能已经能本地跑起来，但还不确定云端部署要检查什么。

也适合做 AI 工具实践交付的人。部署不是“点一下发布”，而是要留下日志、截图、链接和客户待确认项。

## 不适合谁

不适合复杂生产系统、支付、数据库迁移、权限系统和真实用户数据项目。这些需要更完整的运维和安全流程。

也不适合没有客户授权时擅自登录、绑定域名、改 DNS 或配置真实密钥。

## 风险提醒

Vercel 部署常见风险包括：本地和云端 Node 版本不同、环境变量缺失、构建命令错误、草稿页面被收录、域名没确认、客户表单收不到、测试数据公开。

需要账号、域名、DNS、环境变量、团队权限或付费服务时，先记录为客户确认事项。

## 具体步骤

### 1. 本地检查

- lint 是否通过。
- build 是否通过。
- package.json 脚本是否清楚。
- lockfile 是否和包管理器一致。
- Node 版本是否记录。

本地失败时，先不要部署。

### 2. Vercel 配置

- 构建命令是否正确。
- 输出目录是否符合框架要求。
- 项目根目录是否正确。
- Node 版本是否需要指定。
- 是否有 framework preset。

这些配置不清楚时，先保存当前设置，不要来回试。

### 3. 环境变量

- 是否有变量清单。
- 是否区分本地、预览、生产。
- 是否有 `.env.example`。
- 是否避免把真实密钥写进代码。
- 是否需要客户自己配置。

客户项目中，密钥和账号应由客户确认处理。

### 4. 构建日志

- 失败步骤是哪一步。
- 是否是依赖安装失败。
- 是否是 TypeScript 或 lint。
- 是否是静态生成失败。
- 是否是环境变量缺失。

遇到不懂的日志，先用 [报错解释器](/tools/error-explainer)分析。

### 5. 上线后页面

- 首页是否 200。
- 关键页面是否 200。
- 404 是否正常。
- robots 是否符合策略。
- sitemap 是否不含草稿。
- canonical 是否正确。

如果是草稿站或未审核内容，要保持 noindex 策略。

### 6. 客户验收

- 正式域名是否客户确认。
- 表单收件是否客户确认。
- 联系方式是否最终版。
- 图片和 logo 是否授权。
- 未覆盖范围是否写清。

可以用 [模板库](/templates) 保存部署交付记录，用 [报价计算器](/tools/pricing-calculator)估算复查时间。

## 结论怎么写

结论可以分成：可上线、需客户确认、需修复后再部署、暂不部署。可上线表示本地和线上关键路径都通过；需客户确认表示账号、域名、变量或内容未最终确认；需修复表示存在明确构建或页面问题；暂不部署表示风险超出当前范围。

这个结论比“部署好了”更适合项目交付。

## 部署记录模板

```text
项目名称：
分支和 commit：
本地检查命令：
Vercel 部署链接：
构建日志结论：
环境变量状态：
已验证页面：
客户待确认：
未覆盖范围：
下一步建议：
```

这份记录可以直接给客户，也可以留作作品集复盘。不要只保存一个 Vercel 链接，因为链接不能说明你检查了什么。

## 常见客户确认项

部署任务里，客户常需要确认正式域名、DNS 修改、团队权限、环境变量、表单收件邮箱、统计工具、图片授权和隐私政策。只要这些未确认，就不要把结论写成完整上线。

如果客户材料不足，可以先交诊断报告。诊断报告本身也是交付物，不必把所有部署问题都塞进同一轮修复。

## 预览和生产怎么区分

Vercel 通常会有预览部署和生产部署。预览适合检查新分支、客户草稿和修复结果；生产才是正式访问入口。新手交付时要写清自己检查的是哪一个链接，不要把预览链接误当正式域名。

环境变量也要区分预览和生产。有些项目在预览里能跑，是因为变量配置完整；生产失败，则可能是生产变量缺失。反过来也一样。检查时把环境写进记录里，后续排查会快很多。

## 上线后最小复查

最小复查至少包含：首页、一个内容页、一个工具或表单页、404、sitemap、robots、移动端视图。若项目是博客或内容站，还要确认草稿没有出现在 sitemap。若项目是客户服务页，还要确认联系方式和 CTA 链接。

这些复查不复杂，但能挡住很多低级交付问题。部署显示成功，只能证明构建通过，不能证明客户要的路径都可用。
