# Claude Code 检查 Next.js 页面哪些适合写进接单范围

Next.js 页面检查可以做成低风险小单，但要先写清范围。适合写进范围的是一个明确页面、一个明确问题、低权限本地验证和简短交付说明。不适合默认包含的是全站重构、生产部署、数据库、付款流程、账号权限和长期维护。

这篇适合搭配 [Claude Code Next.js 页面检查清单](/blog/claude-code-nextjs-page-check-checklist)、[Claude Code 常用命令检查清单](/blog/claude-code-common-commands-checklist) 和 [/templates](/templates) 使用。

## 适合谁

适合想接页面小改、页面诊断、文案调整、样式微调、组件报错排查的新手。你可以用 Claude Code 辅助定位文件和整理方案，但范围必须由人确认。

也适合准备把服务拆成“页面诊断”和“页面小修”两段的人。信息不足时先诊断，信息完整时再修。

## 不适合谁

不适合把全站 UI 重做、SEO 全站优化、性能优化、登录系统、支付流程或后台权限写进同一个小单。

也不适合客户不提供页面路径、截图、复现步骤和验收方式，却要求固定结果的任务。范围不清时，先诊断。

## 具体步骤

第一，基础范围可以包含页面信息整理。包括页面 URL、路由、截图、复现步骤、预期效果和实际结果。

第二，诊断范围可以包含相关文件定位。让 Claude Code 根据路由、文案、组件名和错误信息列出可能文件，再人工确认。

第三，修复范围可以包含一个明确问题。比如一个按钮文案、一个样式错位、一个组件报错、一个路由链接错误。

第四，验证范围可以包含本地 lint、build、页面预览和手动检查。没有账号或环境变量时，要写清未验证。

第五，交付范围可以包含修改摘要、验证结果、未覆盖范围和客户验收步骤。

## 报价范围示例

```text
本次包含：
- 检查一个明确 Next.js 页面
- 定位相关组件或配置
- 处理一个明确页面问题
- 运行本地基础验证
- 提供交付说明

不包含：
- 全站重构
- 生产部署
- 真实数据处理
- 支付或账号权限
- 长期维护
```

这段能直接放进报价，避免客户把页面小改理解成全站支持。

## 什么时候只做诊断

当客户没有提供页面路径、无法复现、需要登录数据、依赖生产接口或不清楚预期效果时，先做诊断。诊断交付包括问题记录、可能相关文件、需要补充材料和是否适合继续修复。

如果诊断发现问题在接口、权限、缓存、部署或环境变量上，就不要继续按页面小单处理。重新确认新范围和授权。

## 客户确认话术

```text
我可以先检查一个明确页面的问题。本次只覆盖页面层面的诊断和低风险修改，不包含生产部署、账号权限、真实数据或全站重构。请先提供页面路径、复现步骤和验收方式。
```

这句话适合报价前发给客户，把可做和不可做说清楚。

## 怎么拆成服务档位

基础档可以只做页面诊断：整理页面路径、定位相关文件、输出可能原因和需要补充的信息。适合客户信息不完整、还不能确认是否能修的情况。

标准档可以处理一个明确页面问题：例如文案、链接、局部样式或组件小报错。交付时附上本地验证结果和客户验收步骤。

进阶档才考虑多页面一致性检查、响应式复核或较完整的页面说明。只要超过一个页面，就不应该沿用基础小单价格。

## 追加需求怎么处理

客户经常在页面修完后追加“顺便把另一个页面也改一下”。这时要重新确认范围。你可以说：“这个属于新页面范围，我可以继续处理，但需要重新确认页面路径、预期效果和验收方式。”

追加需求不一定要拒绝，但不能默默吸收进原单。边界越清楚，客户越容易理解你是在提供专业服务，不是在无限试改。

## 风险提醒

不要把页面检查包装成全站质量保证。你只能交付本次范围内的检查和验证。

不要在缺少测试环境时承诺交互流程全部正常。没有验证条件，就写成待客户验收。
