# 小网站交付检查清单

这份清单用于交付低风险小网站。完整流程见 [不会写代码怎么交付一个小网站](/blog/deliver-small-website-with-ai-no-code)。每一步都要留下记录，尤其是客户确认、素材授权和部署权限。

## 适合谁

适合正在用 AI、模板或无代码工具练习小网站交付的新手。你可以不会从零写完整项目，但要能按清单验证页面和说明限制。

也适合做作品集的人。一个小网站案例如果记录完整，比十个没有验证的截图更有说服力。

## 不适合谁

不适合交付复杂后台、支付系统、登录系统、客户数据面板、生产数据库或安全修复。那些任务需要更强的工程能力和权限管理。

也不适合客户没有给素材、没有确认范围、没有验收标准时直接开做。范围不清，小项目也会变成长期消耗。

## 风险提醒

检查清单的第一条永远是边界。客户账号、域名解析、付款套餐、API key、生产主机密码都属于敏感事项，不应该随便发给接单者。

如果需要注册平台、绑定域名或开通付费服务，把它写成客户侧待办。你可以提供步骤，但不要代客户做不透明的账号操作。

### 4. 页面检查

- 首页信息是否准确：
- 按钮是否能点击：

### 5. 构建和预览

如果是前端项目，至少运行：

```bash

```

如果没有构建命令，也要记录预览方式。比如本地 HTML、部署预览链接或截图。

- 谁拥有部署平台账号：
- 谁拥有域名：

## 最低交付记录

最低记录包含：完成范围、预览链接或截图、验证命令、客户需确认项、未包含功能、后续维护建议。客户看完后应该知道这个网站能做什么、不能做什么、下一步需要谁操作。

如果某项还没完成，不要写成已交付。比如域名未绑定，就写“当前为预览链接，正式域名需客户完成 DNS 确认”。这比含糊收尾更稳。

## 上线后 24 小时检查

小网站上线后，不要立刻把任务完全关掉。至少在 24 小时内检查一次：正式域名是否能访问、移动端是否正常、图片是否加载、联系按钮是否正确、客户是否发现文案错别字、搜索引擎是否不该收录测试页。

如果上线依赖客户域名或部署账号，把这一步写成共同确认。比如“客户完成 DNS 后，我再检查访问结果”；如果客户还没完成授权，就记录为明日待办。这样你不会因为外部账号未准备好而被动背锅。

上线后检查也要控制范围。你可以检查本次交付页面是否正常，不等于承诺长期维护所有内容。需要持续维护、内容更新或新增页面，应单独确认范围和报价。
