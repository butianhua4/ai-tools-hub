# Codex 生成代码后的人工审核检查清单

很多新手第一次用 Codex 改代码时，会把注意力放在“它有没有给出答案”。真正接近交付时，更重要的问题是：这个答案有没有改对地方、有没有带来副作用、有没有经过运行验证、有没有记录清楚。下面这份清单可以放在每次 AI 生成代码之后使用，也可以作为项目前的小型审核表。

如果你还不知道完整流程，先读 [Codex 生成代码后怎么人工审核](/blog/codex-review-generated-code)。如果你已经发现页面坏了、测试失败或需求越变越大，可以跳到 [常见错误和修复顺序](/blog/codex-review-generated-code-mistakes)。

## 适合谁

适合已经会让 Codex 修改文件、但还没有建立人工复核习惯的人。你可能能看懂一部分代码，也能运行项目，却不知道每次交付前该检查哪些点。清单的价值就是减少遗漏：不靠临场记忆，而靠固定动作。

也适合准备做小额项目的人。客户一般不会关心你用了什么工具，他关心的是结果是否能用、问题是否被解释清楚、后续是否容易维护。你用清单把证据留下来，沟通就会稳很多。

## 不适合谁

不适合完全不运行、不阅读、不保存记录的人。清单不能替代真实验证，也不能把超出能力范围的任务变成简单任务。若项目涉及支付、生产数据库、账号权限、安全漏洞、隐私数据或线上高流量服务，新手不应独立承诺完成。

也不适合用来包装没有把握的能力。做项目时可以说明“我会先做初步检查并反馈风险”，但不要把 AI 生成内容包装成已经充分验证的结果。

## 风险提醒

检查清单容易给人一种“勾完就可以交”的错觉。实际情况更细：有些问题必须看业务上下文，有些结果必须让客户确认，有些边界需要平台规则和合同约定支持。清单只能帮助你发现常见风险，不能替代最终责任判断。

另外，Codex 可能会生成看起来合理的依赖名、配置名或 API 用法。只要项目没有真实安装、真实运行、真实验收，就不要把它写进交付承诺。

## 具体步骤

### 1. 需求检查

- 原始需求是否保存完整，包括截图、报错、链接、仓库、分支和客户原话。
- 目标结果是否能用一句话复述。
- 交付标准是否明确，例如“按钮可点击”“接口返回正确数据”“移动端不遮挡内容”。
- 是否存在必须先问客户的问题，例如权限、环境、截止时间、设计稿、测试账号。

如果需求还散着，可以先用 [Proposal 生成器](/tools/proposal-generator)整理确认问题，不要急着报价。

### 2. 差异检查

- Codex 改了哪些文件。
- 是否新增或删除了文件。
- 是否改了 package、lockfile、配置、路由、中间件或共享组件。
- 是否有与任务无关的大范围重写。
- 是否留下调试文案、临时日志、占位数据或测试账号。

这一项要配合 git diff 看。只看最终页面不够，因为很多副作用藏在共享模块里。

### 3. 依赖和环境检查

- 新依赖是否真的安装。
- 版本是否和项目框架兼容。
- 是否要求新的环境变量。
- 是否修改了构建脚本、部署配置或运行命令。
- 是否在本地、预览环境和客户环境之间存在差异。

如果新增环境变量或权限，应记录为待客户确认事项。不要为了继续推进而猜测客户配置。

### 4. 功能检查

- 正常路径是否能走通。
- 空数据、错误输入、网络失败、刷新页面后是否还能使用。
- 桌面端和移动端是否都看过。
- 表单是否有基础校验。
- 用户看不到的接口或脚本是否有日志或返回值。

遇到报错时，可以用 [报错解释器](/tools/error-explainer)先拆现象，再决定是否继续修。

### 5. 安全和隐私检查

- 是否出现密钥、token、账号、手机号、邮箱、客户内部链接等敏感信息。
- 是否新增了管理员入口或权限判断。
- 是否把客户数据传到未确认的第三方。
- 是否改变了登录、支付、订单、数据库写入等关键流程。
- 是否需要客户书面确认才能继续。

这部分不要省。对新手来说，识别风险比展示速度更重要。

### 6. 交付记录检查

- 改动摘要是否能让客户看懂。
- 运行过哪些命令。
- 通过了哪些测试。
- 哪些场景没有覆盖。
- 后续建议是什么。
- 是否保留截图、日志或录屏。

如果准备把这次练习放进作品集，可以从 [模板库](/templates) 里拿交付记录模板，把过程整理成案例。

## 审核结果怎么判断

可以把结果分成三类。第一类是“可继续交付”：改动范围清楚，本地验证通过，没有发现明显高风险点。第二类是“需要客户确认”：功能大体可行，但缺少权限、素材、验收标准或环境信息。第三类是“暂停处理”：涉及敏感权限、生产数据、重大架构变化，或你无法解释 Codex 为什么这样改。

这三个结论比“我觉得差不多”更有用。它们能帮助你在做项目时减少含糊承诺，也能让客户知道下一步该提供什么。
