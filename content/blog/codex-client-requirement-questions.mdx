# Codex 处理客户需求不清楚时怎么提问

新手用 Codex 接单时，最容易犯的错不是不会写提示词，而是客户需求还没问清楚就让工具开始改。客户说“帮我修一下页面”“做个自动化”“部署有问题”，听起来都像可以马上处理，但里面可能缺少目标、材料、权限、验收标准和风险边界。

这篇是草稿，只做流程学习和人工审核参考。你可以配合 [Proposal 生成器](/tools/proposal-generator)、[报错解释器](/tools/error-explainer) 和 [模板库](/templates) 使用：先把客户需求拆成可确认的问题，再决定是否报价和动手。

## 适合谁

适合准备接第一个小单的新手，尤其是面对客户一句话需求时，不知道该追问什么、怎么追问、哪些问题会显得专业。你可以还不熟悉复杂工程，但至少要能把“我以为客户要什么”和“客户确认要什么”分开。

也适合已经在用 Codex 辅助写 Proposal 的人。工具能帮你整理语气和结构，但不能替你判断客户有没有给足材料，也不能替你承担需求误解造成的返工。

## 不适合谁

不适合在需求模糊时直接承诺结果、直接报价、直接访问后台或直接修改生产项目的人。只要客户没有说明验收标准、测试环境、权限范围和交付边界，就不应该把任务当成已经可执行。

也不适合把 AI 输出原样发给客户的人。客户沟通需要结合真实项目、平台规则和你的能力边界，不能只复制一个看起来礼貌的模板。

## 风险提醒

需求不清楚时，风险会被藏起来。最常见的是：客户没有说这是测试环境还是生产环境，没有说明能否提供仓库，没有说明是否涉及账号权限、支付、隐私数据或真实用户。遇到这些信息缺失，先暂停报价，把问题列出来让客户确认。

如果追问中出现登录、授权、开通服务、修改生产配置、处理真实数据等事项，也不要代替客户执行。先记录为客户待确认项；需要注册或授权的步骤，留到客户明确确认后再处理。

## 具体步骤

1. 先复述客户目标，用一句话写出你理解的任务。
2. 询问当前材料：链接、截图、仓库、报错、账号角色、测试环境。
3. 询问完成标准：客户如何判断这件事已经完成。
4. 询问限制条件：预算、截止时间、平台规则、允许修改的范围。
5. 询问风险点：是否涉及生产环境、真实数据、支付、权限或安全问题。
6. 把客户回答整理成“可做范围”和“待确认事项”。
7. 再让 Codex 辅助拆解步骤、报价和交付说明。
8. 如果仍然无法复述任务，就不要继续报价。

## 可以直接问的 8 个问题

```text
1. Could you share the current link, screenshot, repository, or exact error message?
2. What should be considered finished for this task?
3. Is this a test environment or a live production project?
4. Which files, pages, or features are in scope?
5. Are there any platform, deadline, or budget constraints?
6. Does this involve login, payment, user data, API keys, or admin permissions?
7. Do you already have a preferred solution, or should I suggest one after reviewing?
8. If something goes wrong, is there a rollback or backup plan?
```

中文回复也可以更简短：

```text
我可以先评估，但需要确认几个点：
- 当前问题的链接、截图或报错是什么？
- 你认为完成的标准是什么？
- 这是测试环境还是生产环境？
- 本次允许修改哪些范围？
- 是否涉及账号权限、真实数据、支付或密钥？
确认后我再判断是否适合接，以及如何报价。
```

## 提问后的记录模板

```text
客户目标：
已提供材料：
缺失材料：
验收标准：
允许修改范围：
风险点：
客户待确认：
下一步：
```

这份记录会让你后续用 Codex 更稳。工具看到的是经过整理的上下文，而不是一堆零散聊天；客户看到的是你在认真确认边界，而不是急着成交。

## 为什么先提问反而更快

很多新手担心问题问多了会错过机会，所以倾向于先报价、先承诺、先让 Codex 给方案。实际接单里，真正拖慢进度的往往是后面返工：客户说你理解错了、环境不是你以为的那个、权限不能给、验收标准和你做的结果不一致。

先提问能把这些风险提前暴露出来。你不是在拖延，而是在把任务变成可执行的范围。客户如果愿意回答关键问题，后续合作通常更顺；客户如果拒绝说明目标和权限，本身就是风险信号。
