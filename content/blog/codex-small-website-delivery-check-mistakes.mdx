# 小网站交付检查常见错误和修复顺序

小网站交付最容易出错的阶段，往往不是写代码，而是最后 10%。页面已经能看，客户也快要验收，于是新手容易跳过系统检查。结果上线后才发现手机导航打不开、表单收不到、价格写错、草稿进了 sitemap、域名还没配置好。

这篇文章整理常见错误和修复顺序。完整流程见 [用 Codex 做小网站交付前检查](/blog/codex-small-website-delivery-check)，短表版见 [小网站交付前检查清单](/blog/codex-small-website-delivery-check-checklist)。

## 适合谁

适合已经完成小网站草稿、准备交付或已经收到客户反馈的新手。你可能遇到移动端错位、按钮无效、表单没接通、部署后页面不一致、客户说内容不对。

也适合准备做 AI 辅助网站项目的人。交付能力不只是做页面，还包括发现风险、排序修复和写清验收记录。

## 不适合谁

不适合大型系统、复杂后台、支付和真实用户数据项目。那些场景需要更完整的测试、权限控制和安全审查。

也不适合用来掩盖未完成工作。发现问题后要记录、修复或明确待确认，不能把未知风险当成已完成。

## 风险提醒

常见风险有两类。第一类是公开风险：错误信息、客户私有内容、测试图片、草稿页面、未授权素材被公开。第二类是交付风险：客户以为网站已经可用，但表单、域名、邮件、统计或移动端还没验证。

如果需要客户账号、域名 DNS、邮箱服务、表单后端或统计平台授权，应先列为待确认。不要在未授权情况下替客户注册或修改配置。

## 具体步骤

### 第一步：先停止扩功能

交付前发现问题时，不要继续加新功能。先把当前问题列出来：页面、截图、影响、是否阻止交付。功能扩张会让验收范围更乱。

### 第二步：按严重程度排序

先修阻止访问的问题，例如部署失败、首页打不开、导航失效、表单完全不可用。再修客户信息错误，例如电话、邮箱、价格、服务范围。最后修视觉细节，例如间距、动画和局部文案。

### 第三步：复现并截图

每个问题都要能复现。桌面端和移动端分开截图。客户反馈“手机上有问题”时，要问设备、浏览器、页面路径和截图，不要凭感觉改。

### 第四步：小步修复

让 Codex 帮忙时，限制范围。例如“只检查移动端导航，不要改文案”“只解释 build 报错，不要升级依赖”。这样能减少新副作用。

### 第五步：重新跑交付检查

修完后重新跑 lint、build、关键页面预览和链接检查。不要只看刚修的地方。小网站很多问题是共享组件导致的。

## 常见错误清单

第一类是只看桌面端。客户在手机上打开时，导航、按钮、表单和长标题都可能出问题。第二类是漏测表单。页面显示表单，不代表客户能收到消息。第三类是公开信息错误。客户名称、电话、邮箱、价格和地址必须复核。

第四类是忘记草稿策略。草稿页、测试页和未审核内容不应进入 sitemap 或公开导航。第五类是图片和 logo 授权不清。第六类是部署后不复查，本地没问题不代表线上没问题。

## 修复顺序示例

假设客户反馈手机上按钮被遮挡。先复现移动端问题，截图保存，再检查布局容器和按钮尺寸。修完后同时看桌面和手机，不要只验证一个宽度。

假设表单提交失败。先确认表单是静态链接、第三方服务还是自建接口。若需要客户邮箱服务或 API key，先列为客户确认，不要临时用自己的服务代替。

## 交付后怎么记录

交付记录建议包含已修复问题、未覆盖范围、客户待确认和下一步建议。可以用 [模板库](/templates) 存档，用 [Proposal 生成器](/tools/proposal-generator)整理客户说明，再人工改写成简短版本。

如果修复导致报价范围变化，用 [报价计算器](/tools/pricing-calculator)估算新增时间，并说明原因。不要把新增需求默默塞进原范围。

## CTA：下一步

把你最近一个小网站的问题按严重程度排序：阻止访问、信息错误、功能缺陷、视觉细节。先处理前两类，再回到 [检查清单](/blog/codex-small-website-delivery-check-checklist) 做完整复查。
