# GitHub 和 Vercel 新手建站检查清单

这份清单适合在第一个练习站上线前使用。它的目标不是把你变成全栈开发者，而是帮你确认：代码能保存、构建能通过、部署能复现、敏感信息没有泄漏、页面内容没有夸大承诺。

需要登录 GitHub、Vercel、域名服务商或客户账户的步骤，先记录为明日待办。草稿阶段只写清楚要检查什么，不模拟真实授权，也不替用户做账户操作。

## 适合谁

适合准备发布个人作品集、AI 接单练习站、模板展示页、服务介绍页或小型博客的新手。你可能已经有一份页面代码，但还不确定能不能稳定部署到线上，也不知道上线前该查哪些风险。

也适合正在用 AI 生成页面的人。AI 能帮你写组件和文案，但不能替你确认仓库权限、密钥安全、版权来源、域名设置和线上可访问性。清单的价值就在于把这些检查项拎出来。

## 不适合谁

不适合用于客户生产系统、支付页面、会员中心、数据库后台、隐私数据处理或复杂权限项目。这类项目需要更严格的安全、日志、备份和合规审查。

也不适合把它当作平台官方说明。GitHub 和 Vercel 的界面、免费额度、团队权限和构建默认值可能变化，发布前必须以实际后台为准。

## 检查一：本地项目是否可复现

- 是否知道项目使用的是静态 HTML、Next.js、Astro、Vite 还是其他框架？
- 是否能在本地运行安装命令？
- 是否能跑通开发预览？
- 是否能跑通生产构建？
- 是否写了 README，说明如何启动项目？

如果本地构建都没有跑过，不建议直接导入 Vercel。线上失败时你很难判断是代码问题、依赖问题，还是平台配置问题。

## 检查二：Git 仓库是否干净

- `git status` 是否只有你预期的修改？
- `.env`、API key、私有令牌、客户资料是否被排除？
- README 是否说明项目用途，而不是夸大商业能力？
- 仓库可见性是公开还是私有，是否符合素材和客户信息要求？
- 是否有不该公开的截图、日志、域名、邮箱或后台路径？

新手常犯的错误不是不会提交代码，而是不知道哪些内容不该提交。看不懂的文件先不要公开，留到明日人工复核。

## 检查三：Vercel 导入是否清楚

- 导入的是正确 GitHub 仓库吗？
- Framework Preset 是否和项目类型一致？
- Build Command、Output Directory、Install Command 是否需要手动调整？
- 是否需要环境变量？如果需要，是否只在 Vercel 后台填写，而不是写进仓库？
- 部署失败时，是否保存了完整错误日志？

不要只看“部署中”三个字。真正有用的是部署结果、构建日志、访问链接和失败原因。每一次失败都可以变成作品集里的学习记录，但前提是你有保存证据。

## 检查四：线上页面是否能给别人看

- 首页是否能打开？
- 手机宽度下是否能阅读和点击？
- 导航、内链、联系入口是否正常？
- 404 页面是否不会暴露调试信息？
- 标题、描述、CTA 是否没有夸大能力或承诺结果？
- 图片、字体、图标和案例是否有授权或来自自己的练习？

一个练习站可以很简单，但不能混乱。客户或朋友第一次打开链接时，应该能看懂你做了什么、能提供什么、还在练习什么。

## 具体步骤

1. 在本地运行安装、开发预览和生产构建。
2. 检查 `git status`，确认没有意外文件。
3. 检查 `.gitignore`，确认敏感文件不会进入仓库。
4. 提交一版可说明的修改，写清楚提交信息。
5. 明日登录 GitHub，确认仓库可见性和默认分支。
6. 明日登录 Vercel，导入仓库并核对构建设置。
7. 部署成功后，用手机和桌面各打开一次，并保存截图或检查记录。

## 可复制的本地命令

不同项目命令可能不一样，下面只是常见 Node 项目的检查顺序：

```bash
npm install
npm run build
git status
```

如果项目不是 Node 技术栈，不要硬套这些命令。先看 README、`package.json` 或项目说明，再决定怎么验证。

## 明日待办

- 登录 GitHub，确认仓库公开/私有状态、默认分支、README 和敏感文件策略。
- 登录 Vercel，确认项目导入、团队归属、构建命令、环境变量和部署日志。
- 检查域名服务商设置，确认是否需要添加 DNS 记录。
- 核对 Vercel 当前费用、用量提醒和团队权限。
- 用真实设备打开部署链接，记录移动端和桌面端截图。
- 人工复核页面文案、版权素材、联系方式和免责声明。

## 风险提醒

不要在公开仓库放密钥、客户资料、后台截图、未授权素材或私人通信记录。不要把一个练习站包装成已经验证过的大型商业案例。新手可以展示学习过程，但不能用模糊描述让读者误以为你已经完成过未曾做过的项目。

如果客户要求你接触他的 GitHub 组织、Vercel 团队、生产环境变量、数据库或支付配置，需要先暂停并确认权限范围。能打开一个练习站，不等于可以独立处理客户生产权限。
