# Codex 客户需求澄清常见错误和修复顺序

用 Codex 写客户回复时，语气通常不难变得专业，难的是问题有没有问到点上。很多新手的 Proposal 看起来很顺，但客户真正需要什么、怎么验收、能不能提供材料、是否涉及权限和风险，全都没有确认。结果不是返工，就是接到超出能力范围的任务。

这篇是草稿，只做复盘参考，不替代具体项目判断。还没有固定问题库的，可以先看 [需求澄清主流程](/blog/codex-client-requirement-questions) 和 [检查清单](/blog/codex-client-requirement-questions-checklist)。

## 适合谁

适合已经开始用 Codex 辅助写 Proposal 的新手。你可能能写出礼貌回复，但经常漏问验收、权限、环境和边界，导致后续沟通越来越被动。

也适合复盘失败接单的人。与其责怪客户“没讲清楚”，不如把自己没问的问题整理出来，下次开工前先补齐。

## 不适合谁

不适合客户已经明确要求无授权访问、处理真实数据、修改支付或生产权限的任务。遇到这种需求，补问问题只是第一步，不能代替风险拒绝或暂停。

也不适合想用 AI 自动判断客户意图的人。Codex 可以帮你整理问题，但客户真实约束必须由客户确认。

## 风险提醒

需求澄清缺失会把小单变成大坑。尤其是没有验收标准、没有测试环境、没有权限范围、没有截止时间、没有回滚方式时，不要急着报价。任何涉及登录、授权、真实数据、生产环境的事项，都应先记录为客户待确认。

## 错误一：只问“你要做什么”

客户回答“做一个页面”“修一个报错”并不等于需求清楚。你还要问结果长什么样、在哪里验证、哪些内容不能改、是否已有参考。

修复方式：把目标问题换成可验收问题，例如“你希望用户完成哪个动作？”“完成后你会检查哪个页面或命令？”“是否有截图或参考链接？”

## 错误二：没问材料

没有链接、截图、仓库、报错、复现步骤，就直接报价，很容易低估工作量。Codex 也会因为上下文不足给出泛泛建议。

修复方式：先要最小材料，不要一次要太多。可以说“我可以先评估，但需要当前链接、报错截图和完成标准；涉及密钥或真实数据的内容请先脱敏。”

## 错误三：没问权限和环境

同样是部署问题，测试环境和生产环境完全不同。同样是 API 报错，测试 key 和真实支付配置风险完全不同。没有确认环境就动手，是安全审核里的大红灯。

修复方式：固定询问“这是测试环境还是生产环境？”“是否涉及登录、支付、用户数据、账号权限或密钥？”需要客户操作的授权步骤，先写成待确认项。

## 错误四：直接复制 Codex 的回复

Codex 可能会生成漂亮但空泛的回复，例如“我有丰富经验，可以快速完成”。这种话没有边界，也没有基于客户材料。

修复方式：让工具只做草稿，然后手动加入具体问题、未覆盖范围和风险提醒。客户沟通要少一点空话，多一点可确认事项。

## 具体步骤：发现已经问少了怎么补救

1. 暂停报价或执行，不要继续扩大承诺。
2. 回到客户原始消息，列出缺失信息。
3. 按目标、材料、验收、权限、风险五类补问。
4. 告诉客户：确认这些信息后再给范围和报价。
5. 如果已经报价，说明报价基于哪些前提，哪些变化会影响范围。
6. 把客户回答整理成记录，作为后续 Codex 分析的输入。
7. 对注册、授权、登录、高权限操作，单独列为客户待确认。
8. 仍然无法复述任务时，不要接。

## 补问模板

```text
Thanks for the details. Before I confirm the scope, I need to clarify a few points:
1. What exact result should be considered done?
2. Can you share a screenshot, link, repository, or error message?
3. Is this a test environment or production?
4. Does it involve login, payment, user data, API keys, or admin permissions?
5. Are there any files or features that should not be changed?

After these are confirmed, I can suggest the safest next step.
```

## 如何判断补救是否有效

补问之后，不是客户回了一句“yes”就算清楚。你要能重新写出一段范围说明：本次做什么、不做什么、用什么材料、在哪个环境验证、谁负责授权、完成标准是什么。如果这段话仍然写不出来，说明需求还没有澄清完。

也要看客户是否愿意配合确认边界。愿意补材料、愿意说明验收、愿意把高权限动作留给自己处理的客户，更适合新手合作。反过来，如果客户不断要求你先做、先登录、先改生产配置，却不愿意说明授权和验收，就应该暂停。
