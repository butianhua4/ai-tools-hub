# Codex README 审核检查清单

README 审核不应该只问“写得好不好”。更实用的问题是：它有没有说清项目用途，命令能不能跑，环境变量有没有遗漏，部署说明是否真实，读者遇到问题是否知道下一步。Codex 可以帮你列缺口，但每一项都要人工核对证据。

这份清单适合放在每次文档修改前后使用。完整流程可读 [用 Codex 审核 README 怎么做](/blog/codex-readme-review)，项目边界可读 [README 审核项目能不能接](/blog/codex-readme-review-freelance-scope)。

## 适合谁

适合正在整理自己的项目、开源练习仓库、客户小项目说明的人。你可能不需要重构代码，但需要让文档变得可用：别人打开 README 后，能知道项目是什么、怎么跑、哪里需要配置、哪些地方还没覆盖。

也适合准备做文档类项目的新手。清单可以帮助你把交付从“我润色了一下”变成“我检查了结构、命令、环境变量和风险说明”。

## 不适合谁

不适合只做语言美化、不验证命令的人。README 最核心的作用是帮助用户正确开始，如果命令、依赖和环境说明不准，文字再顺也没有用。

也不适合处理客户未授权公开的材料。截图、内部链接、私有部署地址、真实密钥和客户业务细节，都要在发布前单独确认。

## 风险提醒

清单只能减少遗漏，不能证明文档已经适合所有读者。不同系统、Node 版本、包管理器、部署平台都可能影响运行结果。若你只在一个环境里测试过，就要在交付记录里写清楚。

还要避免把暂未验证的内容写成确定说法。例如没有跑过测试，就不要写“测试已通过”；没有部署过，就不要写“可直接部署”。可以写“待在目标环境确认”，这样更诚实。

## 具体步骤

### 1. 项目基本信息

- README 是否用一两段话说明项目用途。
- 是否说明目标用户或适用场景。
- 是否说明当前状态，例如练习项目、内部工具、演示版或待完善版本。
- 是否有截图或示例链接；如果有，是否经过授权。
- 是否避免夸大功能。

### 2. 快速开始

- 是否说明依赖条件，例如 Node、包管理器、数据库或外部服务。
- 安装命令是否真实存在。
- 开发命令是否能启动项目。
- 默认端口是否写清。
- 第一次运行需要哪些准备步骤。

如果命令失败，先用 [报错解释器](/tools/error-explainer)整理日志，再决定是改 README 还是修项目配置。

### 3. 环境变量

- 是否列出必须的环境变量。
- 示例值是否使用占位符。
- 是否说明哪些变量仅用于本地，哪些用于部署。
- 是否提醒不要提交真实密钥。
- 是否说明缺少变量时会出现什么表现。

这一块很容易被忽略，但对客户项目尤其重要。

### 4. 常用命令

- 是否列出开发、构建、测试、格式化、lint 等命令。
- 每个命令是否在项目脚本里存在。
- 是否说明命令用途，而不是只堆命令。
- 是否标注已验证和待确认的命令。

如果你没有时间跑全部命令，至少要在交付记录里说明未覆盖范围。

### 5. 部署和交付

- 是否说明推荐部署平台。
- 是否说明构建输出或部署入口。
- 是否说明部署前需要配置的环境变量。
- 是否说明域名、数据库、存储等外部资源是否由客户提供。
- 是否列出发布前检查项。

需要客户账号、平台授权或付款配置的事项，只能列为待办或客户确认，不能替客户越过确认流程。

### 6. 风险和限制

- 是否说明项目当前限制。
- 是否说明已知问题。
- 是否说明安全、隐私、许可证或数据边界。
- 是否有联系或反馈方式。
- 是否避免承诺未验证效果。

这一项会让 README 显得更专业，也能降低误解。

## 审核结果怎么写

交付结果可以分成三类：已补充、已验证、待确认。已补充包括结构和文案；已验证包括运行过的命令、截图和日志；待确认包括客户平台、正式环境变量、许可证、截图授权和生产部署细节。

如果要报价，可以用 [报价计算器](/tools/pricing-calculator)估算“阅读项目、运行验证、改写文档、整理交付记录”的时间。准备回复客户时，可以用 [Proposal 生成器](/tools/proposal-generator)生成初稿，再人工删掉不确定承诺。
