# Codex 修改现有网页 UI 使用前怎么判断是否适合

客户说“帮我改一下 UI”，可能只是调一个按钮，也可能是完整重设计。新手项目练习前要先判断：这是可验证的小修，还是模糊审美需求；是公开页面，还是涉及后台、支付、账号和真实数据的高风险页面。

这篇是草稿，只做项目范围判断参考，不构成报价承诺。建议和 [UI 修改安全流程](/blog/codex-edit-existing-ui)、[检查清单](/blog/codex-edit-existing-ui-checklist)、[报价计算器](/tools/pricing-calculator) 一起使用。

## 适合谁

适合准备接网页 UI 小单的新手。你可能能用 Codex 调整样式和组件，但还需要判断客户需求是否足够清楚、页面是否能访问、修改是否能截图验证、是否会影响全局设计。

也适合已经收到客户需求、正在写报价的人。你可以先把任务分成低风险、中风险和应暂停三类，再决定是否接。

## 不适合谁

不适合客户只说“做高级一点”“像大厂一样”“整体重做”，却不给参考图、品牌规则、验收标准和范围限制的任务。模糊重设计很容易超过新手能力和原报价。

也不适合没有授权就要进入生产后台、修改支付页、处理真实用户数据或改账号权限的任务。UI 任务一旦触碰关键业务页面，就必须先确认权限和验证方式。

## 风险提醒

UI 项目的风险常被低估。客户可能只看视觉，但你的改动可能影响转化、可访问性、移动端可用性、后台操作按钮和品牌一致性。报价前要写清验证视口、页面范围和未覆盖状态。

## 具体步骤：三类判断

低风险：客户给了明确页面、截图、目标状态和验收标准，修改集中在局部样式、间距、按钮、卡片、表单、移动端错位。这类可以评估，但要包含截图和响应式验证。

中风险：客户要求统一多个页面、调整组件库、改主题色、优化响应式或处理复杂交互。这类可以先做诊断，再分阶段报价，不要一次承诺全部完成。

应暂停：涉及登录后台、支付流程、真实用户数据、权限页面、生产环境高风险操作，或者客户无法说明验收标准。新手不要直接接完整执行，可以只接需求整理和风险清单。

## 项目前要问的问题

1. 需要修改哪个页面、组件或状态？
2. 是否有修改前截图、参考图或设计稿？
3. 客户认为完成的标准是什么？
4. 需要检查哪些视口：手机、平板、桌面、宽屏？
5. 是否涉及后台、支付、登录、账号权限或真实数据？
6. 是否允许修改全局组件或主题变量？
7. 有没有不能改变的品牌色、字体、圆角、间距和文案？
8. 如果效果不满意，是调整当前范围，还是另开重设计需求？

## 报价范围模板

```text
本次范围：
- 修改指定页面/组件的局部 UI
- 保留修改前后截图
- 检查移动端和桌面端
- 提供修改文件和未覆盖范围

不包含：
- 完整视觉重设计
- 未指定页面的全局调整
- 生产后台高权限操作
- 支付、账号权限和真实用户数据相关修改
```

如果客户要求更多页面或更大风格变化，要重新确认范围。不要让“小改 UI”悄悄变成整站重做。

## 最小可接版本

当客户需求还不够清楚时，可以先接一个诊断阶段：整理现有 UI 问题、截图标注、列出可修改范围、估算验证视口、给出分阶段建议。这比直接承诺重做更安全，也更容易形成可信交付。

诊断阶段的交付可以是：问题列表、页面截图、风险点、建议修改顺序、预计验证方式、客户待确认事项。后续是否正式执行，再根据客户确认决定。
