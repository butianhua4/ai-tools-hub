# Vercel build failed 常见原因：新手先看这份排查顺序

Vercel build failed 最容易让新手慌，因为页面上只看到失败，真正的原因藏在日志、项目配置、环境变量和框架输出里。正确做法不是先改一堆代码，而是把失败拆成几个可验证的环节：有没有成功安装依赖，构建命令有没有跑起来，缺不缺环境变量，项目根目录是不是选错，线上 Node 版本和本地是否一致，构建后有没有生成 Vercel 能识别的输出。

这篇是草稿状态，当前用于整理排查框架。正式发布前需要人工核对平台文档、补充真实截图和具体项目案例。你也可以配合 [Vercel 部署检查清单](/blog/codex-vercel-deploy-preflight-checklist) 一起使用，把每一次失败都记录成可复盘的材料。

## 适合谁

适合正在把 Codex、Claude Code、ChatGPT 或自己写的小项目部署到 Vercel 的新手。你可能能在本地跑通 `npm run dev`，但一推到 GitHub 或 Vercel 就看到 build failed，不知道是代码问题、依赖问题，还是平台配置问题。

也适合准备做部署排查类项目的人。客户经常只发一句“网站部署失败”，如果你不能让对方提供仓库、构建日志、预览链接、环境变量清单和复现步骤，很容易把一个小问题接成边界不清的生产事故。

## 不适合谁

不适合想跳过日志、直接凭感觉改代码的人。部署失败通常不是玄学，日志会给出线索；如果没有日志，就要先补材料，而不是先承诺结果。

如果项目涉及真实用户数据、支付、后台权限、数据库迁移或生产域名切换，新手不应该独立操作线上环境。你可以做初步排查和记录，但关键修改要让客户或有经验的人复核。

## 常见原因一：依赖安装失败

第一类问题出现在安装阶段。常见表现是 `npm install`、`pnpm install` 或 `yarn install` 失败，日志里会出现 package not found、版本冲突、peer dependency、lockfile 不一致等信息。新手常犯的错是本地用一种包管理器，仓库里却同时留下多个 lockfile，线上构建时平台不知道该相信哪一个。

排查时先看仓库里有没有 `package-lock.json`、`pnpm-lock.yaml`、`yarn.lock` 混在一起。再看 Vercel 构建日志中实际执行了什么安装命令。不要急着删文件，先确认团队原本使用哪种工具，并把改动记录在交付说明里。

## 常见原因二：构建命令不对

第二类问题是 build command 配错。比如项目实际需要 `npm run build`，但 Vercel 项目设置里写成了别的命令；或者 `package.json` 根本没有 `build` 脚本，本地只跑过开发命令，没跑过生产构建。

新手要养成一个习惯：上线前一定在本地运行一次生产构建，而不是只看开发环境。开发环境能打开，不代表生产构建能通过。尤其是 Next.js、Vite、Astro、Nuxt 这类框架，开发模式和构建模式对类型、导入路径、环境变量的要求可能不同。

## 常见原因三：环境变量缺失

很多 build failed 的真正原因是线上没有配置环境变量。本地 `.env.local` 里有 API Key、数据库地址或公开配置，但这些内容不会自动出现在 Vercel 项目里。日志里可能会看到 undefined、missing env、invalid URL、database connection、API key required 等线索。

排查时不要让客户直接把敏感值发到聊天窗口。更稳的做法是让客户在 Vercel 后台自行配置，或提供变量名清单和缺失项截图。你负责判断需要哪些变量、变量名是否一致、预览环境和生产环境是否都配置，而不是收集不该保存的秘密信息。

## 常见原因四：根目录或框架识别错误

如果仓库是 monorepo，或者项目放在子目录里，Vercel 可能默认从错误目录构建。表现是找不到 `package.json`、找不到入口文件、构建命令不存在，或者部署出来的是另一个项目。遇到这种情况，先看仓库结构，再看 Vercel Project Settings 里的 Root Directory。

框架识别也要检查。Vercel 通常能识别主流框架，但如果项目结构特殊，仍然需要人工确认 build command、output directory 和 install command。不要把“平台自动识别”理解成“永远不用检查”。

## 风险提醒

部署排查会碰到仓库权限、环境变量、域名、生产数据和客户账号。任何一步都要留下记录，特别是你改了哪些设置、为什么改、改之前是什么状态。不要在没有备份和验收标准的情况下改生产配置。

如果客户要求你直接操作主账号、删除部署记录、替换域名解析或处理支付相关配置，要先暂停确认边界。做项目时可以提供诊断、清单和建议，但不应该把不熟悉的高风险操作包装成简单小活。

## 具体步骤

第一步，复制最后一次失败部署的 build log，标出第一处真正报错，不要只看最后一行 failed。

第二步，在本地运行同样的安装和构建命令，确认问题是否能复现。如果本地也失败，先修代码或依赖；如果本地成功，重点查线上配置。

第三步，对照 `package.json`、lockfile、Node 版本、Root Directory、Build Command、Output Directory 和环境变量。每检查一项都写下结论，别靠记忆。

第四步，小步修改后重新部署，并保存新的预览链接。一次只改一类问题，否则成功了也不知道真正原因是什么。

第五步，完成后写一段交付记录：问题原因、修改项、验证方式、剩余风险、客户还需要自己确认的内容。后续可以把这类记录整理进 [工具导航](/tools) 或自己的案例库。
