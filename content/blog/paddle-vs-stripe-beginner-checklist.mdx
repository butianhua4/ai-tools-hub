# Paddle 和 Stripe 有什么区别：新手检查清单

这份清单用于比较 Paddle 和 Stripe 前的自查。它适合还没有开通真实收款账号、还没有上线支付页面、只是想判断自己下一步该查什么的新手。完整说明可以先看 [Paddle 和 Stripe 有什么区别](/blog/paddle-vs-stripe-beginner)。

清单不会替你注册账号、填写商户资料、判断税务责任或确认法律合规。涉及真实身份、银行账户、公司主体、费率、税务、消费者保护、退款政策和平台条款的部分，都应列为明日待办，由人工查官方文档后再决定。

## 适合谁

适合已经有一个数字产品、模板、轻量软件或订阅想法，但还没有决定使用 Paddle、Stripe 或其他收款方式的新手。你可以用它先整理事实，再决定明天查哪些官方文档。

也适合正在帮自己的网站补齐收款前资料的人。清单关注的是上线前判断，不是后台操作教程，所以不需要你现在输入真实身份、银行账户或客户数据。

## 不适合谁

不适合已经进入真实客户生产支付、税务申报、财务结算、公司法务或高风险商品销售的人。这些事项需要专业复核，不能只靠新手清单决定。

也不适合想跳过平台规则、减少退款说明、隐藏商品限制或临时凑一个付款链接的人。收款前的信息越模糊，后面的争议成本越高。

## 具体步骤

1. 先完成产品类型检查，确认你卖的到底是什么。
2. 再完成客户来源检查，确认是否已有平台收款规则需要遵守。
3. 分别看 Stripe 和 Paddle 方向检查，标出你无法确认的问题。
4. 把无法确认的问题写入明日人工复核清单，不要用猜测填空。
5. 如果准备上线页面，再逐项完成页面上线前检查。
6. 使用记录模板保存结论，方便后续复盘为什么选择某个工具。

## 1. 产品类型检查

- [ ] 我能用一句话说明自己卖的是服务、模板、软件、订阅、课程资料还是组合包。
- [ ] 我知道这个产品是否可以自动交付。
- [ ] 我知道客户付款后会收到什么：文件、链接、账号权限、授权码、服务交付还是预约确认。
- [ ] 我知道是否需要持续维护、售后支持或订阅续费。
- [ ] 我已经排除明显不适合新手独立处理的业务，例如客户生产支付系统、敏感数据、账户权限或复杂税务安排。

如果这些问题还答不上来，先不要比较工具。先把商品、交付和客户承诺写清楚，再谈收款方式。

## 2. 客户来源检查

- [ ] 客户来自自由职业平台时，我已经查看平台自己的收款和沟通规则。
- [ ] 客户来自独立站时，我知道页面需要隐私政策、联系邮箱、退款说明和商品描述。
- [ ] 客户来自社媒或邮件时，我知道需要留下订单证据和付款前确认记录。
- [ ] 我没有为了收款方便而引导客户离开已有平台规则。
- [ ] 我知道不同客户地区可能影响税务、发票、消费者提示和退款处理。

自由职业服务不一定需要 Paddle 或 Stripe。很多时候，先使用平台内置收款或一份清楚的付款前确认表，更适合新手。

## 3. Stripe 方向检查

- [ ] 我知道 Stripe 更适合需要自定义 Checkout、订阅、Webhook、客户门户或复杂支付逻辑的场景。
- [ ] 我知道测试模式和生产模式是两回事。
- [ ] 我准备记录支付成功、支付失败、取消订阅、退款、发票和客户邮件通知。
- [ ] 我知道 Stripe Tax、Billing、Checkout 等功能需要逐项确认，不应只凭二手教程上线。
- [ ] 我已经把账户注册、可用地区、身份验证、银行账户、费率和禁止业务列入明日待办。

Stripe 的优势是灵活，但灵活也意味着更多配置和责任。新手如果只是卖一个模板包，不一定需要从复杂支付系统开始。

## 4. Paddle 方向检查

- [ ] 我知道 Paddle 更常见于软件、订阅、授权或数字产品销售场景。
- [ ] 我知道 Paddle 可能承担一部分商户记录和税务处理相关工作，但仍需要审核和人工确认。
- [ ] 我准备确认商品是否符合 Paddle 当前政策。
- [ ] 我准备检查结账页、退款设置、商品状态、测试环境和真实上线流程。
- [ ] 我已经把服务地区、费率、结算周期、审核资料和税务说明列入明日待办。

Paddle 可能让数字产品销售更集中，但这不是“省掉所有合规工作”。商品描述、退款边界、客户支持和记录保存仍然要自己负责。

## 5. 页面上线前检查

- [ ] 商品页面写清楚产品内容、交付方式、适用对象和不适用对象。
- [ ] 页面没有承诺收入、成交、转化率或客户结果。
- [ ] 页面说明了退款、取消、支持范围和联系渠道。
- [ ] 页面没有使用未授权截图、Logo、客户案例或第三方素材。
- [ ] 页面没有把测试价格、测试链接或后台私密信息暴露给用户。
- [ ] 支付成功后的邮件、下载链接、订单记录和客户确认流程已经测试。

上线前至少用一个低风险测试商品走完整流程。不要第一次真实收款时才发现邮件没发、链接无效或订单状态没有记录。

## 风险提醒

不要把 Paddle 或 Stripe 的某个功能当成最终合规答案。功能是否可用、能否在你的地区开通、是否适合你的商品、是否需要税务设置，都要以当前官方文档和人工复核为准。

不要在未测试的情况下连接真实客户。至少要验证结账页、订单记录、邮件通知、退款路径、取消流程和交付链接。测试完成前，草稿应继续保持不公开。

不要在页面上写收入、成交或转化承诺。支付工具只能处理交易流程，不能替你保证客户结果、销售表现或长期收益。

## 6. 明日人工复核清单

- [ ] Paddle 当前支持哪些国家或地区的卖家。
- [ ] Stripe 当前支持哪些国家或地区的卖家。
- [ ] 两个平台的当前费率、结算周期、退款费用和争议处理规则。
- [ ] 个人主体和公司主体分别需要哪些资料。
- [ ] 是否需要税号、发票设置、销售税或增值税处理。
- [ ] 商品是否涉及禁止或受限业务。
- [ ] 消费者保护、退款说明、隐私政策和服务条款是否需要专业人员复核。

这部分不要靠记忆写结论。每次准备发布前，都应该重新查官方文档，因为支付平台规则会变。

## 7. 记录模板

```text
产品名称：
产品类型：服务 / 模板 / 软件 / 订阅 / 课程 / 其他
客户来源：
交付方式：
是否自动交付：
是否需要订阅：
候选工具：Paddle / Stripe / 其他
必须查证的官方文档：
需要人工复核的问题：
暂不处理的高风险事项：
明日下一步：
```

把这份记录保存下来，比临时听一个推荐更可靠。后续如果客户、价格或产品类型变化，也能看出为什么之前选了某个方向。
