# Codex 生成代码后怎么人工审核

Codex 能把需求快速变成代码，但它给出的结果不能直接等同于可交付作品。人工审核的价值不在于把每一行都背下来，而是确认三件事：它改了什么、为什么这样改、在目标环境里会不会造成新的问题。对项目新手来说，这一步尤其重要，因为客户最终验收的是可运行、可解释、可维护的结果，而不是一段看起来很完整的 AI 输出。

这篇文章提供一套偏保守的审核流程。它适合用于练习项目、作品集项目、低复杂度修复，也适合在项目前判断自己是否能处理某个需求。若你想要更短的版本，可以继续看 [人工审核检查清单](/blog/codex-review-generated-code-checklist)；若你已经遇到改坏页面、测试失败、范围失控等问题，可以看 [常见错误和修复顺序](/blog/codex-review-generated-code-mistakes)。

## 适合谁

适合刚开始使用 Codex、Claude Code、ChatGPT 或其他 AI 编程工具的人。你可能已经能让工具生成 React 组件、修复报错、补一个脚本、改一个样式，但还不确定怎样判断结果是否可靠。你也可能正在准备 AI 辅助项目，希望把一次练习整理成可展示的交付记录。

也适合已经收到客户需求、但还没有决定是否报价的人。比如客户说“帮我修一下网站按钮”“把这个接口接好”“页面在移动端乱了”，这些话都不够完整。审核流程能帮助你先把需求、权限、测试方式和验收标准问清楚，再决定下一步。

## 不适合谁

不适合想把 AI 输出直接当最终交付的人。代码审查需要真实运行、阅读差异、检查依赖、保留截图或日志，并在不确定时暂停。它也不适合处理高风险系统的新手独立承接，例如生产数据库、支付流程、账号权限、安全漏洞、隐私数据、大规模线上服务等。遇到这些场景，应先找有经验的人共同判断，或只把它作为学习记录。

也不适合把项目当成立刻获得稳定收入的人。自由职业需要作品、沟通、交付记录和持续信誉。Codex 可以提升整理和验证效率，但不能替代经验、测试和边界判断。

## 风险提醒

Codex 可能会生成能通过局部检查、但不符合整体需求的代码。常见风险包括：修改了不该动的文件、删掉了隐含业务逻辑、引入未安装依赖、绕开了原有错误处理、让移动端样式变形、把临时调试内容留在页面里，或把客户没有授权的功能假设为可用。

项目场景还要注意平台要求、客户授权和数据边界。不要要求客户提供超出任务所需的敏感凭据，不要把客户代码上传到不可信位置，不要承诺自己没有验证过的效果。任何涉及真实用户数据、付款、登录、权限和安全的改动，都应该提高审核级别。

## 具体步骤

1. 先保存原始需求。把客户消息、报错截图、当前链接、仓库分支、目标效果和截止时间放在同一个记录里。不要只保留自己理解后的版本。

2. 再确认 Codex 的输入。检查你给它的 prompt 是否包含技术栈、运行命令、不能改动的边界、验收标准和失败日志。输入越含糊，输出越需要谨慎对待。

3. 查看代码差异。优先看新增文件、删除文件、依赖变更、配置变更和共享组件变更。只改样式的任务不应顺手改业务逻辑；只修一个报错的任务不应大面积重写结构。

4. 本地运行最小验证。前端项目通常至少跑 lint、build 和关键页面预览；脚本项目至少跑示例输入；接口项目至少跑一次成功请求和一次失败请求。若你不知道怎么解释报错，可以先用 [报错解释器](/tools/error-explainer)拆解日志。

5. 做人工阅读。重点看变量名、条件分支、异常处理、空值处理、权限判断、默认值、文案和边界状态。不要只相信“编译通过”。

6. 记录交付证据。包括改动摘要、运行命令、截图、测试结果、未覆盖风险和需要客户确认的问题。若准备报价，可以用 [报价计算器](/tools/pricing-calculator)估算时间，再用 [Proposal 生成器](/tools/proposal-generator)整理成谨慎版本。

7. 决定是否继续。如果你无法复述代码为什么这样改，无法在本地复现结果，或无法说明失败时如何回退，就先不要把它当作可交付范围。

## 审核时优先看什么

第一优先级是安全和权限。任何新增的 token、密钥、账号信息、跨域设置、数据库连接、管理员入口，都要单独检查。第二优先级是功能路径。用户点击、输入、提交、刷新、失败重试这些路径是否仍然连贯。第三优先级是维护成本。Codex 有时会把简单问题改成复杂结构，短期可运行，长期难维护。

如果任务只是调 CSS，也要打开桌面和移动端看一遍；如果任务是修 API，也要检查错误响应；如果任务是改文案，也要确认没有误导客户或用户。人工审核不是为了找工具的错，而是为了让结果能够承担真实场景里的后果。

## 可以给客户的说明模板

```text
我会先做一次小范围检查：确认当前问题、查看代码差异、在本地运行验证，并记录仍需确认的边界。
如果发现任务涉及生产数据、账号权限或安全相关改动，我会先暂停并和你确认处理方式。
本次交付会包含改动摘要、测试结果和后续建议。
```

这段话的重点是设定边界。你不是承诺所有问题都能一次解决，而是说明自己会用可复核的方式推进。
