# TypeScript 报错新手怎么排查

TypeScript 报错不一定代表程序坏了，它通常是在告诉你“代码和类型约定不一致”。新手最容易做错的事，是只复制最后一行错误给 AI，然后让 AI 改一大段代码。更稳的方式是保存完整报错、定位文件行号、理解类型期待，再小步验证。

这篇仍是草稿，发布前需要人工补充真实截图和不同框架示例。

## 适合谁

适合正在调试 Next.js、React、Vite、Node.js 或小型工具项目的新手。你能运行 `npm run build` 或 `tsc`，也能打开报错指向的文件。

也适合准备接报错修复小单的人。你需要先判断这是简单类型补齐，还是项目依赖、配置和架构问题。

## 不适合谁

不适合直接在客户生产分支上试错。也不适合没有仓库、没有完整报错、没有运行命令的需求。

如果报错来自支付、安全、客户数据、权限系统或复杂后端，不建议新手独立处理。

## 报错通常在说什么

TypeScript 报错常见分几类：变量可能为空、属性不存在、类型不匹配、函数参数不对、第三方库类型缺失、配置没有包含某些文件。每一类都要用不同方式排查。

例如“Property does not exist”通常要看对象来源和类型定义；“Type is not assignable”通常要看传入值和目标类型；“Cannot find module”则可能是依赖、路径或类型声明问题。

## 具体步骤

1. 保存完整命令和完整报错。
2. 记录 TypeScript、框架和 Node 版本。
3. 定位报错文件、行号和相关类型。
4. 先读错误期待的类型，再看当前传入的值。
5. 检查是否最近升级依赖或改过配置。
6. 尝试最小改动，不要一次重构很多文件。
7. 重新运行 `npm run build` 或项目指定检查命令。
8. 把修复原因和验证命令写进记录。

## 接单前怎么判断

如果客户能提供仓库、完整报错、运行命令和预期行为，可以先做诊断。如果客户只说“TS 报错，帮我修一下”，要先追问构建命令、报错截图、最近改动和是否有测试环境。

报价时可以拆成诊断和修复。诊断交付错误分类、影响范围和建议；修复阶段再根据实际文件数量和风险报价。

## 修复后怎么验收

TypeScript 修复后，验收不能只看编辑器不再飘红。至少要运行项目指定的检查命令，例如构建、类型检查或测试。还要确认修复没有把类型保护删掉，没有用大量 `any` 把错误藏起来。

交付给客户时，可以写三句话：第一句说明原错误是什么；第二句说明改了哪些文件；第三句说明用什么命令验证通过。如果还有未处理的相关错误，要单独列出，不要让客户误以为整个项目都已经检查完。

如果这次只是诊断，也要输出清楚结论。比如“当前错误来自依赖类型变化，需要确认是否继续升级”，或者“问题集中在接口返回值可能为空，建议先补数据判断”。诊断不是聊天，它也应该有交付物。

## 常见客户追问

客户可能会问“为什么以前能跑，现在不行”。这时不要猜测，要回到最近改动、依赖版本和构建命令。你可以解释 TypeScript 会随着版本、类型声明和配置变化而暴露新的问题。

客户也可能问“能不能直接关掉检查”。新手要谨慎回复：关闭检查可能让构建通过，但会降低后续维护质量。除非客户明确只是临时绕过，并且已经知道风险，否则不建议这样做。

## 风险提醒

不要为了消除报错随意使用 `any`。不要关闭严格检查来掩盖问题。不要在没有测试和回滚方式时改共享类型。不要把看不懂的依赖升级当成默认修复。
