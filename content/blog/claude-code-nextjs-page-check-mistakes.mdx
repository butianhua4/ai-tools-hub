# Claude Code 检查 Next.js 页面时的常见错误

Next.js 页面小问题看起来简单，但很容易被新手做大。客户说一个页面不对，你可能顺手查到布局、接口、SEO、登录、缓存和部署，最后范围完全失控。Claude Code 能帮你找线索，但你要负责控制任务边界。

这篇适合搭配 [Claude Code Next.js 页面检查清单](/blog/claude-code-nextjs-page-check-checklist) 和 [Claude Code 交付说明检查清单](/blog/claude-code-delivery-note-checklist) 使用。

## 适合谁

适合已经会用 Claude Code 读 Next.js 项目，但经常不知道该停在哪里的新手。你可能原本只想改一个按钮，最后变成重构整个页面。

也适合准备接页面小改服务的人。页面小改能做，但必须把页面、组件、数据和验证方式写清楚。

## 不适合谁

不适合高权限页面任务。涉及后台、付款、真实用户数据、生产配置和账号权限时，不要按普通页面小改单处理。

也不适合客户只说“整体不好看”“帮我优化一下”的模糊任务。这类更像设计或产品需求，需要重新拆范围。

## 具体步骤

错误一：不问页面路径。客户说首页、详情页、后台页都可能不准确。要拿到 URL、路由或截图位置。

错误二：不问预期效果。页面“不对”可能是文案、样式、交互、数据或权限问题。没有预期，就没有验收。

错误三：不确认数据来源。页面显示异常可能来自接口、静态配置、环境变量、CMS 或登录状态。不要只盯组件代码。

错误四：不区分本地和生产。你本地看到正常，不代表客户预览环境正常；客户线上异常，也不代表本地代码一定坏。

错误五：把页面小改扩成全站重构。只处理一个明确页面和一个明确问题，不要顺手改设计系统。

错误六：不写未覆盖范围。比如未处理支付流程、未处理登录权限、未执行生产部署，这些都要写进交付说明。

## 修正模板

```text
目前我可以先检查这个页面的一个明确问题。请确认：
- 页面 URL 或路由
- 复现步骤
- 正常应该是什么样
- 现在哪里不对
- 是否依赖登录、接口、环境变量或测试数据
- 修完后如何验收
```

这段能帮你把“页面不对”变成可执行任务。

## 怎么让 Claude Code 不跑偏

给 Claude Code 的输入越具体，它越不容易跑偏。不要只说“帮我检查页面”，而是说“检查 `/pricing` 页面按钮文案和移动端间距，不处理支付流程，不改全局样式，先列相关文件和低风险验证步骤”。

如果工具建议改全局组件或布局文件，要先判断影响范围。全局文件一改，可能影响多个页面，不适合作为没有复核的小单动作。

## 交付时怎么写

交付说明要明确页面路径和验证方式。例如“已调整 `/templates` 页面卡片间距，本地 build 通过；未处理下载权限和生产部署；请在测试环境移动端宽度下确认卡片不重叠”。

如果没有条件完整验证，就写“已完成代码层面修改，因缺少测试账号未验证登录后流程”。不要把未验证内容写成已完成。

## 改写示例

不推荐写：“页面已经改好了。”客户不知道你改的是哪个页面，也不知道怎么确认。

更稳的写法是：“本次只处理 `/tools` 页面移动端卡片间距问题，修改了页面局部样式；本地 build 通过；未处理工具详情页、登录态和生产部署；请在测试环境手机宽度下确认卡片间距正常。”

这段话把页面、范围、验证和未覆盖都写出来了。它看起来多几行，但能明显减少返工沟通。

## 复盘怎么做

每次页面交付后，记录客户是否一次验收。没有一次验收时，回看是不是页面路径没写清、验收动作太模糊，还是未覆盖范围没提前说。

把这些问题加回下一次页面检查模板。小单越多，模板越应该变得更具体。

## 风险提醒

不要让客户把含有真实用户资料的页面截图原样发给你。截图需要遮挡或使用测试数据。

不要把页面检查说成全站质量保证。页面小单只能证明本次范围内的问题，不兜底所有页面。
