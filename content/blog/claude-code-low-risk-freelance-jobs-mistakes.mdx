# Claude Code 低风险项目常见错误

新手用 Claude Code 做项目时，经常把“客户说简单”当成低风险。真正的低风险要看权限、影响范围、复现、验证和验收。只要这些不清楚，任务就不应该直接进入修复。

这篇适合搭配 [Claude Code 低风险项目检查清单](/blog/claude-code-low-risk-freelance-jobs-checklist) 和 [Claude Code 小项目任务检查清单](/blog/claude-code-small-project-tasks-checklist) 使用。

## 适合谁

适合已经开始用 Claude Code 接小单，但经常被任务范围拖大的新手。

也适合准备写报价模板的人。先知道常见误判，后面沟通会稳很多。

## 不适合谁

不适合高风险生产任务。涉及真实数据、付款、账号权限、数据库、安全事件和长期维护时，不要套低风险模板。

也不适合客户不给信息却要求你承诺结果的任务。信息不足时，只能先诊断。

## 具体步骤

错误一：只听客户说简单。客户的感觉不是风险评估。你要看复现、权限、范围和验收。

错误二：不问权限。页面小改也可能牵涉后台、账号、付款或真实数据。权限不清时不要承诺修复。

错误三：不写验收方式。没有验收，小单就没有结束点。

错误四：不写未覆盖范围。客户会默认部署、长期维护、其他页面都包含，除非你明确排除。

错误五：把诊断做成长期支持。诊断交付的是判断和下一步，不是无限陪客户排查。

错误六：因为 Claude Code 能给很多建议，就把建议都变成本次任务。建议可以记录，执行要重新确认范围。

## 修正模板

```text
我可以先判断这个任务是否低风险。本次需要确认目标、复现步骤、权限边界和验收方式。如果信息不足，我建议先做诊断，不直接承诺修复。
```

这段话能把任务重新拉回可判断状态。

## 改写示例

不推荐写：“这个很简单，我可以做。”这句话没有风险判断。

更稳的写法是：“如果只处理一个页面文案并由你在测试环境验收，我可以接小范围任务；如果涉及登录、付款或生产部署，需要重新确认范围。”这句话既给出可做部分，也排除了高风险部分。

## 复盘问题

每次返工后，问自己：一开始是否问了权限？是否写了未覆盖范围？客户追加的内容是否属于新需求？Claude Code 的建议是否被你无意扩成了交付承诺？

把这些问题写进下一次报价模板。新手项目练习的稳定性来自重复复盘，而不是一次性接更大项目。

## 错误交付示例

不推荐写：“我已经帮你处理好了，后面有问题再找我。”这句话会让客户以为后续问题也包含在本次范围里。

更稳的写法是：“本次完成一个页面文案调整，本地检查通过；未处理其他页面、生产部署和长期维护。请在测试环境确认指定页面文案。”这才是低风险交付。

## 下次怎么改模板

如果这次被追加需求拖住，下次报价模板里就加“新增问题另行确认”。如果这次因为权限不清返工，就加“是否涉及账号、真实数据或后台权限”。模板越具体，越少临场扯皮。

Claude Code 可以帮你生成模板，但哪些边界要保留，必须由你决定。

## 低风险变高风险的信号

客户开始要求你登录后台、查看真实用户、改付款配置、接管部署、长期盯错误时，低风险任务已经变质。此时不要继续沿用原报价。

你可以继续沟通，但要换成新范围。把新增权限、验收、责任和回滚方式写清楚，再决定是否承接。

## 交付句式

```text
本次只完成低风险范围内的诊断/修改，未处理生产、账号、付款、真实数据或长期维护。请按交付说明中的步骤验收。
```

这句可以放在交付结尾，帮你收住边界。

## 风险提醒

不要为了显得能干，把中高风险任务包装成低风险。风险分类错了，后面很难收口。

不要承诺收益、排名、长期稳定或所有问题都解决。只承诺本次范围内的工作和验证。
