# Node 版本不匹配接单前怎么判断能不能做

Node 版本不匹配是适合新手练习的报错类型，但接单前要先判断它是本地环境问题，还是客户 CI、部署平台和团队配置问题。能本地切版本解决的任务，和需要客户平台权限的任务，范围完全不同。

如果你只想学习排查顺序，可以先看 [Node 版本不匹配怎么排查](/blog/node-version-mismatch-fix)。这一篇重点讲接单边界。

## 适合谁

适合会看 `node -v`、`package.json`、锁文件和构建命令的新手。你可以处理本地版本不一致、包管理器用错、项目声明不清这类低风险诊断。

也适合做作品集练习。你可以在自己的项目里设置 `.nvmrc`，模拟本地和 CI 版本不一致，再整理修复记录。

## 不适合谁

不适合完全不了解 Node、npm、pnpm、yarn 和 CI 的人直接处理客户部署问题。部署失败可能牵涉平台设置和生产环境。

也不适合客户不给仓库、不提供报错、不允许你看 package.json，却要求你马上修好。

## 风险提醒

不要在客户项目里随意升级 Node 或删除锁文件。也不要代客户登录部署平台修改设置。涉及 Vercel、Netlify、服务器、GitHub Actions、Docker 的版本配置时，需要客户确认。

如果需要注册平台、开通权限、添加协作者或修改组织设置，把它写成明日待办或客户侧确认事项。

## 具体步骤

### 1. 接单前要材料

至少要客户提供：

- 完整报错。
- 本地 `node -v`。
- 包管理器版本。
- `package.json`。
- 锁文件类型。
- CI 或部署失败链接。
- 是否允许访问仓库。

材料不足时，只报价诊断，不报价完整修复。

### 2. 判断可接范围

适合新手的范围包括：确认版本差异、切换本地版本、按项目包管理器安装、运行 build、写 CI/部署配置建议。

不建议新手独立处理的范围包括：生产服务器升级、组织 CI 策略、复杂 Docker 镜像、私有依赖迁移、大型 monorepo 环境重构。

### 3. 报价怎么拆

可以拆成：

```text
Diagnosis:
- Compare local Node version, project requirements, package manager, and build logs.
- Identify whether the issue is local, CI, deployment, or dependency compatibility.

Fix:
- Apply local or repository changes after confirmation.
- Deployment platform and organization settings require the project owner to confirm.
```

这样能防止你把客户平台权限问题包进一个小修复单。

### 4. 明日待办怎么写

如果卡在客户平台，写成待办：等待客户确认部署平台 Node 版本、添加仓库权限、提供 CI 日志、确认是否允许修改 workflow。待办要写负责人。

例如：明日等待客户确认 Vercel 项目 Node 版本；确认后重新运行 build 并记录结果。这样你的进度和客户授权边界都清楚。

## 暂缓信号

遇到这些情况先暂缓：客户要求你直接改生产服务器；项目缺少锁文件且没人能确认包管理器；CI 和本地都没有完整日志；客户要求删除锁文件；你无法运行任何验证命令。

暂缓不是拒绝，可以先交付诊断清单。

## 适合做成作品集吗

Node 版本不匹配很适合做匿名作品集，因为它能展示你的环境判断能力。作品集可以展示本地版本、项目声明、CI 版本、修复前后验证命令，但不要展示客户仓库地址、内部部署平台、私有日志或账号信息。

如果最终原因是客户部署平台没有配置正确 Node 版本，也可以写成案例：如何判断本地代码已经通过，而外部平台还需要客户确认。这类案例体现的是交付边界，不一定要有大量代码修改。

公开前把所有路径、仓库名和日志脱敏。真实客户没有授权时，只保留问题类型和处理过程。

## 交付边界怎么写

可以把交付边界写成：本次先诊断 Node、包管理器、锁文件、CI 和部署平台版本差异；能在仓库内修复的配置会提交修改；需要客户平台权限的设置，由客户确认后再复测。这样客户知道你不是不处理，而是在把权限放回正确位置。

如果任务只允许诊断，就不要承诺修复所有环境。交付物可以是版本差异表、建议目标版本、复测命令和客户待办。对新手来说，这种清楚的诊断单比盲目改生产平台更可靠。
