# Codex 安装前置条件常见错误和修复顺序

新手配置 Codex 环境时，很多错误看起来像工具问题，其实是前置条件没有准备好。比如终端不在项目目录、Node 版本太旧、npm 路径混乱、Git 不可用、网络访问失败、账号权限缺失。越急着复制下一条命令，越容易把环境改乱。

这篇文章按“先保留现场，再缩小范围”的思路整理。完整准备流程见 [安装 Codex 前要准备什么](/blog/codex-install-prerequisites)，短表版见 [安装前置条件检查清单](/blog/codex-install-prerequisites-checklist)。

## 适合谁

适合已经遇到安装失败、命令不存在、版本报错、权限提示或网络错误的新手。你可能不知道应该先修哪个，也可能已经重复安装过几次，电脑里出现多个 Node 或 npm。

也适合准备项目前做环境排查的人。客户项目跑不起来时，你需要先分清是自己环境问题、项目文档问题，还是客户权限缺失。

## 不适合谁

不适合在公司或客户受管设备上随意修改系统设置的人。设备策略、代理、证书、管理员权限都可能影响安装，不能用个人电脑经验直接套。

也不适合把所有问题都交给 AI 反复生成命令的人。AI 可以解释日志，但执行前你要知道命令会改哪里。

## 风险提醒

安装排错最大的风险是越修越乱。重复安装 Node、清理全局包、修改 PATH、删除 lockfile、切换镜像源，都可能影响其他项目。新手应该先记录，再做小改动。

另一个风险是授权和密钥。遇到 token、API key、平台登录、私有仓库访问时，不要把凭据贴进公开聊天或文档，也不要使用不属于该项目的账号。

## 具体步骤

### 第一步：保留现场

先停止继续安装，把当前报错、命令、所在目录、系统、Node 版本、npm 版本和 Git 版本记录下来。不要只写“安装失败”，要保留完整错误文本。

可以用 [报错解释器](/tools/error-explainer)把日志拆成现象、可能原因、验证命令和下一步。

### 第二步：确认所在目录

很多错误来自终端不在项目目录。先运行 `pwd` 或查看当前位置，再确认这里是否有 package.json、README 或项目文件。没有项目文件时，安装命令很可能跑错地方。

### 第三步：检查版本

运行 `node -v`、`npm -v`、`git --version`。如果命令不存在，说明基础工具缺失；如果版本太旧，要根据项目说明决定升级方式；如果多个终端输出不同版本，要先处理路径问题。

### 第四步：区分权限问题

如果提示 permission、access denied、administrator、policy、certificate 等关键词，先判断是不是权限或设备策略。不要随意用管理员权限重跑所有命令，更不要关闭安全设置。

### 第五步：只改一个变量

每次只做一个修复动作。比如先修 Node 版本，再跑验证；先补 Git，再重新检查；先确认网络，再安装依赖。每次改完都记录结果。

## 常见错误清单

第一类是路径错误：终端不在项目目录，或者把命令跑到桌面、下载目录、系统目录。第二类是版本错误：Node 太旧、npm 太旧、项目要求和本机版本不匹配。第三类是 Git 缺失：无法克隆、无法查看状态、无法保存变更。

第四类是权限错误：没有管理员权限、公司设备限制、文件被占用。第五类是网络错误：访问包源失败、代理证书问题、下载中断。第六类是账号错误：需要登录、授权或客户提供仓库权限。

## 修复顺序示例

假设你运行安装命令后提示 `node` 不是可识别命令。不要先安装 Codex，先确认 Node 是否安装、终端是否重启、PATH 是否生效。确认 `node -v` 可用后，再继续。

假设 npm install 一直失败。先看项目 README 要求的 Node 版本，再看 lockfile 和包管理器。不要马上删除所有依赖文件。若仍然不懂，可以记录日志，作为客户或明日待办。

## 什么时候应该暂停

如果错误已经涉及客户账号、私有仓库权限、公司设备策略、代理证书、生产服务器、支付配置或真实用户数据，就不要继续按普通安装问题处理。你可以把当前状态整理成问题清单，交给客户或更有经验的人确认。暂停不是失败，而是边界判断。

还有一种情况也要暂停：你连续尝试几轮后，环境状态越来越不清楚。比如同一台电脑里出现多个 Node，PowerShell 和 VS Code 终端版本不一致，npm 缓存、镜像、全局包都被改过。这时继续试命令只会增加变量。更稳的做法是重新记录当前状态，必要时新建一个干净练习目录。

## 交付记录怎么写

如果这是项目任务，不要只回复“已经处理”。更好的交付记录包括：客户原始问题、你检查过的版本、运行过的命令、失败或成功结果、未覆盖的风险、需要客户确认的事项。这样即使问题还没完全解决，客户也能看到你做了哪些有效排查。

准备报价或追加范围时，可以用 [报价计算器](/tools/pricing-calculator)估算环境排查耗时，再用 [Proposal 生成器](/tools/proposal-generator)整理下一步建议。所有自动生成内容都要人工删改，尤其是涉及账号、权限和结果承诺的句子。
