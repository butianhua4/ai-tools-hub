# Claude Code 新手适合先接哪些低风险小单

Claude Code 能帮新手更快理解代码和整理修改方案，但它不能替你承担项目风险。刚开始做项目时，最重要的不是接“大项目”，而是先接低权限、低影响、可复现、可验收的小任务。这样的任务更容易练出交付节奏，也更不容易踩到客户权限和生产风险。

这篇适合搭配 [Claude Code 小项目任务检查清单](/blog/claude-code-small-project-tasks-checklist)、[Claude Code 项目客户问题检查清单](/blog/claude-code-client-questions-checklist)、[Claude Code 交付说明检查清单](/blog/claude-code-delivery-note-checklist) 和 [/templates](/templates) 使用。

## 适合谁

适合刚开始尝试 AI 代码项目的新手。你可能能看懂基础代码、能运行简单命令、能让 Claude Code 帮你解释项目，但还不适合接高权限系统和长期维护。

也适合已经接过一两个小单，但发现客户经常追加需求的人。低风险筛选能帮你把任务限定在自己能交付的范围内。

## 不适合谁

不适合客户要求你处理付款、账号权限、数据库、生产部署、真实用户数据、安全事件或长期值守的任务。这些都不是新手低风险小单。

也不适合没有复现步骤、没有测试环境、没有验收标准，却要求固定结果的任务。信息不足时，只能先做诊断。

## 具体步骤

第一，选页面和文案类任务。例如调整一个按钮文案、改一段 README、修一个页面小样式。前提是页面路径和验收方式明确。

第二，选单报错诊断。例如构建失败、依赖冲突、控制台报错。先拿脱敏错误原文，再让 Claude Code 帮你列原因和下一步。

第三，选单文件检查。例如客户给一个 GitHub 文件，让你判断一个明显问题。只看指定文件和必要相邻上下文，不做全仓库审计。

第四，选交付说明和需求清单。例如把客户模糊需求拆成问题清单，或把修改结果写成验收说明。这类任务低权限，很适合练流程。

第五，排除高风险任务。只要涉及生产权限、真实数据、付款、数据库、后台配置、长期维护，就不要按低风险小单处理。

第六，给每个任务写“完成条件”。比如本地 build 通过、页面截图符合预期、客户在测试环境确认、文档覆盖指定步骤。

## 低风险判断标准

低风险任务通常满足五个条件：目标单一、权限低、影响范围小、可在本地或测试环境验证、交付后客户能按步骤验收。缺少其中两项以上，就应该先做诊断。

不要只听客户说“很简单”。客户判断的是感觉，你要判断的是复现、权限、范围和验收。

## 交付怎么写

交付说明可以写：“本次只处理一个明确问题，已运行本地检查，未处理生产部署、真实数据、账号权限和长期维护。请按以下步骤验收。”这类表达能让小单有明确结束点。

如果只是诊断，也要写清已确认事实、可能原因、需要补充材料和是否适合继续修复。诊断是有效交付，不是“没做完”。

## 怎么分层开始

第一层先做只读诊断：看脱敏错误、读指定文件、整理可能原因。第二层才做小范围修改：只改一个明确问题，并运行基础验证。第三层才考虑多文件影响检查或多轮验收支持。

新手不要把三层混成一个低价小单。分层之后，客户也更容易理解为什么有些任务只能先诊断，不能马上承诺修复。

## 什么时候暂停

如果客户开始要求你接收账号、处理生产部署、查看真实数据、修改付款配置，或者让你长期盯项目，就要暂停原范围。暂停时可以说：“这已经超过低风险小单，我需要重新确认授权、范围和验收方式。”

暂停不是拒绝客户，而是防止任务变成你无法控制的风险。

## 风险提醒

不要把低风险小单变成全项目支持。Claude Code 给出的额外建议可以记录，但是否执行必须重新确认范围。

不要接收客户账号、验证码、私有 token 或真实数据截图。需要授权时，让客户在自己的环境中完成。
