# 作品集页面文案模板怎么写真实

作品集页面不是用来吹得很满，而是让客户快速判断：你做过什么、能解决什么小问题、如何验证交付、哪些范围暂时不接。新手最容易犯的错，是把“正在练习”写成“全能服务”，结果后面沟通和交付都会失真。

这篇仍是草稿。正式发布前需要人工补充真实截图、案例授权说明和平台规则复核，尤其要确认没有把客户资料或未授权素材放进公开页面。

## 适合谁

适合正在准备第一个作品集页面的新手。你可能只有练习项目、模拟案例或小型自建页面，但愿意把过程、截图、验证命令和反思记录下来。

也适合想用 Codex、ChatGPT 或其他 AI 工具辅助整理案例的人。AI 可以帮你改文案，但最终真实性、授权边界和交付能力要由你自己确认。

## 不适合谁

不适合想复制别人作品集、夸大项目经验、使用客户截图但没有授权的人。也不适合把 AI 生成案例当成真实客户案例的人。

如果案例涉及客户名称、后台截图、用户数据、支付页面、登录流程或私有代码，不能直接公开。需要脱敏、授权或改成内部练习记录。

## 页面应该写什么

一个新手作品集页面可以分成五块：

1. 你能做的低风险任务，例如静态页面、CSS 修复、报错诊断、部署前检查。
2. 你做过的练习或案例，说明目标、过程、结果和验证方式。
3. 你使用的工具，但不要只堆工具名，要说明工具在流程里的作用。
4. 你的接单边界，明确不处理生产数据库、支付、安全漏洞、敏感账号等高风险事项。
5. 联系方式和下一步，让客户知道应该提供什么材料。

重点是“可信”，不是“看起来厉害”。

## 具体步骤

1. 先列出 3 个你真实完成过的练习项目。
2. 每个项目写清楚目标、输入材料、你做了什么、怎么验证。
3. 截图前检查是否包含客户信息、账号、密钥、邮箱或后台数据。
4. 把工具名放在流程里写，例如“用 Codex 辅助解释报错，用 git diff 审核改动”。
5. 写出你暂时不接的任务范围，避免客户误解。
6. 用手机和桌面预览作品集页面，确认文案不溢出、不遮挡。
7. 发布前请别人复核一遍真实性和风险边界。

## 可复制文案模板

```text
我主要做低风险前端和工具辅助任务：
- 静态页面文案和样式调整
- 小型 CSS bug 诊断
- npm install / build 报错整理
- 部署前检查和交付说明

我会先确认需求、复现问题、记录改动，再给出可验证的交付结果。
暂不独立处理生产数据库、支付系统、客户敏感账号、安全漏洞和大规模后端改造。
```

这段模板要按你的真实能力删改。不会的内容不要写，会的内容也要说明验证方式。

## 一个更真实的写法

如果你只做过练习项目，可以这样写：“我正在用小型练习项目训练前端交付流程，目前完成过一个单页作品集、一个 CSS 问题排查和一次构建报错记录。每个练习都保留了截图、命令和复盘笔记。”这比写“精通全栈开发”更可信。

如果你做过真实小单，也不要只写“完成客户网站”。更好的写法是：“整理了首页文案结构，修复了移动端按钮换行问题，交付前后截图和构建记录。客户确认范围只包含静态页面调整，不包含后端、支付和长期维护。”这样的表达能让客户看到边界，也能减少误会。

作品集文案可以有一点自信，但每一句都要能拿出证据。没有截图、记录、提交或客户确认的问题，先放进内部草稿，不要急着公开。

## 风险提醒

不要承诺收入、成交、排名、转化率或客户一定满意。不要公开客户私有截图、代码、合同、聊天记录或后台数据。不要使用未授权图片、品牌素材和竞品文案。

如果需要注册作品集平台、绑定域名、连接 GitHub 或上传客户案例，先记录为明日待办，并在授权确认后再处理。

## 明日待办

- 补充一张不含客户信息的作品集页面截图。
- 人工核对案例是否有公开授权。
- 检查图片、图标、字体和模板来源。
- 准备一个客户询问表，说明联系前需要提供哪些材料。
