# AI 自动化接单常见合规错误和解决步骤

先给结论：AI 自动化接单最常见的合规错误，不是脚本写错，而是默认权限太大。系统一旦能自动发布、自动承诺、自动使用敏感数据或自动改公开索引，就必须有清晰的人工审核和日志。

这篇文章把常见错误拆成可修正步骤。它适合用来检查已有流程：内容自动化、Proposal 工具、交付脚本、部署脚本和站点 SEO 检查。

## 适合谁

适合已经开始搭建自动化的人。比如你有 500 个选题计划、很多 MDX 草稿、内容质量检查脚本、Vercel 部署、Proposal 初稿工具或报错解释器。

也适合担心“自动化会不会越界”的人。你不需要停止自动化，但需要把自动化从“替我对外行动”改成“帮我准备和检查”。

## 不适合谁

不适合想靠自动化快速绕过审核、批量制造内容、批量联系客户、伪造作品集或处理不该处理的数据的人。这些做法会伤害账号、网站和客户信任。

如果你还没有人工审核流程，也没有日志，就不要让脚本执行公开动作。先让它只做草稿和检查。

## 具体步骤

排查自动化合规错误，可以按四步做：

1. 列出所有自动动作：生成、改状态、发布、提交、发送、部署、收款、调用 API。
2. 标记每个动作的影响范围：本地、后台、公开页面、客户、平台、生产环境。
3. 给高风险动作加人工确认：公开发布、客户消息、付款、生产配置和敏感数据都要确认。
4. 跑检查并保存结果：内容检查、SEO 检查、构建检查、线上检查都要有记录。

如果某个动作说不清输入、输出和回退方式，就先禁用它。

## 错误一：把草稿直接公开

表现是：脚本生成文章后顺手改成 published，或把 draft 加进 sitemap，或把 `noindex` 改成 false。

危险在于，草稿可能有模板句、乱码、事实错误、版权风险和平台风险。搜索引擎看到低质量内容，会影响整站可信度。

解决步骤：

```text
1. 默认 status=draft。
2. 默认 noindex=true。
3. sitemap 只收录 published 且 noindex=false 的内容。
4. 发布前先跑内容质量检查。
5. 发布必须人工确认，不由脚本单独决定。
```

## 错误二：让 Proposal 工具直接发送

表现是：客户需求输入后，工具自动生成并提交 Proposal，或者把报价和工期也自动填完。

危险在于，Proposal 是承诺。AI 可能误解需求、夸大能力、忽略平台限制或给出你无法完成的范围。

解决步骤：

```text
1. Proposal 工具只生成初稿。
2. 人工删掉夸大承诺。
3. 人工确认项目是否适合自己。
4. 人工确认报价、工期和交付边界。
5. 不清楚需求时先提问，不急着投。
```

## 错误三：敏感数据进入普通自动化流程

表现是：脚本、提示词或日志里出现真实 API Key、客户账号、付款截图、生产数据库字段、用户隐私信息。

危险在于，日志和提示词经常会被复制、提交、同步或分享。敏感信息一旦混进普通流程，很容易泄漏。

解决步骤：

```text
1. 使用模拟数据。
2. 删除邮箱、手机号、Token、订单号和账号截图。
3. 把环境变量放在平台配置里，不写进仓库。
4. 日志只记录必要摘要，不记录密钥。
5. 定期搜索仓库里是否出现疑似密钥。
```

## 错误四：没有把规则写进检查脚本

表现是：你口头知道“draft 不能公开”，但脚本没有检查；你知道“发布要人工确认”，但命令没有确认参数；你知道“质量分太低不能发布”，但没有门禁。

危险在于，人会忘，脚本不会主动理解你的原则。规则只存在脑子里，自动化迟早会跑偏。

解决步骤：

```text
建议写进脚本的规则：
- draft/review 不进入 sitemap
- noindex=true 不公开索引
- 发布需要人工确认参数
- 低质量分阻止发布
- 乱码和模板句阻止进入审核
- live check 失败不能继续推进公开动作
```

## 错误五：没有线上验证

表现是：本地 build 通过后就认为线上没问题，没检查 `/blog`、`/sitemap.xml`、`robots.txt` 和核心工具页面。

危险在于，部署平台、缓存、环境变量和构建输出都可能与本地不同。自动化流程尤其要检查最终公开结果。

解决步骤：

```text
每次推送后至少检查：
1. 首页 200。
2. /blog 200。
3. /tools 200。
4. /sitemap.xml 200。
5. sitemap 不包含 draft。
6. robots.txt 指向正确 sitemap。
7. 公开文章数量符合预期。
```

## 风险提醒

不要承诺收入结果，不要承诺一定成交，不要让自动化替你对客户做最终承诺，不要把未经审核内容公开展示为成熟文章。自动化越强，默认安全状态越重要。

如果某个流程会产生公开页面、客户消息、付款变化、生产配置变化或处理敏感数据，就把它视为高风险流程。先停下来加确认和日志，再考虑自动执行。

## 总结

AI 自动化接单的合规错误，通常来自“默认太开放”。把默认状态改成草稿，把公开动作改成人工确认，把敏感数据排除出普通流程，把线上检查放进固定步骤，风险会下降很多。
