# Claude Code 新手常用命令怎么整理成项目笔记

Claude Code 能帮你读项目、定位文件和建议下一步命令，但新手不能把命令行完全交给工具。接代码小单前，最稳的做法是把常用命令整理成自己的笔记：哪些命令只读、哪些命令会改文件、哪些命令会联网、哪些命令会影响部署。先知道命令风险，再让 Claude Code 帮你排查。

这篇适合搭配 [Claude Code 项目客户问题检查清单](/blog/claude-code-client-questions-checklist)、[Claude Code Bug Prompt 检查清单](/blog/claude-code-bug-prompt-checklist)、[/tools/proposal-generator](/tools/proposal-generator) 和 [/templates](/templates) 使用。

## 适合谁

适合刚开始用 Claude Code 处理前端、Node.js、小型脚本或部署问题的新手。你可能会看到客户项目里有很多命令，但不知道哪个先跑、哪个不能碰、哪个输出适合发给客户。

也适合准备做低风险诊断单的人。命令笔记不是炫技，它的作用是让你能解释自己检查了什么、没有改什么、下一步为什么这么做。

## 不适合谁

不适合没有授权就进入客户后台、处理生产数据、保存密钥或操作付款配置的任务。命令行再熟，也不能替代授权和风险确认。

也不适合把 Claude Code 给出的命令不看就执行的人。工具可能根据上下文提出合理建议，但最终执行动作的人是你，尤其是安装、删除、迁移、部署类命令。

## 具体步骤

第一，先列只读命令。比如查看目录、查看 package.json、查看 git 状态、搜索文件、读取日志。这类命令适合用来建立项目上下文。

第二，列本地验证命令。比如安装依赖、运行 lint、运行测试、启动本地开发服务器、执行构建。这些命令可能耗时，也可能产生缓存或构建目录，要先看项目说明。

第三，列会修改文件的命令。比如格式化、代码生成、安装新依赖、更新 lockfile。执行前要确认是否需要提交这些变化。

第四，列高风险命令。比如删除文件、重置分支、数据库迁移、部署、修改环境变量。新手项目练习时不要把这类命令当成普通排查步骤。

第五，给每条命令写用途、风险和输出解读。比如 `npm run build` 的用途是检查生产构建，失败时要记录错误原文；不是看到失败就马上改一堆文件。

第六，把运行结果写进交付说明。客户不需要看完整终端历史，但需要知道你运行了哪些关键检查、结果是什么、还剩哪些未覆盖范围。

## 新手命令笔记模板

```text
命令：
用途：
运行环境：
是否会修改文件：
是否需要客户授权：
关键输出：
下一步判断：
```

这份模板可以让 Claude Code 的建议更容易被人工复核。你可以先让工具解释命令，再决定是否执行，而不是把命令当成魔法按钮。

## 常见命令分类

`git status`、`git diff`、`rg`、`Get-Content` 这类命令偏向读取信息，适合排查初期使用。它们能帮助你确认当前改动、搜索错误关键词、读取配置文件。

`npm install`、`npm run lint`、`npm test`、`npm run build` 偏向本地验证。它们可能暴露依赖、类型、构建或测试问题，适合在测试环境里运行。

部署、数据库、环境变量和账号权限相关命令要单独标记。即使客户说“很简单”，也要先确认授权、备份和验收方式。

## 执行前怎么问 Claude Code

看到不熟悉的命令时，可以先让 Claude Code 用普通话解释三件事：这条命令读取什么、会不会修改文件、失败时应该看哪一段输出。解释不清楚之前，不要急着执行。

如果工具建议连续运行多条命令，先拆开。一次只运行一条关键命令，记录结果，再决定下一条。这样出错时更容易定位，也不会把多个副作用混在一起。

## 交付说明怎么写

交付时不要只写“命令都跑过了”。更好的写法是：“已在本地运行 lint 和 build；输出通过；未运行部署命令；未接触真实数据；业务流程仍需客户在测试环境确认。”这能让客户明白你的工作边界。

如果命令失败，也可以交付诊断结果。比如依赖缺失、Node 版本不匹配、环境变量未配置、脚本不存在。失败输出本身不是坏事，关键是你要能解释它意味着什么，以及下一步需要谁提供材料。

## 风险提醒

不要在客户生产项目里盲跑会修改状态的命令。先问清环境，再确认是否有测试分支、测试数据和回滚方式。

不要把包含密钥、真实用户资料或私有链接的终端输出直接贴给工具或客户。能脱敏就脱敏，能摘要就摘要。
