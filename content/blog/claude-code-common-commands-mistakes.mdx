# Claude Code 新手运行常用命令时的常见错误

Claude Code 给出命令建议后，新手最容易犯的错误是马上复制执行。命令行看起来只是几行文字，但它可能修改文件、联网安装依赖、触发构建、影响部署，甚至改变客户环境。做项目时，快不是目标，清楚命令风险才是目标。

这篇适合搭配 [Claude Code 常用命令检查清单](/blog/claude-code-common-commands-checklist) 和 [Claude Code 项目客户问题检查清单](/blog/claude-code-client-questions-checklist) 使用。

## 适合谁

适合已经会跟着 Claude Code 跑命令，但经常不知道输出意味着什么的新手。你可能看到失败就继续试，看到成功就以为完成，最后却无法向客户解释。

也适合准备接 bug 排查和部署诊断单的人。命令错误很容易把小问题扩大，所以先建立习惯很重要。

## 不适合谁

不适合生产环境高风险任务。涉及数据库迁移、环境变量、后台配置、真实用户数据或付款设置时，普通新手命令笔记不足以覆盖风险。

也不适合客户没有提供测试环境和授权的任务。没有边界时，不要把命令执行当成探索游戏。

## 具体步骤

错误一：不看命令用途。修正方式是先让 Claude Code 解释这条命令读取什么、修改什么、可能失败在哪里。

错误二：不区分环境。客户项目的本地、测试、预览和生产不能混着说。你在哪个环境运行命令，就在笔记里写清楚。

错误三：不记录输出。运行 `npm run build`、`npm test` 或 lint 后，只说“跑过了”不够，要记录是否通过、失败原文和下一步判断。

错误四：把构建通过当成业务验收。构建通过只能说明代码可以生产构建，不代表客户页面流程、表单提交或接口调用都正常。

错误五：看到失败就连续乱试。连续试命令会制造更多变量。更稳的方法是先读错误、搜索相关文件，再决定下一步。

错误六：忽略命令副作用。安装依赖可能改 lockfile，格式化可能改很多文件，生成命令可能覆盖代码。执行前要知道后果。

## 修正模板

```text
命令：
我为什么运行它：
运行位置：
是否修改文件：
结果：
错误原文：
下一步：
```

把这几项写完整，Claude Code 后续帮你分析时也更容易聚焦。否则工具看到的是零散输出，很难判断你已经做过什么。

## 交付前怎么说明

交付说明不需要贴完整终端日志，但要说明关键检查。例如：“已运行 lint 和 build，均通过；未处理生产部署；未接触真实数据；业务流程仍需客户在测试环境验收。”这种写法比一句“已修好”更清楚。

如果命令失败但你完成了诊断，也可以交付失败原因和下一步材料。比如依赖版本冲突、缺少环境变量、客户未提供测试账号。这不是失败，而是诊断单的有效产出。

## 怎么训练自己的判断

每次运行命令后，给自己补一句解释：我为什么运行它？它证明了什么？它不能证明什么？比如 build 通过，能证明项目可以构建，但不能证明客户表单流程一定正常。lint 通过，能证明代码规则没有报错，但不能证明业务逻辑正确。

这个训练会让你更少依赖“感觉”。Claude Code 可以帮助你总结输出，但你要负责判断证据强弱。项目交付时，强证据写进结果，弱证据写成待验证，不要混在一起。

## 下次避免乱试

如果一个错误连续试了三条命令还没有方向，就停下来回到信息收集：读错误原文、查配置、确认环境、问客户最近变更。继续乱试只会让工作区变复杂。

也可以把失败命令整理成一段 prompt，让 Claude Code 帮你判断最小下一步。prompt 里只放脱敏错误、项目类型、已经运行过的命令和当前限制，不放客户敏感材料。

## 风险提醒

不要运行自己解释不清的高风险命令。尤其是删除、重置、迁移、部署、修改环境变量和清理数据相关命令。

不要把含敏感信息的输出发给工具或客户。先脱敏，再保留必要错误信息。
