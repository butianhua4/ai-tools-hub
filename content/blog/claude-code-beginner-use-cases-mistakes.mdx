# Claude Code 新手选使用场景时的常见错误

Claude Code 让代码任务看起来更容易，但新手真正要学的是“选场景”。选对场景，工具能帮你更快完成小范围交付；选错场景，工具会把你带进看不懂、测不了、解释不清的项目里。

下面整理常见错误和修正方法。建议配合 [Claude Code 使用场景检查清单](/blog/claude-code-beginner-use-cases-checklist) 与 [第一次 AI 接单检查清单](/blog/before-first-ai-freelance-job-checklist) 使用。

## 适合谁

适合已经开始尝试用 Claude Code 做练习或小单，但经常不知道哪些需求能接的人。你可能发现工具能给出很多方案，却不知道哪个方案适合新手落地。

也适合接到客户需求后先做风险判断的人。场景选择做对了，后续报价、测试和交付都会更清楚。

## 不适合谁

不适合想用工具挑战高风险任务的人。生产事故、付款系统、隐私数据、安全漏洞、客户核心业务流程，不应该作为新手练习场。

也不适合没有记录习惯的人。Claude Code 参与了哪些步骤、你人工确认了哪些内容、测试结果是什么，都需要留下记录。

## 具体步骤

错误一：把“工具能做”当成“我能接”。修正方法是增加人工判断：我是否能解释代码、跑检查、写交付说明、承担本次范围内的结果。

错误二：选择无法复现的任务。修正方法是先要求客户提供触发路径、截图、日志和预期结果。无法复现时，只做诊断，不承诺修复。

错误三：忽略权限风险。修正方法是把账号、密钥、生产数据、付款和管理员权限列为高风险信号。遇到这些内容，先暂停确认。

错误四：接范围太大的任务。修正方法是拆成一个页面、一个报错、一个配置、一份文档。小范围更适合新手，也更容易验收。

错误五：没有测试证据。修正方法是每次交付前至少保留一种证据：命令输出、截图、构建结果、复现步骤或客户验收路径。

## 风险提醒

不要用 Claude Code 编造自己没有的经验。可以诚实写“我可以先做诊断并给出排查记录”，不要写自己能处理完整系统。

不要把客户项目当作工具演示。客户要的是可靠结果，不是你试用工具的过程。工具过程可以留在内部记录里，交付给客户的是证据和说明。

## 修正模板

```text
这个任务是否适合新手：
- 是否可复现：
- 是否低权限：
- 是否小范围：
- 是否可测试：
- 是否能写交付说明：

结论：
- 直接接小范围
- 先做诊断
- 暂不承接
```

这个模板能让你在报价前先做判断，而不是被客户一句“很简单”带走。

## 常见客户说法怎么处理

客户说“这个应该很简单”时，你可以回复：“我先确认复现路径和验收标准，确认是小范围问题后再报价。”客户说“你直接看一下后台”时，要先问是否能提供测试环境或脱敏截图。客户说“用工具自动处理就行”时，要说明仍然需要人工检查和测试证据。

这些回复不是拒绝客户，而是把任务拉回可控范围。新手最怕因为想显得专业而什么都答应，最后发现自己无法验证结果。

## 错误复盘

如果你已经选错过场景，可以复盘三件事：当时哪些风险信号被忽略？哪些信息本该先问？下次应该把任务降级成诊断还是直接不接？复盘越具体，下次越不容易重复。

工具不会替你积累判断力。判断力来自一次次记录：哪些任务做得顺，哪些任务卡住，哪些任务从一开始就不该接。

## 下次怎么避免

下次接到需求时，先不要打开工具，先写一行判断：“这个任务是否能在低权限环境里验证？”如果答案是否定的，就先问客户要测试环境、脱敏日志或更明确的验收标准。

再写第二行：“我是否能解释修改结果？”如果你只能说“工具会处理”，说明现在还不适合直接接。等你能解释文件、改动和测试，再进入执行阶段。
