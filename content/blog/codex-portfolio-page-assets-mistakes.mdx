# Codex 作品集素材常见错误和修复顺序

新手准备作品集时，最常见的问题不是没有素材，而是素材没有边界：练习项目写得像客户交付，AI 生成图写得像真实截图，客户项目没有授权就展示，案例只写结果不写验证。Codex 可以帮助你整理表达，但它不会自动知道哪些内容能公开、哪些证据足够、哪些说法太满。

如果你需要从头建立流程，先读 [用 Codex 准备作品集页面素材](/blog/codex-portfolio-page-assets)。如果你想逐项核对，可以用 [作品集页面素材检查清单](/blog/codex-portfolio-page-assets-checklist)。

## 适合谁

适合已经收集了一些项目截图、练习记录、交付笔记或客户反馈，但还没整理成可信案例的人。你可能担心页面不够专业，也可能担心自己写得太像广告。

也适合准备把作品集链接发给潜在客户的人。发布前修正这些错误，比上线后再解释要轻松得多。

## 不适合谁

不适合想用虚构案例快速填满页面的人。练习项目可以展示，但必须标清来源。作品集的目标是建立信任，不是制造履历噪音。

也不适合在未确认授权时展示客户项目。即使客户没有明确反对，也不等于可以公开截图、数据或业务信息。

## 风险提醒

作品集素材一旦公开，就可能被截图、转发、搜索或平台审核。敏感信息泄露、版权不清、客户名称误用、承诺过度，都会给后续项目带来不必要的风险。

另外，过度包装会让沟通变难。客户看到你写“完整负责某系统”，后面可能按完整系统能力来要求报价和交付；如果你实际只做了 README 或样式调整，就会产生预期落差。

## 具体步骤

### 第一步：暂停发布

如果你发现案例里有客户截图、真实数据、无法验证的结果或夸张描述，先保持草稿。不要为了页面完整度急着上线。作品集宁可少，也要清楚。

### 第二步：按风险分组

把素材分成四类：可公开、需要匿名化、只能私下展示、暂不展示。可公开素材要有授权和证据；需要匿名化的要打码、改名或裁剪；只能私下展示的要限制发送场景；暂不展示的先放回素材库。

### 第三步：补证据

每个案例至少要补三类证据：原始问题、你的动作、验证结果。原始问题可以是截图或文字描述；动作可以是 diff、任务记录或步骤；验证结果可以是运行命令、截图、客户确认或自测记录。

如果验证时遇到报错，可以先用 [报错解释器](/tools/error-explainer)整理，不要把失败结果藏起来。

### 第四步：改写表达

让 Codex 帮你把案例改成“事实陈述”。例如把“完整打造高转化网站”改成“整理首页结构、修复移动端按钮遮挡，并记录桌面和移动端截图”。越具体，越可信。

### 第五步：重新设计 CTA

CTA 不要承诺超出能力范围。新手更适合写“可以先做小范围诊断”“可以整理需求和报价范围”“可以审核已有 AI 输出”。准备客户回复时，可以用 [Proposal 生成器](/tools/proposal-generator)起草，再人工删掉夸张承诺。

## 常见错误清单

第一类错误是案例来源不清。读者不知道这是客户项目、练习项目还是模板复刻。第二类错误是截图未处理，里面出现客户信息、订单号、后台路径或真实数据。第三类错误是没有前后对比，只放一个漂亮结果图。

第四类错误是结果描述太满。比如没有数据依据，却写“显著提升转化”；没有长期运行，却写“稳定可用”。第五类错误是没有下一步 CTA，读者看完不知道如何联系或判断服务范围。

## 修复顺序示例

假设你有一个 AI 生成的作品集页面，里面用了虚构客户名和漂亮截图。第一步不是继续美化页面，而是把虚构案例改成“练习项目”，删掉客户名，补上项目目标、使用工具和学习结果。第二步再检查截图是否包含敏感信息。第三步才考虑排版和文案。

假设你有真实客户项目截图，但没问过能否展示。先不要上线。可以把它放入“需要授权”列表，明日待办写成“向客户确认是否可匿名展示截图和项目描述”。确认前，只用文字概述或模拟数据。

## CTA：下一步

打开你的作品集草稿，找出 3 句最像夸张宣传的话，把它们改成可验证事实。然后回到 [检查清单](/blog/codex-portfolio-page-assets-checklist)，确认每个案例状态。
