# Claude Code 错误排查哪些适合写进项目范围

客户报错排查很适合拆成小单，但范围要写清楚。新手可以提供低权限诊断、错误信息整理、本地验证和修复建议；不应该默认包含生产后台、数据库迁移、真实数据处理、付款系统和长期维护。

这篇适合搭配 [Claude Code 错误排查检查清单](/blog/claude-code-error-debug-checklist)、[Claude Code 常用命令检查清单](/blog/claude-code-common-commands-checklist) 和 [/templates](/templates) 使用。

## 适合谁

适合想把“帮我看一下报错”拆成诊断服务的新手。你可以先判断错误是否可复现、是否低权限、是否能本地验证，再决定是否进入修复。

也适合需要写报价边界的人。客户报错通常很模糊，越早写清范围，后面越少误解。

## 不适合谁

不适合客户要求你直接进入生产环境、处理真实数据或修改关键配置的任务。没有正式授权和回滚方案时，不要把它当作普通排查。

也不适合客户拒绝提供错误原文和环境信息，却要求固定结果的任务。信息不足时，只能先做有限诊断。

## 具体步骤

第一，适合写进基础范围的是错误整理。包括收集脱敏错误原文、复现步骤、环境信息、最近变更和客户预期结果。

第二，适合写进诊断范围的是只读排查。包括查看配置、搜索错误关键词、读取项目脚本、检查依赖说明和整理可能原因。

第三，适合写进本地验证范围的是 lint、test、build 或本地启动。前提是项目允许、依赖可安装、没有触碰生产数据。

第四，可以谨慎写进修复范围的是一个明确、低风险、可测试的问题。例如修一个配置缺失、一个页面报错、一个脚本参数错误。

第五，不要默认包含生产部署、数据库迁移、环境变量修改、账号权限处理、长期维护和新增功能。这些需要单独确认。

## 报价范围示例

```text
本次包含：
- 整理脱敏错误信息
- 判断复现路径和运行环境
- 运行低风险本地检查
- 输出可能原因和下一步建议
- 如确认低风险，处理一个明确问题

不包含：
- 生产部署
- 数据库操作
- 真实数据处理
- 付款或账号权限
- 长期维护
```

这段能把“报错排查”从模糊请求变成可交付服务。

## 什么时候只做诊断

如果客户没有复现路径、没有测试环境、没有错误原文，或者错误可能来自外部服务和账号状态，就先做诊断。诊断交付包括已确认事实、可能原因、需要补充材料和是否适合继续修复。

如果诊断后发现需要生产权限或客户内部配置，也不要顺手接下。把下一步写清楚，让客户决定是否提供正式授权或换成内部人员处理。

## 客户确认话术

```text
我可以先做一轮低权限诊断，整理错误原因和下一步建议。本次不进入生产后台、不处理真实数据、不修改数据库或付款相关配置。若诊断后适合继续修复，我们再单独确认范围。
```

这句话适合发在报价前，让客户知道你提供的是有边界的诊断。

## 怎么拆成两段交付

第一段是诊断交付：整理错误、确认环境、运行低风险检查、输出可能原因和下一步。它适合信息不完整但客户愿意配合的场景。

第二段是修复交付：只处理一个明确、可测试、低权限的问题，并提供修改摘要和验证结果。不要把第二段默认包含在第一段里。

## 追加范围怎么处理

如果诊断后发现需要部署、数据库、环境变量或账号权限，就暂停原范围，发起新的确认。追加范围要说明需要什么授权、谁负责备份、如何验收、失败后如何回退。

客户可能只是想快点解决问题，但你仍然要把技术风险说清楚。边界写得越早，后面越不容易变成无限支持。

## 风险提醒

不要把错误排查和完整系统维护混成一个服务。小单适合处理明确问题，不适合兜底整个项目。

不要为了让报价看起来完整，就把高风险权限写进基础范围。边界越模糊，后续越难收口。
