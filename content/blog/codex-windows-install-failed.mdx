# Windows 上 Codex 安装失败怎么排查

Windows 上安装 Codex 或其他 AI 编程工具时，新手常见卡点不是工具本身，而是 PowerShell、路径、Node、Git、权限、网络和安全策略。错误提示可能很长，也可能只有一句“command not found”“access denied”“cannot find path”。这时候不要连续复制命令，先把环境状态记录清楚。

这篇文章给一套保守排查流程。你可以先读 [安装 Codex 前要准备什么](/blog/codex-install-prerequisites)，再用本文处理 Windows 特有问题。若只想要短表版，可以看 [Windows 安装失败检查清单](/blog/codex-windows-install-failed-checklist)。

## 适合谁

适合使用 Windows 电脑、PowerShell、VS Code 终端的新手。你可能已经安装过 Node 或 Git，但命令输出不稳定；也可能项目在教程里能跑，到自己电脑就失败。

也适合准备做 AI 工具实践基础环境练习的人。客户项目出问题时，你要先分清是电脑环境问题、项目依赖问题，还是客户授权问题。Windows 排查记录会帮你建立这个判断力。

## 不适合谁

不适合在公司受管电脑、客户电脑或服务器上随意改系统策略的人。管理员权限、执行策略、代理证书和安全软件都可能由组织管理，新手不应擅自绕开。

也不适合把安装成功当成项目判断能力的人。安装只是入口，后面还要会保存日志、读 README、跑 build、检查 diff 和写交付记录。

## 风险提醒

Windows 排错常见风险是乱改 PATH、重复安装 Node、用管理员权限重跑所有命令、删除系统目录里的文件、把客户 token 写进命令行历史。任何会影响整台电脑或客户账号的动作，都要先停下来确认。

如果安装涉及登录、购买、授权、私有仓库、客户 VPN、公司代理或真实密钥，先记录为客户确认或明日待办，不要用自己的账号替客户处理。

## 具体步骤

1. 保存原始报错。复制完整命令、PowerShell 输出、当前路径、系统版本和截图。不要只记最后一行。

2. 确认终端。打开 PowerShell，运行 `pwd`、`Get-Location` 或 `dir`，确认自己在哪个目录。很多安装失败只是因为命令跑错位置。

3. 检查 Node 和 npm。运行 `node -v`、`npm -v`。如果命令不存在，说明基础工具未就绪；如果 VS Code 和系统 PowerShell 版本不同，先记录路径差异。

4. 检查 Git。运行 `git --version` 和 `git status`。如果项目不是 Git 仓库，也要知道自己正在改哪些文件。

5. 检查权限。看到 access denied、execution policy、administrator、permission 等关键词时，不要立刻强行管理员运行。先判断这是个人电脑还是受管设备。

6. 检查网络。安装依赖或下载工具失败时，记录 registry、timeout、certificate、proxy 等关键词。公司网络和代理问题要找客户或管理员确认。

7. 低风险复现。每次只改一个变量，改完就记录结果。遇到日志看不懂，用 [报错解释器](/tools/error-explainer)拆原因和下一步。

## PowerShell 里最该保存什么

建议保存四类输出：当前目录、版本命令、失败命令、错误文本。比如：

```text
pwd
node -v
npm -v
git --version
```

这些不是形式主义。没有它们，别人很难判断你的电脑到底缺什么。做项目时，这些输出也可以变成诊断报告的一部分。

## 项目前怎么处理

如果客户让你处理 Windows 安装失败，先把任务定义为“环境诊断”，不要直接承诺完整修复。你可以交付当前状态、错误类型、建议步骤和客户待确认事项。报价时用 [报价计算器](/tools/pricing-calculator)估算诊断、沟通和复核时间。

需要写客户回复时，可以用 [Proposal 生成器](/tools/proposal-generator)起草确认问题，再人工删改。需要表格时，从 [模板库](/templates) 拿环境检查模板。

## 判断结果怎么分级

排查结束后，不要只写“已处理”或“仍失败”。更清晰的结论可以分成四级。第一级是基础环境缺口，比如 Node、npm、Git 缺失或版本明显不符合项目要求，这类问题需要先补基础工具。第二级是路径和终端问题，比如在错误目录执行命令、VS Code 终端和系统 PowerShell 读取到不同环境，这类问题适合通过固定目录和固定终端复查。

第三级是权限和网络问题，比如执行策略、公司设备限制、代理、证书、下载超时。这类问题要写明需要谁确认，不能把所有责任都归到工具本身。第四级是项目或账号边界问题，比如需要访问私有仓库、登录服务、读取客户配置或使用真实密钥。这类事项应进入待确认清单，确认前只做记录和建议。

这种分级能帮助你把一次混乱的安装失败变成可交付的诊断记录。客户也能看懂：哪些是你能继续排查的，哪些必须由客户或管理员提供信息，哪些暂时不适合继续动。

## CTA：下一步

打开 PowerShell，记录当前目录、Node、npm、Git 四项输出。再把安装失败日志按“命令、错误、已尝试、下一步”整理出来，不要急着执行更多命令。
