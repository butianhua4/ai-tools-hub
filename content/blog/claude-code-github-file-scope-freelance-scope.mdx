# Claude Code 看 GitHub 文件哪些适合写进接单范围

GitHub 文件检查可以做成低风险小单，但范围要非常清楚。适合写进范围的是指定文件、指定分支、只读诊断、必要相邻上下文和简短交付说明。不适合默认包含的是全仓库审计、推送提交、生产配置、数据库、真实数据和长期维护。

这篇适合搭配 [Claude Code GitHub 文件范围检查清单](/blog/claude-code-github-file-scope-checklist)、[Claude Code 常用命令检查清单](/blog/claude-code-common-commands-checklist) 和 [/templates](/templates) 使用。

## 适合谁

适合想接“帮我看这个 GitHub 文件”这类小单的新手。你可以用 Claude Code 辅助理解代码、定位相邻上下文、整理诊断建议，但不应该默认接管整个仓库。

也适合准备把服务拆成“只读诊断”和“小范围修改”两段的人。先诊断，后修改，范围更稳。

## 不适合谁

不适合客户没有明确授权，却要求查看私有仓库、配置文件、密钥、真实数据或生产部署信息的任务。

也不适合客户要求“顺便把整个项目都看看”。全仓库审计和长期维护不是单文件小单。

## 具体步骤

第一，基础范围可以写“查看一个指定文件”。文件路径、分支、commit 或 PR 要明确。

第二，诊断范围可以写“查看必要相邻上下文”。例如调用方、同目录样式、相关配置片段，但必须说明不是全仓库审计。

第三，修改范围可以写“处理一个明确问题”。比如修一个文案、一个类型错误、一个简单逻辑分支。复杂重构要另算。

第四，验证范围可以写“根据客户提供条件做本地或只读验证”。没有依赖、账号或环境变量时，要说明未验证。

第五，交付范围可以包含文件摘要、问题判断、建议下一步、未覆盖范围和客户验收动作。

## 报价范围示例

```text
本次包含：
- 查看指定 GitHub 文件和必要相邻上下文
- 判断一个明确问题
- 输出诊断结论或小范围修改建议
- 写清未覆盖范围和下一步

不包含：
- 全仓库审计
- 生产部署
- 数据库或真实数据
- 推送代码
- 长期维护
```

这段可以直接放进报价，避免客户把单文件检查理解成全项目服务。

## 什么时候只做诊断

当客户只给文件但不给复现步骤、错误原文、测试环境或验收方式时，先做诊断。诊断交付可以包括代码阅读结论、可能原因、需要补充的文件和是否适合继续修复。

如果诊断发现需要看私有配置、接口数据或生产日志，就暂停原范围，让客户确认授权和脱敏方式。

## 客户确认话术

```text
我可以先做指定 GitHub 文件的只读诊断。本次不做全仓库审计、不推送代码、不处理生产配置或真实数据。若需要继续修改，我们再确认具体文件、分支和验收方式。
```

这句话适合报价前发给客户，把服务边界放到明面上。

## 怎么拆成服务档位

基础档可以是只读文件诊断：查看一个文件和必要相邻上下文，输出问题判断和下一步材料。适合客户还没确认是否要修改的情况。

标准档可以是小范围修改：在指定分支或客户提供的文件基础上，处理一个明确问题，并提供修改摘要和验证建议。是否提交 PR 要单独确认。

进阶档才考虑多文件影响分析、较完整的代码说明或协助客户准备 PR 描述。只要超过指定文件，就要重新确认范围。

## 追加范围怎么处理

客户可能在诊断后说“那你顺便把调用它的几个文件也改了”。这时可以继续，但必须重新列出文件、目标和验收方式。

不要把追加范围默默吸收进原单。GitHub 文件检查的价值在于清楚、可追踪、可复核，不在于一次看无限多文件。

## 风险提醒

不要把客户给的 GitHub 链接当成开放许可。访问、下载、修改、提交、部署是不同层级的授权。

不要承诺“看完一个文件就能判断整个项目”。单文件诊断只能覆盖本次指定范围。
