# Windows 上 Codex 安装失败检查清单

Windows 安装失败时，先不要急着换教程。你需要确认 PowerShell、当前目录、Node、npm、Git、权限、网络和日志。下面这份清单适合配合 Codex 使用，但每一项都要人工验证。

完整流程可读 [Windows 上 Codex 安装失败怎么排查](/blog/codex-windows-install-failed)。如果已经越修越乱，可以看 [常见错误和修复顺序](/blog/codex-windows-install-failed-mistakes)。

## 适合谁

适合第一次在 Windows 上配置 AI 编程环境的新手。你可能看到命令不存在、权限不足、路径找不到、npm 下载失败、Git 不可用等问题。

也适合项目前做环境自查的人。客户项目排查前，你至少要知道自己电脑的基础命令是否稳定。

## 不适合谁

不适合公司受管电脑或客户设备上未经确认就修改系统策略的人。设备策略和权限可能属于组织管理范围。

也不适合直接处理客户账号、token、私有仓库和公司网络配置的新手独立承接。

## 风险提醒

清单只能减少遗漏，不能保证安装成功。Windows 环境差异很大，安全软件、代理、路径、管理员权限、终端类型都会影响结果。

遇到账号登录、付费服务、客户授权、私有仓库和真实密钥时，先记录为待确认事项。

## 具体步骤

### 1. 系统和终端

- Windows 版本是否记录。
- 使用的是 PowerShell、CMD 还是 VS Code 终端。
- 终端是否能运行 `pwd` 或 `Get-Location`。
- 是否知道当前目录。
- 是否能复制完整错误输出。

终端不清楚时，先不要继续安装。

### 2. 路径和项目

- 当前目录是否是项目根目录。
- 是否有 package.json。
- 是否有 README。
- 是否有 Git 仓库。
- 是否误在桌面、下载目录或系统目录运行命令。

路径错误是新手高频问题。

### 3. Node 和 npm

- `node -v` 是否输出版本。
- `npm -v` 是否输出版本。
- VS Code 终端和 PowerShell 输出是否一致。
- 项目是否要求特定 Node 版本。
- 是否重复安装过多个版本。

版本混乱时，先记录，不要继续叠加安装。

### 4. Git

- `git --version` 是否可用。
- 是否能查看 `git status`。
- 是否知道当前分支。
- 是否有未提交改动。
- 是否知道不要随意删除客户文件。

Git 状态是排错的安全网。

### 5. 权限

- 是否提示 access denied。
- 是否提示 execution policy。
- 是否需要管理员权限。
- 是否是公司或客户受管设备。
- 是否有安全软件拦截。

权限问题要谨慎，不要强行改策略。

### 6. 网络和下载

- 是否出现 timeout。
- 是否出现 certificate。
- 是否出现 proxy。
- 是否访问包源失败。
- 是否需要公司 VPN 或代理。

网络问题常常需要客户或管理员确认。

## 记录模板

```text
系统：
终端：
当前目录：
Node/npm：
Git：
失败命令：
错误摘要：
已尝试动作：
需要确认：
```

这份记录可以放进 [模板库](/templates)，也可以作为客户诊断报告。

## 结论怎么写

结论可以分成：可继续安装、需补基础工具、需客户确认、暂不处理。可继续安装表示终端、Node、Git、权限和网络基本清楚；需客户确认表示涉及账号、设备策略或私有资源；暂不处理表示风险超出当前能力。

这个结论比“还是不行”更有用。

## 检查时的记录原则

每一项检查都要留下“看到什么”和“因此判断什么”。例如 `node -v` 没有输出，记录的是 Node 命令不可用，而不是直接写 Codex 安装失败。`npm install` 出现 timeout，记录的是下载或网络链路异常，而不是直接重装所有工具。这样写可以避免把不同问题混在一起。

如果你准备把这份清单用于项目，建议再加三列：客户需提供、我方可执行、暂停条件。客户需提供包括日志、系统信息、项目说明和必要授权；我方可执行包括读日志、整理原因、给低风险下一步；暂停条件包括公司设备策略、私有仓库、账号登录、付费服务和真实密钥。清单不是为了显得复杂，而是为了防止新手在边界不清时越做越乱。

最后，把每次尝试控制在一个变量内。只换终端就不要同时重装 Node；只检查网络就不要同时改 PATH。记录越稳定，后续判断越可靠。

## 最低可交付记录

如果这份清单最后要交给客户或留作复盘，最低可交付记录至少包括五句话。第一句写当前环境，比如 Windows 版本、终端类型和项目目录。第二句写基础工具状态，比如 Node、npm、Git 是否能输出版本。第三句写失败命令和原始错误，不要只写概括。第四句写你已经尝试过什么，并说明每一步结果。第五句写下一步建议和待确认事项。

这五句话能让记录从“我试过了”变成“别人能接着判断”。对新手来说，这比一次性解决所有安装问题更现实，也更容易形成稳定服务流程。
