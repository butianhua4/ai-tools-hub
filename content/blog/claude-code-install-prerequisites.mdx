# 安装 Claude Code 前新手要先确认哪些条件

新手安装 Claude Code 时，最常见的问题不是工具本身，而是本机环境没有准备好：终端不会用、Node.js 版本不清楚、Git 没配置、项目权限混乱、网络访问不稳定。安装前先做一轮前置条件检查，能省下很多无效排查时间。

这篇适合搭配 [Claude Code 常用命令检查清单](/blog/claude-code-common-commands-checklist)、[Claude Code 错误排查检查清单](/blog/claude-code-error-debug-checklist)、[Claude Code 交付说明检查清单](/blog/claude-code-delivery-note-checklist) 和 [/templates](/templates) 使用。

## 适合谁

适合准备第一次安装 Claude Code 的新手，尤其是想把它用于学习代码、阅读项目、整理小范围修改建议的人。你不需要一开始就会复杂命令，但要知道自己当前系统和项目状态。

也适合准备给客户做低风险诊断的人。接客户项目之前，先在自己的测试项目里确认工具可用，不要一上来就在客户仓库里试安装和试命令。

## 不适合谁

不适合没有基本电脑权限、无法安装软件、无法使用终端或无法确认网络环境的人。遇到这些情况，要先解决本机基础环境。

也不适合直接拿客户生产项目练手。安装、登录、授权和项目读取都需要边界，不能把客户项目当成测试场。

## 具体步骤

第一，确认系统和终端。记录你使用的是 Windows、macOS 还是 Linux；终端是 PowerShell、Terminal、WSL 还是其他工具。不同环境下命令和路径表现可能不同。

第二，确认 Node.js 和包管理器。运行版本检查，记录 Node、npm、pnpm 或 yarn 的版本。不要在不知道版本的情况下排查安装失败。

第三，确认 Git 是否可用。Claude Code 常常需要读取项目结构和改动状态。至少要能运行 `git status`，并知道当前项目是否有未提交改动。

第四，准备测试项目。先用自己的小项目验证工具，而不是直接打开客户私有仓库。测试项目可以是一个公开教程项目或本地练习项目。

第五，确认账号和网络。需要登录或联网的步骤，要使用自己的账号和合规网络环境。不要让客户把账号、验证码或私密 token 直接发给你。

第六，确认项目权限。只读学习、诊断、修改文件、提交代码、部署，是不同权限层级。安装前就要知道自己要做哪一层。

第七，记录安装过程。每一步命令、错误和结果都简单写下来。以后遇到客户环境问题时，才知道是安装问题、项目问题还是权限问题。

## 安装前记录模板

```text
系统：
终端：
Node 版本：
包管理器：
Git 状态：
测试项目：
账号/网络：
本次目标：
不处理范围：
```

这份记录可以让 Claude Code 的安装问题更容易排查，也能帮助你向客户解释“我先确认本地工具可用，再处理项目问题”。

## 安装失败怎么判断

如果安装失败，先不要急着换很多命令。把错误原文、系统、终端、Node 版本、网络环境和已运行命令写出来，再判断是依赖问题、权限问题、网络问题还是命令输入错误。

如果只是客户项目运行失败，不要把它归咎于 Claude Code 安装。工具能打开，不代表客户项目依赖完整、配置正确、环境变量齐全。

## 验证顺序怎么排

建议先验证本机基础，再验证工具，再验证测试项目，最后才碰客户项目。本机基础包括终端、Node、Git 和网络；工具验证包括能否启动、能否读取文件、能否解释小错误；测试项目验证包括能否在低风险项目里跑通一次阅读和建议。

这个顺序能避免把问题混在一起。如果本机 Node 都不稳定，就不要继续讨论客户项目为什么跑不起来。先把最底层证据补齐。

## 给客户怎么解释

如果客户催你直接处理项目，可以回复：“我会先确认本机工具和测试项目可用，再进入你的项目诊断。这样可以区分工具安装问题和项目本身问题。”这不是拖延，而是减少误判。

安装准备的交付也可以很具体：环境记录、已验证步骤、失败原因、需要客户本人完成的登录或授权动作、下一步是否适合进入项目诊断。

## 风险提醒

不要向客户索要账号密码、验证码、完整密钥或私有 token。涉及授权时，优先让客户自己在自己的环境里操作，或提供最小权限的测试方式。

不要在安装阶段运行删除、重置、迁移、部署等高风险命令。安装准备只证明工具和本机基础环境可用，不代表可以动客户生产系统。
