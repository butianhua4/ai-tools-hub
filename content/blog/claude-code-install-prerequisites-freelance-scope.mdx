# Claude Code 安装准备哪些适合写进接单范围

Claude Code 安装准备可以做成低风险服务，但范围要写得很清楚。适合写进范围的是本机环境确认、安装错误诊断、测试项目验证和简短说明。不适合默认包含的是客户账号处理、私有 token、生产配置、客户真实数据和长期维护。

这篇适合搭配 [Claude Code 安装前置条件检查清单](/blog/claude-code-install-prerequisites-checklist)、[Claude Code 常用命令检查清单](/blog/claude-code-common-commands-checklist) 和 [/templates](/templates) 使用。

## 适合谁

适合想帮客户或同学整理 Claude Code 安装环境的新手。你可以做的是低风险诊断和步骤说明，而不是代管对方账号或生产项目。

也适合准备把自己的安装经验整理成服务包的人。安装准备服务的价值在于把环境问题讲清楚，而不是承诺替客户解决所有代码问题。

## 不适合谁

不适合客户要求你接收账号密码、验证码、完整密钥或私有 token 的任务。这些内容不应通过普通聊天交给你。

也不适合客户要求你顺便处理生产部署、数据库、环境变量、私有仓库长期维护的任务。这些都不是安装准备本身。

## 具体步骤

第一，基础范围可以包含环境确认。记录系统、终端、Node、包管理器、Git、网络和测试项目。

第二，诊断范围可以包含安装错误整理。读取脱敏错误原文，判断可能是权限、网络、版本还是命令问题。

第三，验证范围可以包含测试项目运行。确认工具能读取文件、解释错误、生成小范围建议。

第四，交付范围可以包含一份安装记录。说明已确认项目、仍缺什么、客户下一步应该自己操作什么。

第五，不包含客户账号代管、生产项目操作、私有配置处理、长期远程支持。需要这些时，要另行确认。

## 报价范围示例

```text
本次包含：
- 检查本机系统、终端、Node、Git
- 整理安装错误和可能原因
- 在测试项目里验证基础可用性
- 提供安装记录和下一步建议

不包含：
- 客户账号或验证码处理
- 私有 token 保存
- 生产部署
- 数据库或环境变量修改
- 长期维护
```

这段适合放进报价，也适合发给客户确认边界。

## 什么时候只做诊断

如果客户电脑权限不足、网络不稳定、账号登录无法由本人完成、或者错误来自客户私有环境，就先做诊断。诊断交付可以包括已确认条件、失败原因猜测、需要客户本人完成的步骤。

如果客户要求你远程操作高权限内容，要暂停并重新确认授权方式。新手不应把安装准备做成无限远程支持。

## 客户确认话术

```text
我可以帮你整理 Claude Code 安装前置条件和错误诊断。本次不接收账号密码、验证码、私有 token，也不操作生产项目。需要登录或授权的步骤，请你在自己的环境中完成。
```

这句话可以保护双方边界，也能让服务更清楚。

## 怎么拆成服务档位

基础档可以是环境检查：确认系统、终端、Node、Git、网络和测试项目。客户得到的是一份环境记录和下一步建议。

标准档可以是安装诊断：基于脱敏错误原文判断失败原因，给出一到两条低风险修复路径。仍然不包含客户账号和生产项目。

进阶档可以是测试项目验证和说明文档：确认工具能在低风险项目里读取文件、解释错误、生成建议，再把结果写成客户能理解的说明。

## 追加范围怎么处理

如果客户安装成功后要求你继续修他的项目，这已经是新范围。需要重新确认项目路径、错误原文、权限边界和验收方式。

不要把“安装准备”自然延长成“所有项目问题支持”。两者价值不同，风险也不同。

## 风险提醒

不要把安装服务写成“包解决所有工具和项目问题”。安装成功、项目运行成功、客户业务流程正常，是不同层级。

不要为了完成订单而接收敏感账号材料。遇到授权步骤，优先让客户自己完成。
