# Vercel 部署常见错误和修复顺序

Vercel 部署问题最怕乱试。新手看到 build 失败，可能立刻升级依赖、删除 lockfile、改环境变量、换构建命令。结果原始错误没解决，项目状态反而更难解释。更稳的做法是先保存日志、分类错误、本地复现，再决定是否修改。

这篇文章整理常见错误和修复顺序。完整流程见 [用 Codex 检查 Vercel 部署问题](/blog/codex-vercel-deployment)，短表版见 [Vercel 部署检查清单](/blog/codex-vercel-deployment-checklist)。

## 适合谁

适合部署 Next.js、小网站、作品集页时遇到 build 失败、页面 404、环境变量缺失、线上和本地不一致的新手。

也适合接部署排查小单的人。客户发来“Vercel deploy failed”，你需要先判断是否能做诊断，而不是马上承诺修好。

## 不适合谁

不适合处理生产数据库、支付、登录权限、真实用户数据和重大线上事故的新手独立承接。

也不适合没有客户授权时改项目设置、域名、环境变量或团队权限。

## 风险提醒

部署修复常见风险是扩大影响。一个环境变量改错，可能让预览和生产都异常；一个构建命令改错，可能让所有分支失败；一个域名配置失误，可能影响客户线上访问。

所以每次改动要记录原因、范围和验证结果。涉及客户账号和平台权限时，先确认。

## 具体步骤

### 第一步：保存日志

保存 Vercel 部署链接、失败 commit、分支、build 日志和错误截图。不要只发一句“部署失败”。日志是后续判断的依据。

### 第二步：本地复现

运行本地 lint、build 和项目 README 里的命令。如果本地失败，先修本地；如果本地通过但 Vercel 失败，重点排查环境差异。

### 第三步：分类错误

常见类型包括依赖安装失败、TypeScript 错误、lint 错误、静态生成失败、环境变量缺失、文件路径大小写问题、路由问题。先分类，再修复。

### 第四步：检查环境变量

如果错误看起来像 undefined、missing env、invalid API key，不要把真实变量贴到聊天里。先确认变量名、作用环境和客户配置方式。

### 第五步：小步修复

每次只改一个问题。修 TypeScript 就不要同时改部署配置；补环境变量说明就不要升级依赖。修完后重新部署或本地 build，记录结果。

### 第六步：线上复查

部署成功后，打开关键页面、移动端、sitemap 和 robots。不要只看 Vercel 显示绿色。

## 常见错误清单

第一类是本地没 build。第二类是 Vercel 环境变量缺失。第三类是混用包管理器。第四类是路径大小写在本地不报错，云端报错。第五类是动态路由静态生成失败。

第六类是部署后不复查。第七类是把预览链接当正式域名。第八类是没有记录客户待确认事项。

## 修复顺序示例

如果 build 失败在 TypeScript，先读具体文件和行号，再决定是否修类型。不要直接关闭检查。若错误来自第三方包类型，也要判断是否是版本问题。

如果部署成功但页面 404，先确认路由、文件名大小写、动态参数和部署分支。不要马上改域名。

## 交付记录怎么写

交付记录应包含：原始错误、已检查命令、修改摘要、部署链接、已验证页面、客户待确认事项和未覆盖风险。可以放入 [模板库](/templates)，客户说明可用 [Proposal 生成器](/tools/proposal-generator)起草后人工复核。

报价时可用 [报价计算器](/tools/pricing-calculator)估算诊断、修复、部署和复查时间。

## 如何避免部署排查失控

先把任务拆成诊断、修复、部署、复查四段。诊断只解释日志和缺口；修复才改代码或配置；部署负责生成预览或生产链接；复查负责打开页面和记录结果。四段混在一起时，客户会觉得“上线失败都是你没做好”，你也很难报价。

如果 Codex 建议一次改很多内容，可以要求它缩小范围：“只解释 build 失败原因，不修改代码；只列低风险验证命令；哪些事项需要客户确认。”这种约束能让你保持可控。

## 客户沟通示例

可以这样写：“我会先做部署诊断，检查 build 日志、本地 build、环境变量和关键页面。若问题涉及域名、账号权限、真实密钥或生产配置，会单独列为客户确认项。诊断完成后再决定是否进入修复和上线。”

这段话把责任边界说清楚，适合新手接小范围部署排查。
