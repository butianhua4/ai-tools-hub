# Claude Code 低风险小单哪些适合写进项目范围

低风险小单可以作为新手的起步服务，但范围必须明确。适合写进范围的是页面文案、单报错诊断、单文件检查、README 整理、交付说明。不适合默认包含的是生产部署、真实数据、账号权限、付款流程、数据库和长期维护。

这篇适合搭配 [Claude Code 低风险项目检查清单](/blog/claude-code-low-risk-freelance-jobs-checklist)、[Claude Code 小项目任务检查清单](/blog/claude-code-small-project-tasks-checklist) 和 [/templates](/templates) 使用。

## 适合谁

适合想从低风险服务开始练习的 Claude Code 新手。你可以提供诊断、整理、小范围修改和交付说明，但要避免把低风险服务写成全能服务。

也适合准备写报价范围的人。报价范围写得越清楚，客户越不容易追加到失控。

## 不适合谁

不适合客户要求你处理生产事故、数据库、真实用户数据、付款、账号权限、安全事件、全站重构或长期维护的任务。

也不适合客户拒绝提供复现和验收方式，却要求固定结果的任务。

## 具体步骤

第一，可以写进基础范围的是页面文案和局部样式。必须限制在一个明确页面或组件。

第二，可以写进基础范围的是单报错诊断。需要脱敏错误原文、环境和复现步骤。

第三，可以写进基础范围的是单文件检查。比如一个 GitHub 文件、一个配置片段、一个 README 段落。

第四，可以写进基础范围的是交付说明。把修改结果、验证方式和未覆盖范围写清楚。

第五，不要默认包含部署、数据库、账号、付款、真实数据、长期维护。只要出现这些，就提高风险等级。

## 报价范围示例

```text
本次包含：
- 一个明确低风险问题
- 低权限诊断或小范围修改
- 基础验证结果
- 交付说明和验收步骤

不包含：
- 生产部署
- 真实数据
- 账号或付款权限
- 数据库操作
- 长期维护
```

这段可以直接放进报价或客户确认消息。

## 服务档位

基础档：低风险诊断。交付错误整理、可能原因、下一步建议。

标准档：一个明确小修。交付修改摘要、验证结果、未覆盖范围。

进阶档：多文件影响检查或多轮验收支持。只要进入进阶档，就要重新确认价格和边界。

## 客户确认话术

```text
我可以先处理一个低风险小范围任务。本次不包含生产部署、真实数据、账号权限、付款流程或长期维护。若发现需要这些内容，我们再重新确认范围。
```

这句话适合报价前使用，能把低风险范围固定下来。

## 交付物怎么写

交付物至少包括四项：处理范围、验证结果、未覆盖范围、客户验收步骤。如果是诊断单，还要写需要客户补充的材料。

不要只写“完成”。低风险服务也需要证据和边界，否则客户会把它当成无限支持。

## 追加服务怎么报价

如果客户想要第二个页面、第二个报错、多轮答疑或部署协助，可以作为追加服务。追加服务要重新写目标、权限、验证方式和交付物。

不要用“顺便”处理追加服务。顺便越多，原本低风险的小单越容易变成没有边界的支持。

## 低风险服务说明

```text
低风险服务只覆盖一个明确目标，交付诊断或小范围修改说明。任何涉及生产、账号、真实数据、付款、数据库或长期维护的内容，都需要重新确认范围。
```

这段可以放在报价底部，简单但很有用。

## 验收范围怎么写

低风险服务的验收应当很具体。页面类任务验收页面和截图，报错类任务验收错误是否复现，文档类任务验收指定步骤是否覆盖。不要让客户用“整体满意”来验收一个小单。

如果客户需要多轮验收或多人评审，那已经超过基础低风险服务。可以做，但要作为追加范围。

## 复盘字段

```text
是否低风险：
是否一次验收：
是否有追加需求：
下次报价要补充：
```

这些字段能帮助你把低风险服务越做越稳定。

## 风险提醒

不要把低风险服务描述成“帮你解决所有技术问题”。这会让后续期望失控。

不要因为客户急就跳过授权、脱敏和验收确认。越急越要把范围写清楚。
