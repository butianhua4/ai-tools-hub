# Tailwind 移动端样式检查项目能不能接

Tailwind 移动端样式检查很适合做成小范围项目服务：客户页面已经有了，但手机上不好看，需要你检查溢出、导航、按钮、表单和布局。它比从零开发轻，但也有边界。你要先确认页面数量、设备要求、是否涉及共享组件、客户素材是否最终版，以及交付物是问题清单还是实际修复。

如果你还没有检查流程，先看 [用 Codex 检查 Tailwind 移动端样式](/blog/codex-tailwind-mobile-check)。如果要逐项核对，可以用 [Tailwind 移动端样式检查清单](/blog/codex-tailwind-mobile-check-checklist)。

## 适合谁

适合已经能读 Tailwind class、看响应式断点、使用浏览器设备模拟、保存截图的新手。你不一定要重构整个页面，但要能定位问题、说明影响，并给出保守修复建议。

也适合做小网站交付检查的人。移动端样式检查可以作为交付前复核的一部分，也可以单独作为诊断小单。

## 不适合谁

不适合客户要求你重新设计整站、重做组件库、保证所有设备完全一致，却只按小修复付费的任务。

也不适合涉及复杂交互、图表、地图、后台系统或真实交易流程的项目由新手独立承接。那已经不是简单移动端样式检查。

## 风险提醒

这类任务最容易范围膨胀。客户说“手机上改一下”，可能包括导航重构、内容重写、图片裁剪、表单接入、性能优化和浏览器兼容。报价前要把检查和修复拆开。

还有共享组件风险。修一个按钮组件，可能影响全站；修一个 grid，可能影响多个页面。项目前要问清页面范围和验收页面。

## 具体步骤

### 1. 先确认范围

至少问：需要检查哪些页面、哪些设备或宽度、是否只出问题清单、是否包含修复、是否有设计稿、客户素材是否最终版、是否有必须保留的组件。

可以用 [Proposal 生成器](/tools/proposal-generator)整理确认问题，再人工改成自然回复。

### 2. 设计服务档位

第一档是移动端诊断：截图、问题类型、建议动作。第二档是小修复：修明确的溢出、遮挡、按钮和堆叠问题。第三档是响应式重构：涉及页面结构、组件和设计调整。

新手更适合第一档和少量第二档。第三档要谨慎报价。

### 3. 设置暂停条件

如果发现问题来自设计稿缺失、客户素材未确认、共享组件影响范围很大、需要重新设计页面，或需要访问客户私有仓库和部署账号，应暂停确认。

需要登录、授权、购买工具或访问客户私有环境的事项，记录为客户确认或明日待办。

### 4. 设计交付物

诊断交付可以是问题表：页面、视口、截图、问题类型、建议动作、是否阻止交付。修复交付要包含改动摘要、验证截图和未覆盖范围。

报价时可以用 [报价计算器](/tools/pricing-calculator)估算页面数量、视口数量、截图记录和修复时间。

## 可以考虑接的任务

客户有一个 3 页以内的 Tailwind 落地页，希望你检查 375px、768px 和桌面端，并提供问题清单。这类任务边界清楚，适合新手练习。

客户指出一个具体问题，例如“手机端价格卡片横向滚动”，并提供页面链接和截图。这类小修复也可以考虑，但要说明修复范围只覆盖指定问题和相关页面。

## 报价时怎么拆

报价可以按页面数、视口数和是否包含修复来拆。比如 3 个页面、3 个视口、只输出诊断报告，是一个范围；3 个页面、3 个视口、包含 P1 问题修复，是另一个范围；如果还要重做页面结构、补设计稿或修改文案，就应该另算。

不要把截图记录、复查和客户沟通时间漏掉。移动端任务看似只是改 class，但真正花时间的是复现、截图、比对和说明。用 [报价计算器](/tools/pricing-calculator)估算时，要把这些步骤都算进去。

## 应该暂停的任务

客户要求你“顺便重做移动端设计”，但没有设计稿、没有页面范围、没有验收标准。或者客户要求你处理登录后台、支付页、真实订单页的移动端问题，这类任务需要重新评估。

客户只给一张模糊截图，不给链接或代码，也要求你立即报价完整修复，应先要求补充材料。

## 交付话术示例

可以这样写：“本次检查覆盖首页、服务页和联系页，视口为 375px、768px 和桌面宽度。报告会包含问题截图、等级、建议动作和是否阻止交付。若发现共享组件影响多页面，或需要客户确认素材和文案，会单独列为待确认。”

这段话不会承诺所有设备，也不会把诊断和修复混在一起。对新手项目练习来说，它是一个比较稳的边界。
