# n8n AI Agent 怎么接知识库和记忆：RAG、上下文和状态存储

n8n AI Agent 能调用工具和处理流程，但如果没有知识库和记忆，它很容易变成每次都从零开始的助手。知识库让 Agent 能查资料，记忆让 Agent 能保留偏好、任务状态和项目背景。两者不能混为一谈：RAG 解决“资料在哪里”，记忆解决“这个用户或任务以前发生了什么”。

本文是待复核草稿。Agent 记忆设计可以看 [Agent 记忆怎么设计](/blog/agent-memory-design-guide)，RAG 基础可以看 [RAG 知识库怎么搭](/blog/rag-knowledge-base-beginner-guide)。

## 适合谁

适合做 n8n 客服 Agent、销售 Agent、内部问答、工单处理、文档摘要和项目助理的人。你希望 Agent 既能查资料，又能保留当前任务上下文。

也适合自动化接单项目。客户往往会提出“能不能记住客户需求”“能不能查公司资料”“能不能下次接着聊”，这就是知识库和记忆的边界问题。

## 不适合谁

不适合把所有聊天记录都塞进上下文的人。这样成本高、速度慢，也容易把无关信息带入回答。

也不适合把长期记忆直接放进公开工作流。涉及客户资料和内部信息时，要有权限控制和删除机制。

## RAG 查资料

RAG 适合存产品文档、FAQ、SOP、制度、合同模板和培训资料。Agent 需要回答业务问题时，先检索知识库，再基于片段回答。

关键是要返回来源。没有来源的知识库回答很难复核。RAG 引用可以看 [RAG 怎么显示引用来源](/blog/rag-citation-source-trace-guide)。

## 短期上下文

短期上下文是当前任务里必要的信息，例如用户刚提交的表单、当前工单、最近几轮对话和当前步骤。它不一定需要长期保存。

n8n 工作流里可以通过变量和节点输出传递短期上下文。上下文要精简，只保留当前步骤需要的字段。

## 长期记忆

长期记忆适合保存稳定偏好、项目事实、客户状态和历史决策。它应该有来源、更新时间、权限和过期规则。

如果要用数据库保存记忆，可以参考 [Agent 记忆用 Postgres 怎么设计](/blog/agent-memory-postgres-schema-guide)。不要把敏感信息默认长期保存。

## 状态存储

很多 n8n Agent 不是一次性任务，而是多轮流程。比如销售线索从收集、评分、补充信息、分配跟进到发送提醒。每一步都需要状态。

状态和记忆也不同。状态是流程进行到哪里，记忆是可复用的信息。状态可以短期保存，流程结束后归档或清理。

## 常见错误

第一个错误是把知识库、记忆和状态混在一个字段里。后续检索、更新和删除都会很难。

第二个错误是每次都把完整历史传给模型。更好的做法是摘要、筛选和按需检索。

第三个错误是没有测试问题。知识库是否能检索到、记忆是否会误用，都要用案例验证。

## 交付检查

如果给客户交付 n8n RAG Agent，建议把“资料从哪里来”和“记忆存在哪里”写清楚。知识库资料要有来源、版本、权限和负责人；记忆要有用户范围、保存时间、删除方式和纠错入口；流程状态要能查询当前任务进行到哪一步。

验收时不要只问几个简单问题。要准备知识库命中问题、知识库找不到的问题、用户偏好更新问题、旧记忆冲突问题和工具调用失败问题。只有这些场景都跑过，才算接近真实使用。

后续还可以拆成更细的搜索专题：n8n 怎么接向量数据库、n8n 怎么保存聊天历史、n8n Agent 怎么做客服知识库、n8n Agent 怎么把用户偏好写入数据库。这些都是用户搭到一半会搜的问题，比单纯介绍 n8n 更容易带来有效咨询。

## 风险提醒

n8n Agent 接知识库和记忆后，可能处理客户资料、公司内部资料和业务状态。要控制谁能读、谁能写、谁能删除。

如果 Agent 的记忆出错，要有人工纠错入口。错误记忆会影响后续任务，不能忽视。

## 具体步骤

第一步，区分知识库、短期上下文、长期记忆和流程状态。第二步，为知识库准备文档和来源。第三步，为记忆设计字段和删除规则。第四步，在 n8n 中传递必要上下文。第五步，记录状态变化。第六步，用真实任务测试检索和记忆。需要设计表，可以从 [工具导航](/tools) 下载或联系人工协助搭建。
