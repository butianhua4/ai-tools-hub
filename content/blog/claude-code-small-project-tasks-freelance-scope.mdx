# Claude Code 小项目任务哪些适合写进项目范围

小项目最重要的是范围。适合写进范围的是单页面、单报错、单配置、单文档和交付说明。不适合默认包含的是生产部署、数据库、付款流程、账号权限、安全事故、全站重构和长期维护。

这篇适合搭配 [Claude Code 小项目任务检查清单](/blog/claude-code-small-project-tasks-checklist)、[Claude Code 项目客户问题检查清单](/blog/claude-code-client-questions-checklist) 和 [/templates](/templates) 使用。

## 适合谁

适合想把 Claude Code 用在低风险服务上的新手。你可以做诊断、整理、局部修改和交付说明，但要避免承接超出能力和权限的任务。

也适合准备写报价的人。报价范围越清楚，客户越不容易把小单理解成全项目支持。

## 不适合谁

不适合客户要求你兜底整个项目、长期在线支持、处理生产事故、访问真实数据或修改付款系统的任务。

也不适合需求模糊但客户催得很急的任务。越急越要先确认范围。

## 具体步骤

第一，基础范围可以包含单页面。比如一个页面文案、一个按钮跳转、一个移动端样式问题。必须写清页面路径和验收方式。

第二，基础范围可以包含单报错。比如构建失败、控制台错误、依赖冲突。必须有脱敏错误原文和复现步骤。

第三，基础范围可以包含单配置。比如解释一个脚本、修一个路径、补一段说明。不要默认接管整套部署。

第四，基础范围可以包含单文档。比如 README、交付说明、安装记录、客户需求清单。这类任务风险低，也适合新手练习。

第五，修复范围只包含一个明确问题。多个问题要拆成多个任务。客户追加需求时，重新确认范围。

## 报价范围示例

```text
本次包含：
- 一个明确页面/报错/配置/文档问题
- 低权限诊断
- 小范围修改建议或修改
- 基础验证和交付说明

不包含：
- 生产部署
- 数据库和真实数据
- 付款或账号权限
- 全站重构
- 长期维护
```

这段可以直接放进报价，也可以作为项目前确认。

## 怎么拆成服务档位

基础档：只做诊断和交付说明。适合信息不完整、客户只想先判断原因的任务。

标准档：处理一个明确、低权限、可测试的问题。交付修改摘要、验证结果和未覆盖范围。

进阶档：多个页面或多个文件的影响检查。进阶档必须重新报价，不要塞进基础小单。

## 客户确认话术

```text
我可以先处理一个明确的小范围任务。本次不包含生产部署、真实数据、账号权限或长期维护。如果后续还有新问题，我们可以单独确认范围。
```

这句话可以在报价前发给客户，避免后续边界不清。

## 怎么写交付物

基础小单的交付物不要写得太空。可以写：“一份诊断记录”“一个明确问题的修改说明”“一组本地验证结果”“一段客户验收步骤”。交付物越具体，客户越知道自己收到什么。

如果客户希望你提供录屏、完整文档、多轮答疑或部署陪跑，就把它放进进阶范围。不要把进阶交付物默认塞进基础小单。

## 什么时候提高风险等级

只要出现账号、付款、数据库、真实用户数据、生产部署、长期维护，就把风险等级提高。提高风险等级不代表一定不做，而是不能沿用小项目报价和流程。

风险变高时，至少需要更清楚的授权、备份、回滚、验收和责任边界。新手如果无法确认这些条件，就应该暂不承接。

## 验收支持怎么限定

基础小单可以包含一次验收说明和一次必要澄清。例如告诉客户打开哪个页面、运行哪个命令、确认哪个结果。多轮远程协助、陪客户部署、继续查新错误，不应默认包含。

如果客户验收时发现新问题，先判断是否属于本次目标。属于就处理；不属于就重新确认新范围。

## 交付物示例

```text
交付物：
- 一个明确问题的诊断或修改摘要
- 一条本地或测试环境验证结果
- 一段未覆盖范围说明
- 一段客户验收步骤
```

这四项足够支撑基础小单，也能避免服务描述太虚。

## 风险提醒

不要把“低风险小项目”包装成“所有问题都能解决”。范围越小，交付越稳。

不要让 Claude Code 的建议清单变成客户的免费追加清单。建议可以记录，是否执行要重新确认。
