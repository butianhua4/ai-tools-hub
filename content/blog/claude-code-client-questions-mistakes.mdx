# Claude Code 项目前问客户时的常见错误

新手用 Claude Code 接代码小单时，最容易把注意力放在工具上，却忘了问客户关键问题。工具能读代码，但不能替你确认客户到底要什么、问题怎么复现、哪些权限不能碰、修完怎么验收。漏问这些问题，小单很容易变成无边界返工。

下面整理常见错误和修正方法。建议配合 [Claude Code 项目客户问题检查清单](/blog/claude-code-client-questions-checklist) 与 [/tools/proposal-generator](/tools/proposal-generator) 使用。

## 适合谁

适合已经开始回复代码小单客户，但经常发现后续信息不够的新手。你可能报价太快，修到一半才发现客户说的是另一个环境。

也适合准备建立固定客户问题模板的人。问题问清楚，后面的 prompt、报价、测试和交付都会更稳。

## 不适合谁

不适合客户不愿提供任何信息，却要求你承诺结果的任务。这样的需求不适合直接修复。

也不适合高风险生产任务。客户要求你直接进后台、拿密钥、改付款或处理真实数据时，普通沟通模板不够，需要更严格授权。

## 具体步骤

错误一：不问复现步骤。修正方式是要求客户写清打开哪里、点击什么、输入什么、何时报错。不能复现时，先做诊断。

错误二：不问运行环境。客户说线上坏了，可能本地正常；客户说本地坏了，可能部署没问题。环境不同，报价和风险不同。

错误三：不问预期结果。客户说“不对”，你要问“对的结果是什么”。否则工具无法判断修复方向。

错误四：不问权限边界。是否涉及密钥、账号、数据库、付款、客户数据？这些信息决定能不能接。

错误五：直接接受“很简单”。客户觉得简单，不代表技术上简单。你要看复现、范围和测试条件。

错误六：不问验收方式。没有验收方式，交付后很容易出现“我以为你还要做这个”的争议。

## 修正模板

```text
目前信息还不足以确认报价。我需要先确认：
- 问题复现步骤
- 错误原文
- 当前环境
- 预期结果
- 权限和敏感数据边界
- 验收方式
```

这个模板可以防止你被模糊需求推着走。客户愿意补信息，任务才适合继续。

## 复盘建议

每次返工后，回看自己一开始漏问了什么。是没问环境，还是没问验收，还是没问客户是否有测试数据？把漏问项加进下一次模板。

项目判断能力不是一次练成的，它来自每次沟通后的修正。问题模板越稳定，后续交付越稳定。

## 下次避免漏问

可以把每次沟通分成三段：先判断事实，再判断风险，最后判断交付。事实包括复现步骤、错误原文、环境和最近变更；风险包括权限、数据、账号、付款和生产影响；交付包括处理范围、测试方式和客户如何验收。

如果你发现自己经常漏问同一类问题，就不要靠记忆。把它写进固定回复模板里。比如每次都问“是否涉及真实用户数据或管理员权限”，每次都问“修完后以什么结果验收”。模板不是让沟通变僵，而是帮你在紧张时不漏底线。

## 交付前核对

修完之前，再回看最初的问题清单。客户当时要的是哪个现象，后来新增了哪些要求，哪些要求没有包含在本次范围内。交付说明里要把这些写清楚，避免客户把诊断、修复、部署、长期维护混成一个任务。

如果 Claude Code 帮你生成了修改摘要，也要人工改成客户能读懂的话。不要把技术细节堆给客户，也不要省略关键限制。好的交付说明应该同时说明“做了什么”“怎么验证”“哪些没有动”。

## 风险提醒

不要为了显得响应快而跳过确认问题。快速回复可以，但回复内容应该是问题清单，而不是过早承诺。

不要让客户把敏感信息直接发来。你可以要求脱敏截图、错误码和测试数据，不要要求完整密钥或真实用户资料。
