# Claude Code 新手使用场景检查清单

用 Claude Code 接小单前，先判断这个任务是不是适合新手。适合的任务通常有四个特征：问题能复现、权限低、改动范围小、结果能验证。不符合这些条件时，可以先做诊断或学习记录，不要直接报价为完整修复。

这份清单可以搭配 [Claude Code 新手适合用在哪些接单场景](/blog/claude-code-beginner-use-cases)、[Claude Code 自动化人工审核清单](/blog/claude-code-automation-human-review-checklist) 和 [/templates](/templates) 使用。

## 适合谁

适合准备承接小型报错、代码阅读、配置文档、轻量页面修改的新手。你已经能基本运行项目，也愿意保留检查记录。

也适合还在筛选任务的人。客户发来的需求不一定都能接，清单能帮助你快速判断是继续沟通、先诊断，还是直接拒绝。

## 不适合谁

不适合高权限、高风险、高责任任务。涉及付款、密钥、生产数据库、真实用户信息、安全漏洞、管理员后台时，新手不应独立处理。

也不适合没有复现条件的任务。客户没有截图、没有错误信息、没有环境说明，却要求你保证修好，这种需求要先补信息。

## 具体步骤

1. 问题是否能复现？是否有触发路径、错误信息、截图或命令输出。
2. 是否可以用测试数据？如果必须接触真实客户数据，要提高风险等级。
3. 是否只涉及小范围文件？一个组件、一个配置、一段文档更适合新手。
4. 是否能看懂当前项目结构？看不懂时先做代码阅读，不直接修改。
5. 是否需要客户账号、密钥或管理员权限？需要时优先暂停确认。
6. 是否有可运行检查？lint、build、手测、截图、内容检查至少有一种。
7. 是否能写清交付说明？做了什么、怎么验证、未覆盖什么。
8. 是否有客户验收步骤？客户能否按说明确认结果。
9. 是否有回滚或撤销方式？改动出问题时能否恢复。
10. 是否避免过度承诺？不写无法验证的业务结果。

## 场景评分

低风险：报错解释、代码阅读、配置文档、测试环境小改、交付说明整理。可以继续确认报价。

中风险：需要改多个模块、涉及线上部署、需要客户临时权限、测试条件不完整。建议先做诊断或拆小范围。

高风险：生产数据、付款、密钥、安全漏洞、客户核心业务流程。新手不建议独立承接。

## 风险提醒

不要因为 Claude Code 能读代码，就默认自己能接所有代码单。工具能提高速度，但不能替你承担客户授权、测试和交付责任。

不要把客户敏感材料复制到工具里做判断。只需要任务摘要时，就不要提供完整后台截图；只需要错误码时，就不要附上客户真实数据。

## 交付前自问

如果客户问“为什么这个任务适合你做”，你能否解释？如果客户问“怎么验收”，你能否给出步骤？如果客户要求扩大范围，你能否判断是否新增报价？

这些问题回答清楚，说明任务更适合继续。如果回答不清楚，先补信息，不要让工具替你硬做。

## 降级处理

如果清单里有两三项回答不清楚，不要硬接。可以把任务降级为诊断：先阅读脱敏材料、整理可能原因、列出缺失信息和下一步建议。诊断比盲目修复更适合信息不足的情况。

降级处理也要有交付物。比如一页排查报告、一份客户问题清单、一组建议截图。这样客户仍然得到帮助，你也没有越过自己的能力边界。

## 记录建议

每次判断任务后，记录最终结论：直接接、先诊断、暂不承接。再记录原因：是否可复现、是否低权限、是否有测试方式。几次之后，你会看到自己适合处理哪些场景。

这份记录也能变成你的报价依据。你会更清楚哪些任务需要一小时诊断，哪些任务可以直接做小范围修复，哪些任务必须请人复核。
