# 新手如何记录每天的项目练习

新手做 AI 项目练习，最容易把记录写成流水账：今天用了 Codex，今天看了 Upwork，今天改了一段网页。这样的记录能证明你很忙，但很难证明你变强了，也很难转化成作品集或客户沟通材料。

更有用的记录应该回答四个问题：今天练的是什么任务、你原本不会哪里、用了什么步骤验证、最后留下了什么可复查证据。项目不是靠记忆交付，而是靠可解释、可验证、可复盘的过程。

## 适合谁

适合刚开始用 ChatGPT、Codex、Claude Code、Cursor 等工具练习项目的人。你可能还没有真实客户，但已经在做小网页、报错排查、文案整理、表格自动化、模板生成等练习。

也适合想做作品集却不知道写什么的人。作品集不一定从大型项目开始，很多可信案例来自小练习：原始问题、处理过程、截图、命令、结果和限制说明。

## 不适合谁

不适合只想复制别人的案例、批量伪造练习记录的人。练习记录的价值在于真实过程，不在于包装得多漂亮。

也不适合把 AI 输出直接当成最终能力证明的人。客户关心的是你能不能判断需求、处理边界、验证结果和清楚沟通，而不只是工具能生成什么。

## 风险提醒

练习记录不要包含客户隐私、平台违规内容、未授权代码、敏感账号、付款信息或他人作品截图。即使只是练习，也要养成最小化记录敏感信息的习惯。

另外，不要把每天练习等同于马上可以接复杂项目。记录能帮助你看清能力边界：哪些任务已经能独立完成，哪些还需要复核，哪些必须暂缓。

## 具体步骤

### 1. 每天只选一个主任务

不要一天同时练五个方向。可以选一个清楚的小任务，例如修一个 npm 报错、改一个落地页按钮、写一封客户回复、整理一个报价模板。任务越小，越容易验证。

记录格式：

```text
今天主任务：
为什么选它：
预期结果：
```

### 2. 写下原始问题

保留原始报错、需求截图或自己写的练习题。不要只写你理解后的版本，因为后续复盘时你需要知道自己有没有误读问题。

如果涉及真实客户或公开平台，先去掉姓名、邮箱、订单号、仓库地址、付款信息等敏感内容。

### 3. 记录 AI 怎么参与

不要只写“问了 AI”。要写清楚你给了什么上下文、AI 给了什么方案、你采纳了哪一部分、你拒绝了哪一部分。这样才能证明你不是盲目复制。

示例：

```text
我让 AI 解释报错原因，但没有直接使用它给出的删除配置方案。
我改为先检查 Node 版本和依赖锁文件。
```

### 4. 记录验证结果

每个练习都要有验证结果。网页练习可以截图，代码练习可以记录 `npm run build`，文案练习可以保留修改前后版本，报价练习可以写清楚假设条件。

没有验证的记录，只能说明你做过；有验证的记录，才更接近可交付。

### 5. 写一句客户表达

每天练习结束，试着把结果写成一句客户能听懂的话。例如：“我已定位到推送失败来自认证权限，而不是代码错误；下一步需要仓库所有者添加协作者权限。”

这个练习能帮你从“我会操作工具”过渡到“我能解释交付状态”。

## 最低记录模板

```text
日期：
练习任务：
原始问题：
使用工具：
关键步骤：
验证方式：
结果截图或命令：
没解决的部分：
可写进作品集的一句话：
```

每天填完这九项，就足够形成一条可复查记录。坚持一周后，你会比只收藏教程的人更清楚自己的真实能力。

## 怎么筛选成作品集

不是每条练习都要公开。适合进入作品集的记录，通常要满足三个条件：问题能让别人看懂，过程能证明你做了判断，结果能被截图、命令或前后对比验证。只写“我学会了某工具”的记录，适合个人复盘，不适合作品集展示。

筛选时可以给每条记录打三个标签：可公开、需脱敏、仅自用。可公开记录可以整理成案例；需脱敏记录要删除客户信息、仓库地址和敏感截图；仅自用记录保留在个人笔记里，用来提醒自己哪里还需要练。

如果当天练习卡在客户授权、平台账号、私有仓库或付款流程，就不要硬写成完成案例。把它写成待办：明日需要客户补充什么、需要自己验证什么、哪些内容不能公开。清楚记录边界，比勉强包装成果更有长期价值。
