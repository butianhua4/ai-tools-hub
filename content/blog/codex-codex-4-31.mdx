# Codex 是什么：新手怎么开始

Codex 可以理解为一个帮助你读代码、改代码、解释报错和执行开发任务的 AI 编程助手。对新手来说，真正重要的不是马上把它当成项目捷径，而是学会把需求说清楚、把改动看明白、把验证过程留下来。

这篇仍是草稿。发布前需要人工核对 Codex 官方说明、当前可用入口、账号或授权要求，以及不同环境下的真实操作截图。

## 适合谁

适合刚开始接触 AI 编程工具的人。你可能只会一点 HTML、CSS、JavaScript，或者还不熟悉 Git、终端和构建命令，但愿意从小项目开始练习。

也适合准备做自由职业前置练习的人。你可以先用 Codex 帮自己完成页面修改、报错解释、文档整理和交付记录，而不是一上来就承诺处理客户复杂项目。

## 不适合谁

不适合想把 Codex 输出直接交付给客户的人。AI 可以生成方案，但你必须审核 diff、运行检查、确认是否符合需求。

也不适合还不会区分本地练习和客户生产环境的人。如果项目涉及支付、登录、数据库、隐私数据、服务器权限或安全漏洞，新手不应该独立处理。

## Codex 能帮你做什么

新手可以先把 Codex 用在低风险任务上：

1. 解释报错日志，帮你拆出下一步排查方向。
2. 阅读小项目代码，说明文件之间的关系。
3. 修改一个页面文案、样式或小组件。
4. 生成测试清单、交付说明和客户确认问题。
5. 帮你整理 `git diff`，提醒哪些改动需要复核。

这些能力的前提是你给出清楚上下文。比如项目目标、当前报错、相关文件、运行命令、预期结果。输入越模糊，输出越需要谨慎审核。

## Codex 不能替你做什么

Codex 不能替你保证项目一定能接、一定能交付，也不能替你承担客户沟通、平台规则、版权、安全和隐私责任。它也可能误解需求、漏看边界、生成不适合当前项目的代码。

所以新手的目标不是“让 AI 全自动做完”，而是建立一个可复查流程：提出需求、获得建议、检查改动、运行验证、记录结果、再决定下一步。

## 具体步骤

1. 先准备一个本地练习项目，不要用客户生产项目练手。
2. 记录项目目标，例如“修改首页按钮文案”或“修复一个 npm install 报错”。
3. 把当前文件、报错、运行命令和预期结果发给 Codex。
4. 让 Codex 给出小步修改，不要一次要求重构整个项目。
5. 修改后查看 `git diff`，确认每个文件为什么被改。
6. 运行项目已有检查，例如 lint、build 或测试脚本。
7. 写一份练习记录：改了什么、怎么验证、还不懂什么。
8. 把不需要你授权或注册的事项当天处理，需要账号、平台权限或客户确认的事项放进明日待办。

## 可以复制的练习模板

```text
Codex 入门练习记录

练习目标：
项目路径：
相关文件：
我给 Codex 的上下文：
Codex 建议：
实际改动：
已运行检查：
手动验证结果：
还需要人工复核的点：
明日待办：
```

常见本地检查可以从这些命令开始，具体以项目脚本为准：

```bash
git status
git diff
npm run lint
npm run build
```

## 第一个项目前练什么

项目前建议至少做三个练习：修一个本地报错、改一个页面、写一份交付说明。每个练习都要留下截图或命令输出。这样你以后回复客户时，不只是说“我会用 AI”，而是能说明你怎么排查、怎么验证、怎么交付。

如果练习中需要注册平台、连接 GitHub、添加 API key 或授权某个服务，先记录为明日待办。不要为了赶进度把密钥、账号或客户信息贴进聊天。

## 风险提醒

不要承诺收入结果，不要把 AI 生成内容当成最终事实，不要复制客户代码或截图做公开案例。使用平台做项目时，需要遵守对应平台的沟通、付款和交付规则。

如果你看不懂 Codex 改了什么，就不要交付。最稳妥的动作是把不懂的 diff 标出来，继续学习或找人复核。
