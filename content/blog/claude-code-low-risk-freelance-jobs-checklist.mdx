# Claude Code 低风险项目检查清单

低风险不是“任务看起来不难”，而是目标明确、权限低、影响范围小、能复现、能验证、能验收。用 Claude Code 项目前，先用这份清单判断任务是否适合新手。如果关键项不清楚，先做诊断，不要直接承诺修复。

建议先读 [Claude Code 新手适合先接哪些低风险小单](/blog/claude-code-low-risk-freelance-jobs)，再搭配 [Claude Code 小项目任务检查清单](/blog/claude-code-small-project-tasks-checklist) 和 [/templates](/templates) 使用。

## 适合谁

适合准备接页面小改、报错诊断、GitHub 文件检查、README 整理、交付说明这类低权限任务的新手。

也适合报价前复核。清单能帮你判断任务到底是低风险，还是只是客户说得轻巧。

## 不适合谁

不适合处理生产事故、真实数据、付款流程、账号权限、数据库、安全事件、长期维护和全站重构。

也不适合客户拒绝提供复现步骤、错误原文、测试环境和验收方式的任务。

## 具体步骤

1. 目标是否只有一个？多个目标先拆分。
2. 是否有复现步骤？没有复现就先诊断。
3. 是否有错误原文、截图或页面路径？截图要脱敏。
4. 是否低权限？不碰账号、付款、数据库、后台和真实数据。
5. 是否影响范围小？一个页面、一个文件、一个配置、一个报错更适合。
6. 是否能本地或测试环境验证？没有验证条件就要写限制。
7. 是否有客户验收动作？客户要知道如何确认完成。
8. 是否能写未覆盖范围？生产部署、长期维护、新增需求默认排除。
9. 是否需要客户补材料？缺材料时先问，不急报价。
10. 是否适合用 Claude Code 辅助？工具能帮读代码，但不能替你承担授权。

## 判断结果分类

低风险：目标单一、权限低、可测试、可验收。可以进入小范围报价。

中等不确定：信息缺失但客户愿意补材料。先做诊断。

高风险：涉及生产、数据、付款、账号、安全、长期维护。暂不按普通小单处理。

## 客户确认模板

```text
我可以先判断这个任务是否适合低风险处理。请确认：
- 一个明确目标
- 复现步骤或错误原文
- 当前环境
- 是否涉及账号、付款、真实数据或管理员权限
- 修完后怎样验收
```

这段回复能把客户的模糊需求变成可判断信息。

## 交付记录模板

```text
任务目标：
风险等级：
处理范围：
验证结果：
未覆盖范围：
客户验收：
```

低风险任务也要写交付记录。记录越清楚，客户追加新问题时越容易重新确认。

## 常见低风险任务

页面文案、README 整理、单文件解释、脱敏错误诊断、简单配置说明、交付说明整理，通常更适合新手。它们权限低、影响小、容易写验收。

但同一个任务也可能升级。例如页面文案如果涉及付款流程或生产后台，就不再是普通低风险任务。

## 验收示例

页面小改：客户在测试环境打开指定页面，确认文案、按钮或间距符合说明。报错诊断：客户确认错误原文、可能原因和下一步材料是否准确。文档整理：客户确认文档覆盖指定安装或交付步骤。

验收示例要写在报价或交付里。没有验收示例的小单，很容易变成“再帮我看看”。

## 停止条件

出现生产环境、真实数据、付款、账号权限、数据库、安全事件、长期维护时，停止按低风险小单处理。接下来不是继续加内容，而是重新确认范围。

这个停止条件可以提前告诉客户。提前说清，后面就不尴尬。

## 客户回复样例

```text
这个任务目前看起来可以先做低风险诊断。本次只确认一个明确问题，不进入生产环境、不处理真实数据、不修改账号或付款设置。诊断后如果适合继续修复，我们再确认下一段范围。
```

这段话适合在报价前发出。它没有夸大能力，也给客户留了继续合作的路径。

## 复盘记录

每次小单结束后，记录这次是否真的低风险：有没有追加需求、有没有权限问题、验收是否一次通过。下次报价时，把这些记录变成更清楚的条件。

## 风险提醒

不要把低风险检查当成结果承诺。检查只能证明当前信息下适合或不适合处理。

不要用客户敏感材料训练自己。练习应该在自己的测试项目里完成。
