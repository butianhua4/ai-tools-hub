# TypeScript 报错新手怎么排查：接单前怎么判断能不能做

TypeScript 报错修复看起来像小单，但范围可能很大。一个错误可能只需要补类型，也可能牵出依赖升级、配置变化、共享类型和数据结构问题。接单前要先判断材料是否足够，以及你能否验证修复。

这篇仍是草稿，发布前需要补充真实客户案例。

## 适合谁

适合准备接小型前端、Next.js、React、Node.js 报错修复任务的新手。你能读日志、运行构建命令，也愿意把诊断和修复拆开。

也适合正在写 Proposal 的人。你需要先问清楚材料，而不是直接承诺当天修完。

## 不适合谁

不适合没有完整日志、没有仓库访问、没有验证命令的需求。也不适合生产系统紧急事故。

如果任务涉及客户数据、支付、权限或安全逻辑，需要更谨慎的人工复核。

## 接单前先问什么

先问完整报错、运行命令、项目框架、最近改动、是否有测试环境、修复完成的标准。没有这些信息时，最多只能做初步判断。

还要问错误影响哪里。是本地构建不过、部署失败、页面类型提示，还是运行时也有问题。TypeScript 报错和真实运行错误要分开看。

## 判断能不能做

可以做的情况：错误集中在少数文件，客户能提供仓库和验证命令，修复后能用构建或测试确认。

需要先诊断的情况：错误很多、依赖刚升级、配置刚改过、项目年代较久，或者客户不确定最近发生了什么。

应该暂缓的情况：客户不给访问方式，却要求保证修复；任务牵涉生产权限；客户要求你改大量类型但没有验收标准。

## 具体步骤

1. 保存客户原始描述。
2. 要完整日志和运行命令。
3. 判断错误数量和分布。
4. 先做诊断报价。
5. 写清不包含生产发布和长期重构。
6. 修复后运行同一验证命令。
7. 给客户一份原因和文件变更说明。

## 报价边界

单个文件、少量类型错误，可以按小任务报价。多个目录、依赖升级、配置修改，应先按诊断报价。共享类型、后端接口和数据库结构相关问题，不应按简单前端报错处理。

交付说明要写清是否包含代码修改、构建验证、部署协助、文档说明和后续维护。不要把“修 TS 报错”默认为所有相关问题都包含。

## 验收边界

验收标准要写成可执行命令。比如“`npm run build` 通过”“指定页面不再出现该类型错误”“部署预览构建成功”。不要写“项目正常”，因为这个范围太大。

如果客户希望你顺便清理所有 TS 问题，要重新评估范围。一个项目里可能有几十个历史类型问题，和本次报错不一定同源。你可以先修当前阻塞项，再给出后续整理建议。

交付时要说明是否改了业务逻辑。如果只是补类型声明，影响较小；如果改了数据判断、接口处理或组件渲染，就要提醒客户测试相关页面。TypeScript 修复有时会改变运行行为，不能只看类型检查。

## 客户材料不足时怎么回复

可以回复：“我需要完整报错、运行命令、仓库或相关文件、最近改动和验收标准。没有这些信息时，我可以先判断可能方向，但不能确认修复范围。”这样的回复既推进项目，也保护你不盲目承诺。

如果客户补充材料后仍然不完整，可以把任务改成“诊断报告”而不是“修复保证”。报告里写清已知信息、缺失信息、可能原因和下一步。这样仍然有交付物，也不会把不确定性都压到你身上。

## 风险提醒

不要承诺所有报错一次清完。不要为了快速通过使用大量 `any`。不要关闭类型检查。不要在没有回滚方式时改客户共享类型或配置。
